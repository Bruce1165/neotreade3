# Market Intelligence 边界恢复样本维护说明

日期：2026-06-15

## 1. 目的

- 本说明只解决一个问题：
  - 当某只股票在当前规则下可以保留为 `推荐`，但团队仍要求“特别标注、人工重点复核”时，如何维护这类样本。
- 本说明不讨论：
  - 推荐逻辑修改
  - 风险标签重构
  - 渗透率规则设计
  - 前端结构调整

## 2. 当前机制

- 当前边界恢复样本通过单独配置维护：
  - `config/market_intelligence/boundary_recovered_candidates.json`
- 当前机制特点：
  - 不改变 `recommendation_status`
  - 不改变推荐排序
  - 不参与主题匹配
  - 只额外输出：
    - `special_markers`
    - `special_marker_notes`
- 当前前端展示位置：
  - 焦点候选
  - 建议层候选列表
  - 联动摘要

## 3. 什么时候加入边界恢复名单

- 仅当同时满足以下 3 条时，才考虑加入：
  - 该样本在当前规则下可以接受为 `推荐`
  - 团队明确认为它需要被特别标注
  - 特别标注的目的，是提醒“人工重点复核”，而不是替代推荐降级

说明：
- 如果团队认为该样本其实不应保留 `推荐`，那应进入规则复核，不应加入边界恢复名单。
- 边界恢复名单不是“有争议样本池”，而是“允许推荐但要求显式提示”的样本池。

## 4. 什么时候不要加入

- 以下情况不要直接加入边界恢复名单：
  - 明显非 AI 主线样本
  - 缺少可确认主题锚点的样本
  - 本应通过规则降为 `观察` 或 `回避` 的样本
  - 只是暂时没看明白、但团队尚未明确接受为 `推荐` 的样本

## 5. 配置文件结构

当前文件：

```json
{
  "version": 1,
  "updated_at": "2026-06-15",
  "items": [
    {
      "stock_code": "000338",
      "marker": "boundary_recovered_candidate",
      "note": "边界样本：因补齐渗透率标注后恢复为推荐，需人工重点复核主题集中度。"
    }
  ]
}
```

字段说明：
- `stock_code`
  - 6 位股票代码
- `marker`
  - 当前固定值：
    - `boundary_recovered_candidate`
- `note`
  - 给团队看的简短说明文案

## 6. 最小维护步骤

### 6.1 新增样本

1. 打开：
   - `config/market_intelligence/boundary_recovered_candidates.json`
2. 在 `items` 中新增一条记录：
   - `stock_code`
   - `marker`
   - `note`
3. 更新：
   - `updated_at`

### 6.2 移除样本

当团队不再要求特别标注时：

1. 从 `items` 中删除对应股票
2. 更新：
   - `updated_at`

### 6.3 修改说明文案

仅当团队对“特别标注原因”的表述发生变化时，才修改 `note`。

要求：
- 文案应描述“为什么需要人工重点复核”
- 不要把 `note` 写成新的推荐规则

## 7. 推荐文案口径

当前推荐口径：

- 标记文案：
  - `边界恢复`
- 说明文案：
  - `边界样本：因补齐渗透率标注后恢复为推荐，需人工重点复核主题集中度。`

如需换文案，建议保持两点不变：
- 明确说明这是“已接受推荐”的样本
- 明确说明仍需“人工重点复核”

## 8. 最小验证步骤

配置修改后，至少验证以下两类结果。

### 8.1 接口验证

检查：

- `GET /api/market-intelligence/recommendations`
- `GET /api/market-intelligence/review-board`

重点看：
- 目标样本是否仍保持原本 `recommendation_status`
- 是否新增：
  - `special_markers`
  - `special_marker_notes`

### 8.2 对照样本验证

除目标样本外，至少再看 2 类对照：

- 正常推荐样本
  - 不应自动带上边界恢复标记
- 非推荐样本或弱样本
  - 不应因为该配置而受到影响

## 9. 当前已确认样本

截至当前，边界恢复名单中仅包含：

- `000338`

这表示：
- 它可以保留 `推荐`
- 但团队要求特别标注，提醒人工重点复核主题集中度

## 10. 维护边界

- 边界恢复名单应保持小而明确。
- 如果名单开始持续扩大，优先考虑先回看：
  - 是否出现新的系统性规则问题
  - 是否某类样本应被统一建模，而不是继续手工挂标

## 11. 结论

- 当前边界恢复机制的正确用法是：
  - 先接受推荐结果
  - 再显式标注人工重点复核对象
- 后续如要维护，只应改配置：
  - `config/market_intelligence/boundary_recovered_candidates.json`
- 不应把这份名单当成替代推荐规则的补丁池。
