## 1. 对项目的理解（待 owner 校正）

### 1.1 愿景（Vision）

NeoTrade3 不只是一个选股脚本，目标是做一个**数据管理驱动的量化操作系统**（CLAUDE.md 原文）：统一数据控制、统一日常编排、统一实验室注册、统一学习闭环、统一问题聚合。分工哲学是——**人专注于模型与筛选器的调整，平台把每日数据与运行管理尽可能自动化**。

我的理解：这个愿景的本质是"把 owner 从运维中解放出来，让人的时间只花在机器做不了的事上——判断、口径、取舍"。数据主链（每天数据可用、可追溯、可回放）是这个愿景的地基，与 owner "没有及时的数据更新，其它都是瞎扯"完全同构。

### 1.2 目标（Goal，owner 2026-07-22 校准口径）

- **20–60 个交易日内有机会涨 30% 以上；100 个交易日内有希望涨 50% 以上**
- 低频/中低频，不做高频、不做日内
- 双评估标尺：**跟踪池质量**（筛选/甄别水平）+ **入场/离场择时**（执行水平）；两者无绝对先后，但**当前工作顺序是先池后择时**
- 回测验收：模型推演结果 vs 历史最强 Top200 牛股对照；Top200 是 oracle，**绝不反向框定候选宇宙**

⚠️ 口径差异待裁决（见第 7 章 Q1）：PROJECT_STATUS Code-Wiki 写的是"80%+ 机会涨幅 30%+"，owner 7/22 校准为"有机会/有希望"的两档表述；rulebook §1.2 短档占位条目（RB.M2.CERTAINTY_SHORT.001，planned）写的是 10–20 日/+30%，与 owner 的 20–60 日/+30% 也不一致。

### 1.3 要达成的结果（Desired Outcomes）

按 PROJECT_STATUS 的四大主链验收口径：

1. **数据主链**：昨天数据在、交易日历更新、质量检查结果可见
2. **选股主链**：每天产出候选股集，进监控池/跟踪池，池成员变化可见、可下载 CSV
3. **研究主链**：因子沉淀为可解释矩阵，结构可视化、历史增长可见、单票因子可解释
4. **闭环主链**：异常/退化/无效筛选器能被发现、定位、调整、可审计

最终形态（我的归纳）：每天早上 owner 打开 Dashboard，能回答三个问题——**数据昨晚没出问题吧？池子里现在有哪些票、为什么？有没有该动手的票、依据是什么？** 其余一切（抓取、校验、回测、报告）都应该已经在夜里自动跑完。

### 1.4 工作方法论（Methodology）

这是我理解中最需要说透的部分，因为它是这个项目区别于普通脚本集的地方：

**a) M1–M6 分层 + Step1–8 闭环（M6 架构）**

- M1 事实层：数据接入、标准化、质量证明（防前视）
- M2 周期识别与确定性层：输出候选跟踪池 + 每票的确定性读出 + 身份（龙头/中军/跟随）+ 证据束
- M3 决策引擎层：跟踪池状态机（跟踪→成熟→入场候选）、入场窗口、持有期滤噪、离场、hazard 见顶紧迫度、风控（-5% 硬止损 + 系统性 risk-off 一票否决）
- M4 评估层 / M5 治理层 / M6 交付解释层
- 运行闭环：Step1 全市场周期定位 → Step2 证据增强+确定性 → Step3 身份聚焦 → Step4 入场窗口 → Step5 持有滤噪与离场 → Step6 风控 → Step7 低频纪律（大量高频式交易=异常信号）→ Step8 池质量评估 → M4 评估 → M5 提案治理 → 回到 M2/M3 调整

**b) rulebook = 行为契约，不是文档**

`docs/architecture/lowfreq_v16_model_rulebook.md` 的 Contract Registry 把每条行为规则注册为可机器验收条目（implemented 必须有测试证据；planned/deferred 不得被线上消费）。这是这个项目最可贵的资产——**"已设计"和"已实现"在机制上被强制区分**，与 CLAUDE.md 铁律互为表里。

**c) handbook = PROJECT_STATUS.md**

运行边界、Code-Wiki、交接记录的单一入口；每次重大工作以 dated handoff 段续写。

**d) 受约束的自进化**

允许"候选变更→评估→可审计→可回滚"，**禁止黑箱自动改参直接上线**；阈值/权重调整只能走 M5 治理版本推进，不得边跑边改。

**e) 两条贯穿一切的红线**

- No-lookahead：在线决策只用 target_date 当日及历史；未来窗口真值只能存在于 M4/M5 评估层
- Fail-closed：数据不足时状态=pending、输出清零、不得产生任何硬动作

**f) 体系正在升级：旧口径 → 混沌模型 SSOT**

当前处于**新旧体系切换的中途**：`chaos_rulebook.md` 已把混沌模型（yin/yang/net_energy 阴阳序列及其历史沿革）定义为全链路唯一判定标准，形态/筛选器/评分降级为证据输入；但旧口径（wave_phase/pattern/certainty_score）仍是线上主链路，rulebook 明确"冻结待退出"。v1 因子清单已冻结（技术面:资金人气:综合面 = 3:4:3），但线上实现还是 net_energy 正负拆分、未升级到按因子贡献逐项求和。**这个切换的战略方向、节奏、风险，是 owner 指定要专项讨论的事，讨论前我不动混沌口径的任何代码。**

**g) 业务依据的四层结构（PROJECT_STATUS §0）**

五维框架（周期位置/主流指数择时/行业渗透率/技术面/资金情绪面）→ 两份补充修正 → 口头执行约束（-5% 硬止损；确定性>85 仅作参考不再是硬门槛）→ 2026-06-12 政策材料的非平均吸收（重点=第二大类硬科技细分龙头，AI 龙头优先，ETF 高配+机构高配为关键增强证据）。

**方法论的已知问题（owner 自己说的"可能现在还有很多问题"）**：版本体系未统一（脚本线 v16/v17 与 V3 线 v2_enhanced 三口径并行）；学习闭环只到"出建议"、未落地"审批→落地→回归"；回测缺真实交易约束（涨跌停/流动性/手续费滑点）。这些不是我发现的，是 PROJECT_STATUS 自己登记的已知差距。

## 4. 整体推进思路与计划

## 5. 建议的合作模式

### 5.1 分工

- **owner**：目标与口径的唯一权威、优先级裁决、领域判断（政策解读、市场理解、策略直觉）、混沌模型思想源头
- **我**：工程实现、主动审查、运维值守、证据生产；把 owner 的口径翻译成契约、代码与测试

### 5.2 决策门（CLAUDE.md 铁律的落地形式）

每个非平凡提议，我按固定五段式提交，owner 一句话即可裁决：

> **问题**（证据）→ **方案**（改动范围）→ **风险**（会动到什么）→ **验证**（怎么证明成了）→ **回退**（怎么撤）

琐碎事项（卫生级、无行为变化）我直接做、在日报里报备；凡触及口径、数据语义、调度、删除，一律走决策门。

### 5.3 节奏

- **日报**：每天结束前三行字——今天做了什么 / 发现了什么 / 明天计划（写入 PROJECT_STATUS handoff 段）
- **Phase 评审**：每个 Phase 结束一次正式交付，owner 确认后进入下一 Phase
- **随时沟通**：owner 发起即响应；我发现 P0 级问题不等日报、立即上报

### 5.4 冲突与口径管理

- 我的理解 vs owner 口径冲突 → **owner 最新口径为准**，我负责回填文档消除歧义（K6 类问题不再累积）
- 代码 vs rulebook 冲突 → 先停在原地报告，不擅自"修复"任何一边
- 新会话/重启的连续性 → PROJECT_STATUS handoff + 我的长期记忆文件双轨，任何会话从零恢复不超过 5 分钟

### 5.5 我的红线（自我约束，欢迎监督）

- 不碰实盘资金、不发起任何真实交易
- 不读取/外传 env.secrets 的值（只验证键存在）
- 不擅自切换混沌模型主链路（等专项讨论）
- 不删除任何数据；删除类操作永远先给清单再执行
- 不声称"已设计=已实现"；结论必带证据，证据不足明说边界
