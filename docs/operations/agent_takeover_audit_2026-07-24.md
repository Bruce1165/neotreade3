## 1. 对项目的理解（owner 已逐条校正确认）

### 1.1 愿景（owner 原话，2026-07-24 01:24）

> **以混沌阴阳的概念为根本，构建一个能够不断循环、自我更新的中低频量化交易模型，每天为交易团队提供：选股、建仓、离场等建议，并能够阐明原因，尤其是跟人的主观认知相左的因素。**

三个关键含义（owner 确认"基本到位"）：

1. **混沌阴阳不是"正在进行的升级"，是地基。** M1–M6、Step1–8、跟踪池、风控都是它的载体和表达
2. **产品是"每天的建议"，不是系统本身。** 受众是交易团队；系统与自动化为这份日产出的可靠性与可信度服务
3. **解释的重点是"与人的主观认知相左的因素"。** 模型的增量价值恰恰在于指出人看不到或不愿相信的东西，并能讲清为什么。证据链与解释层（M6）是核心价值的一半：一个与人相左但解释不清的建议，等于没有建议

护城河公式：**混沌阴阳的统一读出 × 可解释性（尤其反直觉部分）× 每日循环的可靠性**。

### 1.2 目标（owner 2026-07-24 01:37 定稿 + Q1 裁决口径）

1. **能力目标**：有能力经过时间验证，识别并抓住 Top200 牛股中的**大多数**；屏蔽各种噪音，尽力坚持持有到真正见顶的时候。配合涨幅刻度（Q1 已裁决）：**20–60 个交易日内有机会涨 30% 以上；100 个交易日内有希望涨 50% 以上**。三者合成：圈得住、拿得稳、退得准
2. **输出铁律**：输出的结果必须保证 **100% 客观真实，零蒸馏，零猜测**。零蒸馏 = 发布的任何数字必须可从存储事实直接复算，禁止无法还原的平滑、聚合、润色（owner 确认）
3. **闭环目标**：通过 6 层架构能够自查问题、精准定位、根据结果分析差距、调整各相关参数，形成"推进-复盘-回测-更新-推进"的正向闭环
4. **传达目标**：通过 Dashboard 高效、简洁地向交易团队传达信息，说人话

### 1.3 每日必须产出的七样东西（owner 2026-07-24 01:45/01:46 定稿）

1. **需要关注的票**（关注池）
2. **明天可以买入的票**（入场窗口已开、风险约束全通过）
3. **明天需要警惕的票（在持的）**（hazard 见顶紧迫度升高的持仓）
4. **明天必须离场的票**（无模糊地带）
5. **单票问诊**：用户输入股票代码，系统反馈该票当前状态的综合意见
6. **系统自主操作的记录**（模拟盘逐笔留痕、可查）
7. **总体盈利状况**（模拟盘账户盈亏，天天可见）

含义：模拟盘不是附属实验，是模型能力的公开成绩单。七样产出每一样都必须能从落盘证据直接复算（零蒸馏约束）。

### 1.4 工作方法论（owner 2026-07-24 01:55 定稿）

1. **分工**：owner 提要求/目标，Agent 负责具体实施
2. **先方案后实施**：方案、计划定稿在前，编码在后，保证编码跟文档同步
3. **不欠账**：推进过程中不留"先这样吧"的尾巴
4. **不跑偏**：阶段性复查两件事——与目标的一致性、成果的语法/语义双重正确性
5. **边界明确**：设定明显边界，不越界、不私自篡改
6. **回测铁律**：坚持 no-look-ahead
7. **fail-closed**（owner 2026-07-24 01:57 正式采纳为红线）：数据不够就不许装知道——关键数据不足以形成可靠输出时，系统必须明确输出"不确定"（pending/清零），并拒绝产生任何动作。与零猜测互为两面：零猜测管"输出不掺主观修饰"，fail-closed 管"证据不足必须明说算不了"

### 1.5 混沌阴阳的逻辑基础（owner 2026-07-24 02:22 定稿，深入讨论留待专项）

**主导思想**：根据股票的阳性能量与阴性能量的相互关系及其历史沿革，再加板块阴阳属性的关系、大盘阴阳属性的关系，判断该股票未来的走势（个股/板块/大盘三层嵌套）。

**能量计算**：对所有可获得因子按性质分类——阳（推动上涨）/阴（推动下跌）；贡献值正记阳、负记阴，净能量 = 阳 − 阴。可加性分解天然可解释。

**两级权重标定**：因子级权重（回测确定，或按数值统一打分）→ 技术/资金（人气）/综合三层间的相互权重（rulebook 冻结初始值 3:4:3）→ 层面内因子的二次 calibration。目标：力争高概率预判未来走势。

**Agent 提出、owner 确认"很关键、需进一步丰富完善"的风险防御（待写入 rulebook 硬约束）**：

- R-A **回测定权重是全系统最大的过拟合敞口**：防御 = walk-forward 分段验证、样本外不达标不上线、权重调整只走 M5 治理版本
- R-B **因子共线性导致能量虚增**：防御 = 定期相关性审计，高相关因子簇合并或降权
- R-C **层面权重可识别性有限**：3:4:3 当粗先验，只允许治理缓慢调整，每次调整须全历史回填证据
- 丰富方向：净能量跨票归一化（防高波动票系统性占优）；历史沿革精确定义（速度/加速度/flip_rate 窗口口径）；三层聚合规则；持续校准机制（bins 曲线 → M5 提案 → 版本化）

### 1.6 当前阶段定位（owner 2026-07-24 02:07 确认）

**不存在"两个体系的切换"，这个项目本来就是混沌阴阳模型。** rulebook 已把 SSOT 定义完整，实现还没长齐：

- 混沌日度快照、M4 评估监控已落地有测试；M5 治理仍 planned
- 线上计算仍是 `net_energy` 正负拆分的过渡实现，未升级到"按因子贡献逐项求和"（rulebook §2.3 自注差距）
- 旧口径字段（wave_phase / certainty_score）作为历史遗留存在，不得被视为最终动作语义

**方向早已定死，没有战略悬置**：剩下的工作是按 M5 治理节奏把实现补齐到 rulebook 已定义的混沌 SSOT。混沌专项讨论的议题 = 补齐的顺序、节奏、旧字段退出安排（Q4：Phase 1 报告后同场，议程含 R-A/R-B/R-C 防御落地）。

## 5. 合作模式（与 §1.4 方法论对齐）

### 5.1 分工

- **owner**：提要求/目标，做口径裁决与领域判断（政策解读、市场理解、策略直觉、混沌思想源头）
- **Agent**：具体实施——方案、编码、审查、运维、证据生产；代码与文档同步

### 5.2 决策门（五段式，owner 一句话裁决）

> **问题**（证据）→ **方案**（改动范围）→ **风险**（会动到什么）→ **验证**（怎么证明成了）→ **回退**（怎么撤）

琐碎事项（卫生级、无行为变化）直接做、日报报备；触及口径、数据语义、调度、删除一律走决策门（Q1–Q6 为首批范例）。

### 5.3 节奏

- **日报**：今天做了什么 / 发现了什么 / 明天计划（写入 PROJECT_STATUS handoff 段）
- **阶段性复查**：与目标的一致性 + 成果的语法/语义正确性（owner 方法论第 4 条）
- **Phase 评审**：每个 Phase 结束一次正式交付
- P0 级问题不等日报、立即上报

### 5.4 冲突与口径管理

- 理解 vs owner 口径冲突 → owner 最新口径为准，Agent 回填文档（本次第 1 章重写即范例）
- 代码 vs rulebook 冲突 → 停在原地报告，不擅自"修复"任何一边
- 连续性 → PROJECT_STATUS handoff + Agent 长期记忆双轨

### 5.5 Agent 红线（自我约束）

- 不碰实盘资金、不发起真实交易
- 不读取/外传 env.secrets 值
- 不越界、不私自篡改（owner 方法论第 5 条）
- 不删除任何数据；删除类操作永远先给清单再执行（Q5 为范例）
- 不声称"已设计=已实现"；结论必带证据；遵守 no-lookahead 与 fail-closed
