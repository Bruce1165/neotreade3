# M6 交付与观测层详细设计

日期：2026-07-07

## 1. 文档目标

本文档用于冻结 NeoTrade3 量化体系中 `M6 交付与观测层` 的详细设计，明确其正式定义、职责边界、正式输入对象、正式交付对象、观测视角、交付规则、语义护栏以及第一阶段实施范围。

本文档解决的问题：

- `M6` 到底是什么，不是什么。
- `M6` 应该投影哪些正式对象，不能投影什么。
- 页面、接口、告警与回放应如何高保真交付前五层权威结果。
- 单股、样本池、系统级三种视角应如何被统一组织。
- 展示层如何避免自行推断、改写或漂移上游正式语义。

本文档不展开：

- 页面视觉设计与具体交互稿。
- API 的最终字段命名与协议细节。
- 告警投递系统、通知通道与消息基础设施。
- 回放播放器的底层工程实现。
- 面向 BI 或通用分析平台的扩展接口设计。

## 2. 设计背景

顶层架构已经冻结为 `M1-M6` 六层结构，其中 `M6` 被定义为交付与观测层，用于把 `M1-M5` 已形成的正式事实、正式状态、正式评估与正式治理对象，以一致、可追溯、可操作的方式交付给人和系统消费者。

当前项目中的现实基础包括：

- `M1` 已被设计为事实与质量状态源。
- `M2` 已冻结 `small_cycle` 为首期唯一正式生产周期主链。
- `M3` 已冻结正式行为状态机与行为归因对象。
- `M4` 已冻结正式差距对象、样本摘要与 trace 对象。
- `M5` 已冻结诊断链、治理对象、blocker 与治理闭环状态。

用户已确认的关键方向：

- 前端页面不是判断源，而是 `M6` 对前五层正式对象的投影结果。
- 页面看到的一部分交易结果确实来自 `M3`，但必须是 `M3` 的权威输出，而不是前端自行推断。
- 同一正式对象在不同视图下可以有不同展示方式，但不能改变正式语义。
- 回放看到的必须是“当时系统正式知道什么”，不能把未来理解倒灌到历史画面。

## 3. M6 的正式定义

`M6` 不是新的分析层，也不是新的治理层。

`M6` 的正式定义为：

- 在不改写 `M1-M5` 正式输出语义、不自发生成新的事实真值、周期真值、交易真值或治理真值的前提下，
- 对正式对象进行统一交付、展示、检索、观测、回放和告警投影的交付与观测层。

`M6` 的核心承诺只有四个：

- 高保真投影
- 统一来源标记
- 可回链交付
- 视图不改语义

## 4. M6 的职责边界

### 4.1 M6 负责

- 统一投影 `M1` 的事实状态与质量状态。
- 统一投影 `M2` 的周期状态、证据、置信度与失效条件。
- 统一投影 `M3` 的行为状态、生命周期与动作归因。
- 统一投影 `M4` 的差距对象、达标等级、样本评估与回溯结果。
- 统一投影 `M5` 的诊断链、Change Request、Attention Item、blocker 与治理状态。
- 提供单股视角、样本池视角、系统视角三种观测入口。
- 提供面向人和程序的稳定消费面，如页面、API、报表、告警视图、回放视图。

### 4.2 M6 不负责

- 不自行推断缺失的上游语义。
- 不把局部状态包装成更高层真值。
- 不自己生成交易建议或治理判断。
- 不在展示时修改正式分类、正式理由、正式严重度。
- 不因为“用户更容易理解”就偷偷合并或改写关键对象边界。

### 4.3 与其他层的关系

`M6` 是投影层，不是新真值层：

- `与 M1`
  - 投影事实、质量状态、Freshness、Coverage、Replay、Attention 等对象，但不能改写事实口径。
- `与 M2`
  - 投影周期状态、证据束、confidence、invalidation、稳定等级，但不能自行把研究标签升级为正式标签。
- `与 M3`
  - 投影 `identify / tracking / entry / hold / exit`、生命周期和归因对象，但不能前端自行推断动作语义。
- `与 M4`
  - 投影达标等级、gap 台账、trace view、样本摘要，但不能自行改写偏差分类。
- `与 M5`
  - 投影 diagnostic chain、CR、blocker、治理状态、人工关注队列，但不能代替 `M5` 做治理决策。

## 5. 正式输入对象、交付对象与三种观测视角

`M6` 不能只是“把所有对象堆到页面上”，必须先冻结输入对象、交付对象与观测视角。

### 5.1 正式输入对象

`M6` 的输入必须全部来自 `M1-M5` 的正式输出对象。

首期冻结五类输入：

- `I1` `M1` 事实与质量对象
- `I2` `M2` 周期与证据对象
- `I3` `M3` 决策与生命周期对象
- `I4` `M4` 评估与差距对象
- `I5` `M5` 诊断与治理对象

#### `I1` M1 输入边界

至少包括：

- 核心事实状态
- `Freshness Proof`
- `Coverage / Consistency / Replay` 状态
- `Data Capability Gap`
- 原始 `Attention Item`

#### `I2` M2 输入边界

至少包括：

- `cycle_state`
- `state_stability_level`
- `state_transition_log`
- `confidence`
- `invalidation`
- `evidence_bundle`
- 必要的 `phase_candidate / transition_candidate`

#### `I3` M3 输入边界

至少包括：

- `identify_state`
- `tracking_state`
- `entry_state`
- `hold_state`
- `exit_state`
- `decision_lifecycle_log`
- `transition_reason`
- `action_justification`
- `local_exit_semantics`
- `global_thesis_end_semantics`

#### `I4` M4 输入边界

至少包括：

- `assessment_grade`
- `gap_record`
- `trace_bundle`
- `sample_bucket_summary`
- `gap_group_distribution`
- `interaction_guardrail_breach`

#### `I5` M5 输入边界

至少包括：

- `diagnostic_chain`
- `Change Request`
- `Experiment Request`
- `Promotion Blocker`
- `Governance Decision Record`
- 编排后的 `Attention Queue`

### 5.2 正式交付对象

首期至少冻结五类正式交付对象：

- `O1` 单股视图对象
- `O2` 样本池视图对象
- `O3` 系统总览对象
- `O4` 回放与追踪对象
- `O5` 告警与关注对象

#### `O1` 单股视图对象

必须支持单只股票完整逻辑链，至少包含：

- `M1` 事实可靠性状态
- `M2` 周期状态与证据
- `M3` 行为状态与生命周期
- `M4` 偏差与达标情况
- `M5` 诊断链与治理状态

#### `O2` 样本池视图对象

至少支持：

- `B1 / B2 / B3 / B4` 样本池切换
- 主要偏差分布
- 主要 blocker
- 主要协同风险
- 样本池级治理趋势

#### `O3` 系统总览对象

至少支持：

- 各层当前健康状态
- 主要风险位置
- blocker 总览
- 治理堆积状态
- 关键 guardrail 状态

#### `O4` 回放与追踪对象

至少支持：

- 单股历史链路回放
- 某次状态迁移的前后文
- 某个 gap 的全链路 trace
- 某条 `CR / Experiment / Validation` 的历史记录

#### `O5` 告警与关注对象

用于投影正式高价值风险对象，而不是泛化告警噪音。

### 5.3 三种正式观测视角

首期只冻结三种视角：

- `V1 Per-Symbol`
- `V2 Per-Sample-Bucket`
- `V3 System-Wide`

#### `V1` Per-Symbol

面向“这只股票到底发生了什么”，围绕同一只股票的完整逻辑链组织所有对象。

#### `V2` Per-Sample-Bucket

面向“这一类样本上系统总是怎么犯错”，服务于系统性偏差归纳。

#### `V3` System-Wide

面向“整个体系当前健康度如何、哪里在积压风险、哪层在失稳”，服务于全局治理优先级判断。

## 6. 页面、接口、告警与回放的正式交付规则与语义护栏

`M6` 最容易出问题的地方，不是显示不出来，而是显示得很顺手但偷偷改写了上游语义。

### 6.1 总体交付原则

所有交付形式都必须满足四个正式要求：

- `R1` 权威来源明确
- `R2` 语义不漂移
- `R3` 可回链
- `R4` 可区分视图与真值

#### `R1` 权威来源明确

每个关键对象都必须明确追溯到来源层：

- 事实状态来自 `M1`
- 周期状态来自 `M2`
- 行为状态来自 `M3`
- 偏差对象来自 `M4`
- 治理状态来自 `M5`

#### `R2` 语义不漂移

`M6` 不得在交付时改变对象原始语义。

允许：

- 附加解释
- 增加 tooltip
- 增加来源说明

禁止：

- 改写正式分类
- 替换正式理由
- 把局部语义改写为全局语义

#### `R3` 可回链

任一关键交付结果都应可追到原始正式对象。

#### `R4` 可区分视图与真值

必须能让用户明确区分：

- 这是正式对象本身
- 还是为了浏览便利而做的视图组织

### 6.2 页面交付规则

页面交付至少遵守：

- `P1` 页面组件必须绑定正式对象来源
- `P2` 页面上的关键状态必须显示层级归属
- `P3` 关键信号必须可点开查看原始理由对象
- `P4` 页面聚合卡片不得替代原始正式对象
- `P5` 局部/全局语义差异必须显式展示，不能隐藏

### 6.3 接口交付规则

API/数据接口至少遵守：

- `A1` 返回对象带来源层信息
- `A2` 关键状态保留原始分类字段
- `A3` 不得把多个不同语义对象混成模糊 summary 字段
- `A4` 关键 trace/版本字段不得被省略
- `A5` 研究级对象与正式对象必须有明确区分标记

### 6.4 告警交付规则

`M6` 不应把所有异常都推成告警。

首期告警至少分三类：

- `Alert-1` 数据与门禁风险
- `Alert-2` 协同与语义风险
- `Alert-3` 治理阻塞与人工关注风险

告警可以投影：

- `Freshness Proof` 未通过
- `interaction_guardrail_breach` 已触发
- `Promotion Blocker` 正在生效

告警不能追加：

- 交易建议
- 趋势终结判断
- 治理结论升级

### 6.5 回放交付规则

回放不是“看历史图表”，而是“重建当时系统正式对象链”。

回放必须至少支持：

- 当时的 `M1` 事实状态
- 当时的 `M2` 周期状态
- 当时的 `M3` 行为状态
- 当时的 `M4` 评估对象
- 当时的 `M5` 治理状态（若已存在）

#### 回放语义护栏

- 回放只显示对应时点已经正式存在的对象
- 不得把后来形成的理解回写成当时页面信息
- 若对象后来发生重算或升级，必须显式显示版本差异，不能默默覆盖历史画面

### 6.6 解释性文本规则

解释性文字不能替代正式对象。

更合理的方式是：

- 正式对象原文/原字段在前
- 解释性文本在后
- 解释性文本必须指向来源层
- 解释性文本不能发明上游没有给出的结论

### 6.7 单股页面特殊护栏

单股页必须强制显示：

- 上游数据是否可靠
- 当前周期状态来自 `M2`
- 当前行为状态来自 `M3`
- 当前偏差对象来自 `M4`
- 当前治理状态来自 `M5`

### 6.8 样本池页面特殊护栏

样本池页不得为了汇总方便压平正式分类。

### 6.9 系统总览页特殊护栏

系统总览页应聚焦：

- 正式风险总览
- blocker 总览
- 诊断链堆积情况
- 样本池问题聚类
- 关键 guardrail 状态

不应充斥大量无法追溯的装饰性指标。

## 7. 第一阶段实施范围、非目标与验证等级

`M6` 首期的目标不是“把所有信息都做成漂亮界面”，而是先建立高保真、可回链、可区分层级来源的正式交付主链。

### 7.1 第一阶段锚点

`M6` 首期边界必须锚定到五个前提：

- `M1` 已能输出正式事实与质量状态对象
- `M2` 已能输出 `small_cycle` 主链相关正式周期对象
- `M3` 已能输出正式行为状态、生命周期与归因对象
- `M4` 已能输出正式差距对象、样本摘要与 trace 对象
- `M5` 已能输出正式诊断链、治理对象与 blocker 状态

### 7.2 第一阶段必须完成

- `P1` 正式输入收口
- `P2` 三种正式观测视角
- `P3` 单股逻辑链总览
- `P4` 高价值告警投影
- `P5` 回放与追踪入口
- `P6` 语义护栏

### 7.3 第一阶段聚焦的核心交付问题

- `Q1` 用户能否在单股视角下看见完整逻辑链
- `Q2` 同一正式对象在不同页面、接口、报表中是否保持同一语义
- `Q3` `M3` 的局部退出语义与全局终局语义是否会在展示层被混淆
- `Q4` `M4/M5` 的偏差对象和治理对象能否真正被用户看见、查到、追到
- `Q5` 回放看到的是否是“当时系统知道的东西”，而不是“事后总结版页面”

### 7.4 第一阶段明确不做

- 不追求全量 BI 化仪表盘宇宙
- 不追求任意维度自由拖拽式分析平台
- 不追求前端自行补全上游缺失语义
- 不追求一次性做完所有报表、告警、页面和程序化消费面
- 不把研究级对象包装成正式生产对象
- 不把 `M6` 做成新的分析或治理判断层

### 7.5 验证等级

`M6` 首期采用三层验证等级：

- `V1 Projection Ready`
- `V2 Delivery Ready`
- `V3 Trusted Observability`

`V1 Projection Ready`

- 已能把部分正式对象稳定展示出来，但回链、来源标记、视角统一仍不完整。

`V2 Delivery Ready`

- 已能稳定交付正式对象，支持统一视角、来源标记、回链能力和高价值风险投影。

`V3 Trusted Observability`

- 已经过持续使用验证，证明不同交付面之间语义一致、回放可靠、告警克制、用户可高效定位问题。

### 7.6 首期目标等级与最低验收标准

首期目标明确设为：

- `V2 Delivery Ready`

最低验收标准至少包括：

- `A1` 只基于 `M1-M5` 的正式对象交付页面、接口、告警与回放
- `A2` 单股视角能完整串起 `M1 -> M2 -> M3 -> M4 -> M5` 的逻辑链
- `A3` 样本池视角能展示正式样本分类、主要偏差分布与协同风险
- `A4` 系统视角能展示正式风险、blocker、治理堆积与关键 guardrail 状态
- `A5` 页面、接口、告警、回放都保留来源层标记与原始正式分类字段
- `A6` `M3` 的 `local_exit_semantics` 与 `global_thesis_end_semantics` 在交付层保持严格分离
- `A7` 回放视图只显示对应时点正式存在的对象，不得把未来理解回写进历史画面

## 8. 当前结论

本文档冻结以下结论：

- `M6` 是正式对象的统一交付与观测层，不是分析层、治理层或新的真值层。
- `M6` 只能高保真投影 `M1-M5` 的正式对象，不得生成新的事实真值、周期真值、交易真值或治理真值。
- `M6` 必须仅基于正式对象构建单股、样本池和系统级三种统一观测视角。
- 同一正式对象在不同视图下可以改变展现形式，但不得改变其正式语义。
- `M6` 的页面、接口、告警与回放只能作为 `M1-M5` 正式对象的高保真交付形式。
- `M6` 首期应以 `Delivery Ready` 为目标，而不是过早宣称自己已具备长期可信的完整观测体系。
