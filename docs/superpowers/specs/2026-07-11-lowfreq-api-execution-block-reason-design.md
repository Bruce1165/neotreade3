# Lowfreq API Execution Block Reason Design

Date: 2026-07-11

## 1. Goal

This design covers the next narrow slice after the `buy signal audit contract` extraction.

This slice freezes only the duplicate `execution_block_reason` normalization that still lives inline in:

- [main.py](file:///Users/mac/NeoTrade3/apps/api/main.py)
  - `_lowfreq_normalize_execution_block_reason(...)`
  - the blocked-branch consumption inside `_lowfreq_execution_contract_from_intent(...)`

The goal is to:

- reuse the canonical shared normalization already owned by `neotrade3/decision_engine/buy_signal_audit_contract.py`
- keep API-only aliases in `apps/api/main.py`
- preserve current API-emitted `execution_block_reason` strings exactly
- add direct API-focused tests for the projection contract

This design is not:

- a rewrite of the shared owner
- a rewrite of `_lowfreq_execution_contract_from_intent(...)`
- a rewrite of intent lifecycle or execution orchestration
- a rewrite of report-side attribution logic

Project-phase note:

- domain: `API execution block reason projection`
- change type: `migration / refactor`
- NeoTrade2 remains reference only, not an active dependency
- dual-axis target: `M3 / G6`

## 2. Scope

Included:

- API reuse of the shared `normalize_execution_block_reason(...)`
- API-local supplement for aliases that are only produced inside `apps/api/main.py`
- one narrow API-focused test carrier for normalization and blocked execution contract emission

Excluded:

- changes to `neotrade3/decision_engine/buy_signal_audit_contract.py` behavior
- action field or funnel-stage mapping changes
- execution intent state transition changes
- report or frontend copy changes

## 4. Approach Options

### Option A: Reuse the shared normalization only for the proven common aliases, then apply API-local supplements for API-only aliases (Recommended)

- keep `_lowfreq_normalize_execution_block_reason(...)` in API as a thin wrapper
- delegate only the proven shared alias subset to `normalize_execution_block_reason(...)`
- keep `signal_expired` and API-local failure aliases in the API wrapper

Pros:

- removes true duplication without broadening behavior
- preserves the API-only contract surface
- fits the existing narrow-slice discipline

Cons:

- the API wrapper remains as a small projection shim

### Option B: Replace the API helper directly with the shared owner for every raw reason

Pros:

- smaller local helper

Cons:

- broadens behavior beyond current API evidence
- would normalize extra raw reasons not currently proven on the API path

### Option C: Expand the shared owner so it knows all API-only aliases

Pros:

- one global helper would cover more consumers

Cons:

- moves API projection-only details into the canonical M3 owner
- blurs the owner boundary without evidence of cross-layer reuse

Decision:

- choose Option A

## 5. Design

### 5.1 Ownership Decision

Canonical ownership remains in:

- `neotrade3/decision_engine/buy_signal_audit_contract.py`

API ownership remains limited to:

- projection-only alias supplement for reasons produced inside `apps/api/main.py`
- blocked execution contract assembly in `_lowfreq_execution_contract_from_intent(...)`

### 5.2 API Wrapper Boundary

`_lowfreq_normalize_execution_block_reason(...)` should become a thin projection shim:

1. normalize raw input to `str(...).strip()`
2. if the raw reason is one of the proven shared aliases, delegate to `normalize_execution_block_reason(...)`
3. if the raw reason is `signal_expired`, map to `entry_window_missed`
4. if the raw reason is one of the API-only local failures, map to `execution_rule_blocked`
5. otherwise passthrough unchanged

The proven shared alias subset is frozen to:

- `no_slots`
- `reserved_due_to_full_book`
- `no_cash`
- `reservation_expired`
- `pending_conflict_older_intent_wins`

The API-local supplement is frozen to:

- `signal_expired -> entry_window_missed`
- `no_open_price -> execution_rule_blocked`
- `no_price -> execution_rule_blocked`
- `invalid_code -> execution_rule_blocked`
- `already_holding -> execution_rule_blocked`
- `position_missing -> execution_rule_blocked`
- `no_shares -> execution_rule_blocked`
- `abandoned -> execution_rule_blocked`

### 5.3 Behavior Preservation Rules

This slice must preserve current API semantics exactly:

- blocked buy/sell/abandon intents still emit the same `execution_block_reason`
- `_lowfreq_execution_contract_from_intent(...)` still decides `action_type`, `order_action`, `reserve_action`, and `execution_status` locally
- successful non-blocked intents still emit an empty `execution_block_reason`
- unknown reasons still pass through unchanged

### 5.4 Testing Strategy

Add one API-focused carrier:

- `tests/unit/test_lowfreq_api_execution_block_reason.py`

Minimum cases:

- shared aliases still normalize to the current canonical buckets
- API-local aliases still normalize to current buckets
- unknown reason still passes through
- blocked execution contract still emits `execution_rule_blocked` for an API-local failure

## 6. Risks and Guardrails

Main risk:

- silently broadening API behavior by delegating raw reasons that are not proven on the API path

Guardrail:

- delegate only the frozen common alias subset

Secondary risk:

- moving API-only details into the shared M3 owner

Guardrail:

- keep local-only aliases in `apps/api/main.py`

## 7. Implementation Outline

Planned steps:

1. import `normalize_execution_block_reason(...)` into `apps/api/main.py`
2. turn the API helper into a thin wrapper over the proven common subset plus local supplements
3. add API-focused tests
4. run focused verification

## 8. Success Criteria

This slice is complete when:

- API reuses the canonical shared normalization for the proven common subset
- API-only aliases remain local
- emitted API `execution_block_reason` values stay unchanged
- API-focused tests pass
- syntax verification passes
