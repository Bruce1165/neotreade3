# App 导航 IA 收口设计

日期：2026-07-09

## 1. 目标

本切片只解决一个问题：

把 `App.jsx` 中侧边栏导航的信息架构（Information Architecture, IA）收口到当前已经成形的 3.0 工作台入口集合，并为该导航 contract 建立独立测试载体。

本切片的目标不是重写整个 `App` 壳层，也不是顺手整理 Header 文案、全局错误横幅或其他壳层 copy，而是只处理：

- 侧边栏入口集合
- 当前路由高亮规则
- 对应的独立导航测试

## 2. 范围

包含：

- `neotrade3-dashboard/src/App.jsx` 中 `Sidebar()` 的导航入口集合
- `Sidebar()` 当前 active 命中规则
- 一个新的独立测试文件，例如 `neotrade3-dashboard/src/App.navigationIa.test.jsx`

不包含：

- `Header()` 中 `API: 本地模式` / `当前环境：本地工作区` 的环境文案
- `GlobalBanner()` 的 Tushare 黄旗逻辑
- `GlobalApiErrorBanner()` 的环境不可达横幅逻辑
- `App.test.jsx` 中已有的共享 copy / 告警测试主题重整
- 路由表本身的增删
- 各页面内部 UI 变更

## 3. 现有证据

当前仓库中的可验证证据如下：

- `App.jsx` 的侧边栏当前 drift 把入口集合从：
  - `今日总览`
  - `低频交易`
  - `筛选器`
  - `单股核验`
  调整成：
  - `今日总览`
  - `主线审阅`
  - `选股工作台`
  - `运维中心`
- 同一批 drift 中，导航 active 判断也从“仅精确匹配”改成了“允许子路径前缀命中”
- `Header()` 环境文案同时也发生了变化，但这与导航 IA 属于另一条主题线
- `App.test.jsx` 当前把共享 copy、导航文案、环境不可达横幅等断言混在一个测试文件中，不利于继续收口 `App` 导航主题

因此，当前问题不是“`App` 全部壳层都需要重做”，而是“导航 IA 已经形成一条可独立收口的窄线，但仍与文案和测试漂移混在一起”。

## 4. 职责归属

本切片职责属于 `App` 壳层中的导航 contract。

它只回答两个问题：

- 侧边栏应当暴露哪些 3.0 一级入口
- 哪些路径应当视为该入口的 active 子路径

它不回答以下问题：

- 当前环境文案是否应统一改成“本地工作区”
- 全局 API 错误横幅是否需要继续扩写
- `App.test.jsx` 是否应整体拆分
- 页面标题、副标题是否需要调整

因此，这一刀应明确叫做“导航 IA 收口”，而不是“App shell 总整理”。

## 5. 方案比较

### 5.1 方案 A：继续扩写 `App.test.jsx`

做法：

- 在现有 `App.test.jsx` 中追加导航入口与 active 断言

优点：

- 文件数最少

缺点：

- 继续扩大 omnibus 测试文件
- 混淆导航 IA、共享文案、错误横幅三种主题
- 不符合当前项目已经形成的独立测试载体原则

### 5.2 方案 B：导航 IA 独立收口

做法：

- 只修改 `App.jsx` 的 `Sidebar.navItems`
- 只修改 `Sidebar` 的 active 命中规则
- 新增 `App.navigationIa.test.jsx`

优点：

- 边界最清晰
- 导航主题与共享 copy / 错误横幅分开
- 后续如要继续处理 `Header` 或环境横幅，不会互相污染

缺点：

- 新增一个测试文件

### 5.3 方案 C：App shell 一次性并刀

做法：

- 导航 IA、Header 文案、全局横幅测试一起收口

优点：

- 一次性覆盖更多 drift

缺点：

- 明显超出当前最窄线
- 难以形成干净原子提交

### 5.4 选择

本切片建议采用方案 B。

## 6. 设计要求

### 6.1 导航入口集合

侧边栏本轮应只保留以下一级入口：

- `今日总览`
- `主线审阅`
- `选股工作台`
- `运维中心`

这些入口已经与当前前端工作流的主使用面更贴近，也与此前已落地页面形成较清晰的工作台分组。

本切片不要求把：

- `筛选器`
- `单股核验`

保留为一级入口。

这些能力可以继续由更具体页面内部入口、工作台内部跳转或后续 IA 线处理，而不是在本刀中扩成更多导航决策。

### 6.2 active 规则

导航 active 判断应满足：

- `/` 精确命中 `今日总览`
- `/market-intelligence` 命中 `主线审阅`
- `/lowfreq` 以及 `/lowfreq/...` 命中 `选股工作台`
- `/ops` 以及 `/ops/...` 命中 `运维中心`

目的不是做通用路由匹配框架，而是保证一级导航与其子路径页面在视觉上保持一致高亮。

### 6.3 不修改 Header 文案

尽管当前 drift 中 `Header()` 文案已有变化，但本切片必须明确排除：

- `API: 本地模式`
- `当前环境：本地工作区`

这类环境 copy contract。

原因：

- 它们不影响导航 IA 是否成立
- 单独成线更符合“边界清晰”的原则

### 6.4 不改全局横幅逻辑

本切片不接触：

- `GlobalBanner()`
- `GlobalApiErrorBanner()`

也不处理其对应测试。

否则本刀会从“导航 IA”扩成“App 全局状态可视化整理”。

## 7. 测试设计

### 7.1 测试载体

建议新增独立文件：

- `App.navigationIa.test.jsx`

原因：

- `App.test.jsx` 已承载共享 copy 和告警主题
- 导航 IA 应有自己的独立测试边界

### 7.2 最小断言集合

推荐最小断言集合：

- 侧边栏只显示 4 个目标入口
- 不再显示 `筛选器`
- 不再显示 `单股核验`
- 当前路径为 `/lowfreq` 时，`选股工作台` 为 active
- 当前路径为 `/lowfreq/backtest-reports/:id` 时，`选股工作台` 仍为 active
- 当前路径为 `/ops` 时，`运维中心` 为 active

### 7.3 不扩展的测试范围

本切片测试不应扩展为：

- Header 文案测试
- 环境不可达横幅测试
- Tushare 黄旗测试
- 各页面内容测试

## 8. 非目标

本切片明确不处理：

- Header 环境文案收口
- 全局 API 错误横幅测试整理
- `App.test.jsx` 的全面拆分
- 路由表新增或删除页面
- 页面间深层导航策略

这些都属于后续独立主题线。

## 9. 提交边界

目标实现提交只允许包含：

- `App.jsx` 中与 `Sidebar` 导航 IA 直接相关的最小修改
- 新增 `App.navigationIa.test.jsx`
- 如确有必要，对现有 `App.test.jsx` 做极小兼容处理

必须排除：

- `Header()` 文案变更
- `GlobalBanner()` / `GlobalApiErrorBanner()` 逻辑调整
- `App.test.jsx` 中告警主题扩写
- 其他页面文件

## 10. 验证要求

预期最小验证闭环：

- 运行新的 `App.navigationIa.test.jsx`
- 如需，补跑现有 `App.routeGuard.test.jsx`
- 确认导航入口集合与 active 规则符合设计
- 确认本刀不带入 Header 或全局横幅主题线

## 11. 结论

本切片的本质不是“整理整个 `App` 壳层”，而是把已经成形的导航结构变成一个独立、可验证、可提交的 IA contract。

只有先把导航这一刀单独收口，后续再处理 Header 文案或全局告警时，才不会互相污染边界。
