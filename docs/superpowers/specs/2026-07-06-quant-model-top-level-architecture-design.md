# 量化模型顶层架构总蓝图设计

日期：2026-07-06

## 1. 文档目标

本文档用于冻结 NeoTrade3 量化模型体系的顶层结构、模块边界、信息流、依赖纪律与优先级，不涉及模块内部算法细节和具体代码实现。

本设计回答以下问题：

- 整个量化体系应拆成哪些一级模块。
- 哪些模块属于模型核心，哪些属于治理与交付系统。
- 每一层的数据来自哪里、通过什么手段获取、向下游输出什么。
- 哪些层可以依赖哪些层，哪些反向依赖必须禁止。
- 哪些能力必须先设计，哪些模块必须先落地。

## 2. 设计背景

当前项目中的低频引擎已具备一定的交易决策与回测能力，但多个职责仍物理上和逻辑上混杂在同一引擎文件中，尚未形成可持续演化的模型体系。

现状特征：

- 已有 `identify -> tracking -> entry -> hold -> exit` 的交易主链雏形。
- 已有局部 `wave_phase` 启发式识别，但尚未形成独立的多尺度周期识别系统。
- 已有回测结果、审计日志、运维页面等基础设施，但尚未形成独立的 benchmark 和演化闭环。
- 事实输入、周期识别、交易决策、回测评价、运维交付等职责大量混合在 `lowfreq_engine_v16_advanced.py` 中。

用户确认的核心方向：

- 模型必须形成清晰的周期认知系统，而不只是若干买卖规则。
- `1浪 -> C浪` 应被视为完整周期，不同小周期之间具有连续性与衔接性。
- 若干个完整小周期可以递归组合为一个中周期；若干个中周期又可以递归组合为更大周期。
- 最大周期（例如康波周期）的判断属于全体系最高难度目标，但系统必须为这种递归周期认知预留正式结构。
- 强调周期结构的目的不是让系统僵硬化，而是让系统更清晰地知道“每天所处的位置”。
- 人工标注只能用于比对，不能作为模型真值依据。
- benchmark 的第一真值源应来自历史牛股全程回放与逐日阶段快照。
- 模块边界必须清晰，后续每个部分可以独立进化、独立优化、独立验收。
- 任何需要人工关注的事项必须显式暴露，不允许埋在日志里。

## 3. 顶层设计原则

本体系以以下原则为总纪律：

### 3.1 事实优先

- 所有上层判断都必须建立在完整、真实、有效、可追溯的事实基础之上。
- `M1` 的能力上限决定 `M2/M3/M4/M5` 的能力上限。
- 如果上游事实不足，下游层必须降级目标，而不能靠猜测补齐。

### 3.2 模块独立

- 每一层只回答一类问题。
- 每一层都必须有明确输入、明确输出、明确禁止事项。
- 下游层不得隐式重算上游真值，也不得静默改写上游结论。

### 3.3 周期驱动

- 交易决策层不再被视为原始真相的生产者，而是周期识别层的消费者。
- 周期识别系统必须逐步从局部波段标签提升为小周期/中周期/长周期的多尺度状态系统。
- 多尺度周期不是平行标签集合，而是递归组合关系：多个小周期组成中周期，多个中周期组成更大周期。
- 周期识别层必须能够表达“当前所处位置”以及不同尺度周期之间的衔接关系，而不是只输出孤立阶段名词。

### 3.4 Benchmark 独立

- benchmark 不是模型本体的一部分，而是模型的外部检验系统。
- benchmark 只能评估差距，不能反向充当事实源或模型真值源。

### 3.5 治理可见

- 所有需要人工关注、批准、补数据、改规则的事项，必须形成正式 Attention Item。
- 注意事项必须在前端治理页面可见，而不是只存在于后端日志或临时报告中。

### 3.6 杜绝漂移

- 任何模块设计都必须显式说明：目标、证据、边界、依赖、禁止事项、未决问题、当前结论。
- 没有证据就不能当事实，没有前提就不能下顺序结论。

## 4. 六层总体结构

本体系拆分为六个一级模块：

- `M1 事实层 Fact Layer`
- `M2 周期识别层 Cycle Intelligence`
- `M3 交易决策层 Decision Engine`
- `M4 Benchmark 层`
- `M5 演化控制层 Evolution Controller`
- `M6 交付与观测层 Delivery / Observability`

其中：

- `M1` 是事实地基。
- `M2 + M3` 构成模型核心。
- `M4 + M5` 构成模型治理系统。
- `M6` 构成交付与观测系统。

## 5. 各层定义

### 5.1 M1 事实层

职责：

- 接入所有外部数据。
- 做标准化、清洗、历史回补、口径统一与派生事实生成。
- 只输出事实，不输出周期结论和交易结论。

数据来源：

- 个股行情、指数行情、板块与行业映射、市场结构、资金参与代理、基金/机构行为代理、事件与主题信息、基础元数据。

获取手段：

- 外部数据服务接口拉取。
- 调度任务日更或定期同步。
- 派生计算生成标准化事实视图。

输出：

- 原始事实表。
- 派生事实表。
- 标准化视图。
- 数据质量状态。

禁止事项：

- 不输出 `小周期/中周期/长周期` 判断。
- 不输出 `entry/exit` 结论。
- 不因下游反馈而篡改已记录事实。

### 5.2 M2 周期识别层

职责：

- 消费 `M1` 事实层输出。
- 识别小周期、中周期、长周期所处阶段。
- 给出阶段置信度、转折证据、周期连续性与衔接关系。
- 识别不同尺度周期之间的递归组合关系，明确若干完整小周期如何构成中周期，若干中周期如何构成更大周期。
- 为最大周期判断预留正式结构与字段位置，即使其精度在早期阶段无法达到最终目标。

数据来源：

- 全部来自 `M1`。

获取手段：

- 基于事实序列和派生事实进行状态计算、聚合、连续性判断。

输出：

- 小周期状态。
- 中周期状态。
- 长周期状态。
- 跨尺度组合关系。
- 证据包。
- 置信度。
- 转折候选。
- 周期连续性状态。

禁止事项：

- 不直接决定买卖。
- 不使用 benchmark 结果重写周期真值。
- 不用收益结果修饰当日周期识别。

### 5.3 M3 交易决策层

职责：

- 消费 `M2` 的周期状态与 `M1` 的执行事实。
- 将其翻译为 `identify -> tracking -> entry -> hold -> exit` 决策主链。

数据来源：

- 周期状态来自 `M2`。
- 执行约束事实来自 `M1`。

获取手段：

- 决策规则映射。
- 风控与执行约束应用。
- 状态迁移记录。

输出：

- 候选状态。
- tracking 状态。
- entry / hold / exit 决策。
- 交易记录。
- 审计事件。
- 决策归因字段。

禁止事项：

- 不重新发明周期判断。
- 不把 benchmark 结果作为当日交易真值。
- 不直接定义模型达标标准。

### 5.4 M4 Benchmark 层

职责：

- 用历史牛股完整周期与逐日阶段快照定义目标达标标准。
- 对 `M2/M3` 的输出进行独立检验。

数据来源：

- `M1` 的事实。
- `M2` 的周期输出。
- `M3` 的交易输出。
- 历史牛股回放样本。
- 人工标注对照样本。

获取手段：

- 历史样本构建。
- 单票完整周期建模。
- 逐日快照比对。

输出：

- 周期识别偏差。
- 阶段偏差。
- 持有质量。
- 漏抓率。
- 卖飞率。
- 达标评分卡。

禁止事项：

- 不参与实时交易决策。
- 不反向定义事实真值。
- 不反向定义周期真值。

### 5.5 M5 演化控制层

职责：

- 检测漂移与偏差。
- 进行层级归因。
- 形成 Change Request 与 Attention Item。
- 管理实验、验证与升级门禁。

数据来源：

- `M1` 数据质量状态。
- `M2` 周期偏差。
- `M3` 决策偏差。
- `M4` benchmark 结论。

获取手段：

- 差距检测。
- 层级归因。
- 变更请求生成。
- 影子实验或回测验证。

输出：

- Drift 检测结果。
- Change Request。
- Attention Item。
- 实验结论。
- 升级门禁结论。

禁止事项：

- 不允许静默改写 `M1/M2/M3`。
- 不允许自动篡改模型真值。
- 不允许绕过人工批准直接将重大变更推入正式链路。

说明：

- `M5` 不只是监控层。
- `M5` 必须能够向 `M1/M2/M3` 发出正式的改进请求。
- 但这些请求必须通过治理流程生效，而不是原地自动回写。

### 5.6 M6 交付与观测层

职责：

- 通过 API、前端、报告、运维与治理页面向人交付权威输出。
- 确保 Attention Item 和治理状态可见。

数据来源：

- `M1-M5` 的权威输出接口。

获取手段：

- API 投影。
- 前端页面消费。
- 回测报告生成。
- 运维治理视图。

输出：

- 用户页面。
- 回测报告。
- 治理页面。
- Attention Queue。

禁止事项：

- 不在展示层自行推断上游真值。
- 不生成伪归因。
- 不将展示层反向变成真值源。

## 6. 前向生产流与反向改进流

### 6.1 前向生产流

系统的正式生产流为：

- `M1 -> M2 -> M3 -> M4 -> M5 -> M6`

其中：

- `M1 -> M2 -> M3` 构成决策主链。
- `M1 + M2 + M3 -> M4 -> M5 -> M6` 构成治理主链。

### 6.2 反向改进流

系统还必须支持治理驱动的反向改进流：

- `M5 -> Change Request / Attention Item -> M1/M2/M3`

这不是反向依赖，而是治理反馈传播。

允许的事情：

- `M5` 发现 `M2` 周期识别不足，提出 `M2` 改进请求。
- `M5` 发现 `M3` 持仓或退出偏差，提出 `M3` 规则优化请求。
- `M5` 发现事实不足，要求 `M1` 补充新的数据源、粒度或标准化视图。

禁止的事情：

- `M5` 静默回写 `M1` 事实。
- `M5` 静默覆盖 `M2` 周期真值。
- `M5` 不经治理直接修改 `M3` 正式决策逻辑。

## 7. 依赖与禁止反向依赖规则

### 7.1 每层允许依赖谁

- `M1`：仅依赖外部数据源和标准化规则。
- `M2`：仅依赖 `M1`。
- `M3`：依赖 `M1` 的执行事实和 `M2` 的周期状态。
- `M4`：依赖 `M1/M2/M3`。
- `M5`：依赖 `M1/M2/M3/M4`。
- `M6`：依赖 `M1-M5` 的权威输出接口。

### 7.2 硬性禁止规则

- `R1`：`M3` 不能反向改写 `M2`。
- `R2`：`M4` 不能反向定义 `M1/M2/M3` 的真值。
- `R3`：`M5` 不能静默在线改写模型核心。
- `R4`：`M6` 不能自行推断上游权威结论。
- `R5`：人工标注不能作为模型真值源。
- `R6`：任何下游层不得覆盖已记录的 `M1` 事实。
- `R7`：`M5` 可以向 `M1/M2/M3` 发出正式改进请求，但不得以静默反向变更替代治理流程。

## 8. M1 事实层的特殊地位

`M1` 不只是一个普通模块，而是全体系事实地基。

如果 `M1` 缺乏关键数据种类或足够颗粒度：

- `M2` 无法宣称具备强周期识别能力。
- `M3` 无法宣称具备高质量周期驱动决策能力。
- `M4` 无法建立可靠 benchmark。
- `M5` 无法形成真实有效的闭环。

因此，本体系正式采用以下硬约束：

- 没有对应数据能力，就不能宣称对应模型能力。
- 如果数据不足，只能降低目标，不允许输出伪精度。
- 新增更细粒度数据源时，应优先增强 `M1`，再决定是否升级 `M2/M3`。

## 9. M1 数据域框架

`M1` 的一级数据域定义为以下八类：

- `D1` 个股行情事实
- `D2` 指数与市场结构事实
- `D3` 板块/行业事实
- `D4` 资金与参与结构事实
- `D5` 基金/机构行为代理事实
- `D6` 事件与主题驱动事实
- `D7` 元数据与映射事实
- `D8` 派生事实与标准化视图

后续 `M1` 详细设计和接口审计，必须围绕这八类展开。

## 10. Attention Item 机制

Attention Item 是本体系的一级治理对象，必须在前端治理页面可见。

### 10.1 作用

- 暴露需要人工关注、批准、补数、改规则、处理风险的事项。
- 让治理闭环从“日志可见”升级为“流程可执行”。

### 10.2 输出来源

- `M5` 生成主要的 Attention Item。
- `M1` 数据审计阶段也会生成与数据缺口和数据质量有关的 Attention Item。

### 10.3 典型类别

- `Source Gap`
- `Source Quality Risk`
- `Normalization Conflict`
- `Cycle Recognition Gap`
- `Decision Logic Gap`
- `Benchmark Drift`
- `Promotion Blocker`

### 10.4 自动化分级

- `Type A`：全自动处理，无需人工关注。
- `Type B`：系统可生成建议，但需要人工批准。
- `Type C`：需要人工明确处理，系统只能提出请求。

### 10.5 最低字段

- `id`
- `created_at`
- `source`
- `target_layer`
- `issue_type`
- `severity`
- `automation_class`
- `evidence`
- `recommended_action`
- `human_action_required`
- `status`
- `owner`
- `blocking_scope`

### 10.6 前端要求

Attention Item 至少应在以下位置可见：

- 治理/运维页面的完整 Attention Queue。
- 每日摘要中的关键 Attention 统计和阻塞项。
- 相关模块页面中的上下文级 Attention 展示。

## 11. 现有实现到目标结构的映射

### 11.1 当前核心事实

当前多个职责集中在：

- `lowfreq_engine_v16_advanced.py`

主要映射如下：

- `_detect_wave_phase_from_series()`：`M2` 原型的一部分。
- `generate_buy_signals()`：`M3` 候选与入场决策的一部分。
- `_position_contract_snapshot()`：`M3` 持仓表达的一部分。
- `check_sell_signal_v2()`：`M3` 离场决策的一部分。
- `run_backtest()`：同时承担 `M3/M4/M5` 的部分职责。

### 11.2 结构判断

- `M1`：现有有素材，但没有独立层。
- `M2`：现有只有启发式原型，离目标差距最大。
- `M3`：是现有最完整的层，但需要从混合职责中逐步收口。
- `M4`：基本待新建。
- `M5`：有调度和巡检基础，但缺正式演化控制语义。
- `M6`：有交付平台，但缺来自上游的完整权威投影。

## 12. 设计顺序与落地顺序

### 12.1 设计顺序

- `D1` 顶层总蓝图设计
- `D2` `M1` 事实层详细设计
- `D3` `M2` 周期识别层详细设计
- `D4` `M4` Benchmark 层详细设计
- `D5` `M5` 演化控制层详细设计
- `D6` `M3` 决策层重构设计
- `D7` `M6` 交付与观测层设计

### 12.2 落地顺序

- `P0` 冻结总蓝图纪律
- `P1` 优先落地 `M1`
- `P2` 落地 `M2`
- `P3` 落地 `M4`
- `P4` 落地 `M5`
- `P5` 重构并接入 `M3`
- `P6` 升级 `M6`

原因：

- 不先冻结总蓝图，后续边界会再次漂移。
- 不先落地 `M1`，后续所有层都缺乏事实地基。
- 不先有 `M4`，`M5` 无法形成真实闭环。
- `M3` 作为现有最重的一层，只能在上游和治理层清晰后再收口。

## 13. 后续图纸顺序

本总蓝图冻结后，后续模块设计按以下顺序推进：

1. `M1` 数据域、外部接口、获取手段、标准化与审计框架
2. `M2` 多尺度周期识别系统设计
3. `M4` benchmark 样本与评分体系设计
4. `M5` 演化控制、Change Request、Attention Item 设计
5. `M3` 决策层与新上游契约对接设计
6. `M6` API、前端、运维治理投影设计

## 14. 当前结论

本设计冻结以下核心结论：

- NeoTrade3 量化体系应拆分为 `M1-M6` 六层结构。
- `M1` 是事实地基，`M2+M3` 是模型核心，`M4+M5` 是治理系统，`M6` 是交付系统。
- 模型能力声明必须服从数据能力声明。
- `M5` 不允许静默回写上游，但必须能够通过治理流程将改进请求向 `M1/M2/M3` 传播。
- Attention Item 是正式治理对象，必须在前端可见。
- 设计顺序与落地顺序必须分开定义，落地优先级应先打牢 `M1`。

## 15. 不在本文档中解决的问题

以下内容明确不在本总蓝图中展开：

- `M2` 的具体阶段算法。
- `M1` 的具体接口名单与当前适配结果。
- `M4` 的具体评分公式。
- `M5` 的具体实验门禁策略。
- `M3` 的具体规则集重构方案。
- `M6` 的具体页面与 API 字段设计。

这些内容将进入各自子模块的详细设计文档。
