# M1 第一阶段 + M2/M3 最小闭环 Phase 0-1 执行任务清单

日期：2026-07-07  
对应计划：`docs/superpowers/specs/2026-07-07-m1-m2-m3-minimal-closed-loop-implementation-plan.md`

## 1. 任务清单目标

本清单只覆盖实施蓝图中的：

- `Phase 0`：现实盘点与契约冻结
- `Phase 1`：M1 第一阶段正式供给最小闭环

本清单的目标不是直接编码，而是把第一批工作压缩到：

- 明确的文件级落点
- 明确的对象级交付物
- 明确的测试与验证动作
- 明确的阻塞项与决策点

本清单不覆盖：

- `M2` small_cycle 状态机编码实现
- `M3` 行为状态机编码实现
- 完整 API/前端接线
- `M4/M5/M6` 的正式工程实现

## 2. Phase 0：现实盘点与契约冻结

### 2.1 目标

- 明确首批正式对象的真实上游来源。
- 明确首批不会改动的模块，避免范围蔓延。
- 为 `Phase 1` 的事实供给收口建立稳定前提。

### 2.2 任务组 A：M1 现实供给链盘点

#### Task A1：盘点 `D1` 现实供给链

目标：

- 确认当前哪些表、流程、接口已经构成 `D1` 的真实供给主链。

建议检查位置：

- `neotrade3/data_control/pipeline.py`
- `apps/api/main.py`
- `docs/operations/production_task_registry.md`
- 与日线更新相关的 launchd/template/scheduler 文件

产出：

- `D1` 现实供给清单
- `D1` 当前主真相源说明
- `D1` 当前已存在但尚未正式化的字段清单

完成判定：

- 能明确回答：`D1` 当前到底由谁生产、在哪更新、哪些字段可正式消费。

#### Task A2：盘点 `D7` 现实供给链

目标：

- 确认交易日历、股票元数据、映射状态等是否已经具备稳定来源。

建议检查位置：

- `apps/api/main.py`
- 现有元数据相关更新入口
- `stocks`/映射/日历相关表或接口

产出：

- `D7` 现实供给清单
- 当前缺口说明
- 哪些字段可直接进入首批正式契约

完成判定：

- 能明确回答：`D7` 中哪些对象已经足够正式，哪些仍是能力缺口。

#### Task A3：盘点 `D8` 现实派生链

目标：

- 确认当前哪些派生事实已经稳定存在，哪些仍散落在引擎或页面层。

建议检查位置：

- `lowfreq_engine_v16_advanced.py`
- `neotrade3/analysis/`
- `apps/api/main.py`
- 任何当前直接在回测或页面中临时计算的派生字段

产出：

- `D8` 首批候选字段清单
- “可进入 `M1` 正式派生” 与 “暂不进入首批” 的分界

完成判定：

- 形成首批 `D8` 最小字段集，不再模糊说“派生以后再看”。

### 2.3 任务组 B：M2/M3 现实资产盘点

#### Task B1：盘点现有引擎中可映射到 `M2` 的周期资产

目标：

- 找出当前哪些字段、标签、判断块可被映射到 `cycle_state / evidence_bundle / invalidation`。

建议检查位置：

- `lowfreq_engine_v16_advanced.py`
- 相关测试文件：
  - `tests/unit/test_lowfreq_engine_v16_signal_convergence.py`
  - `tests/unit/test_lowfreq_engine_v16_sell_logic.py`

产出：

- “现有周期启发式 -> M2 正式对象”映射表
- 明确哪些逻辑是可复用资产，哪些只能视为旧实现残留

完成判定：

- 不再需要靠印象描述“引擎里大概已有周期判断”。

#### Task B2：盘点现有引擎中可映射到 `M3` 的行为资产

目标：

- 找出当前哪些状态、事件、审计字段可直接映射到：
  - `identify_state`
  - `tracking_state`
  - `entry_state`
  - `hold_state`
  - `exit_state`
  - `decision_lifecycle_log`

建议检查位置：

- `lowfreq_engine_v16_advanced.py`
- 当前 buy/sell audit 结构
- 现有 lifecycle / event / contract 相关输出

产出：

- “现有行为链 -> M3 正式对象”映射表
- 当前最缺的字段清单

完成判定：

- 能明确说明 `M3` 是“复用哪些已有资产 + 新增哪些正式对象”。

### 2.4 任务组 C：对象与文件落点冻结

#### Task C1：冻结首批正式对象最小集合

目标：

- 把 `Phase 1` 之后必须存在的对象名称、层级归属和最小字段边界固定下来。

冻结对象：

- `M1`
  - 事实对象
  - 质量状态对象
  - `Freshness Proof`
  - `Attention Item`
- `M2`
  - `cycle_state`
  - `state_transition_log`
  - `evidence_bundle`
  - `confidence`
  - `invalidation`
- `M3`
  - `identify_state`
  - `tracking_state`
  - `entry_state`
  - `hold_state`
  - `exit_state`
  - `decision_lifecycle_log`

产出：

- 一份对象最小集合表
- 每个对象的来源层与消费层说明

完成判定：

- 首批对象不再处于“边实现边想名字”的状态。

#### Task C2：冻结文件级改动边界

目标：

- 确认第一批会改哪些文件，不会碰哪些文件。

第一批建议可改：

- `neotrade3/data_control/` 相关模块
- `apps/api/main.py`
- `config/data_control/` 或相关 schema/contract 文件

第一批建议暂不大改：

- 前端主页面
- `M4/M5/M6` 文档外的治理/展示工程
- 不相关调度和历史遗留功能

完成判定：

- 第一批实施不会变成全仓库扩散式修改。

### 2.5 Phase 0 验收条件

- 已形成 `D1/D7/D8` 现实供给盘点结果
- 已形成 `M2/M3` 现实资产映射结果
- 已冻结首批正式对象最小集合
- 已冻结第一批文件级改动边界

## 3. Phase 1：M1 第一阶段正式供给最小闭环

### 3.1 目标

- 将 `D1 + D7 + D8` 收口为 `M2/M3` 可正式消费的最小 `M1` 输出。

### 3.2 任务组 D：正式对象边界定义

#### Task D1：定义 `D1` 最小正式事实对象

目标：

- 固定个股日线行情对象的最小字段契约。

至少应明确：

- 主键/标识
- 时间字段
- 价格字段
- 成交相关字段
- `updated_at` / 数据版本相关字段
- 可回放所需字段

产出：

- `D1` 最小字段清单
- 字段来源说明
- 缺口字段清单

完成判定：

- `D1` 不再是模糊概念，而是可写 schema 的正式对象。

#### Task D2：定义 `D7` 最小正式事实对象

目标：

- 固定元数据、交易日历、映射状态的最小正式对象。

至少应明确：

- 股票基础标识
- 是否可交易/是否停复牌
- 交易日历有效性
- 映射/元数据必要字段

产出：

- `D7` 最小字段清单
- 字段来源与刷新逻辑说明

完成判定：

- `M2/M3` 后续可不再绕开 `M1` 临时读取元数据。

#### Task D3：定义 `D8` 最小正式派生对象

目标：

- 只收口首批真正被 `M2/M3` 需要的派生事实。

要求：

- 禁止把未来用得到的所有派生一次性打包进来
- 禁止把引擎内临时计算继续留在 `M3` 或页面层

产出：

- `D8` 首批字段清单
- 每个字段为何属于 `M1` 而不是 `M2/M3`

完成判定：

- `D8` 成为最小够用的正式派生层，而不是新的杂物层。

### 3.3 任务组 E：质量门禁与 Freshness Proof

#### Task E1：定义首批质量门禁字段

目标：

- 为 `D1/D7/D8` 建立最小质量状态对象。

首批至少包括：

- `Source Status`
- `Freshness Status/Proof`
- `Coverage Status`
- `Replay Status`

产出：

- 质量状态最小对象定义
- 对应失败时的下游消费规则

完成判定：

- 能明确回答“什么情况下 `M2/M3` 可以正式消费，什么情况下不能”。

#### Task E2：定义首批 Freshness Proof 规则

目标：

- 将“数据存在”与“数据就绪”正式分开。

至少应明确：

- 每类对象的新鲜度判断口径
- 判定依赖哪些字段
- 失效时如何暴露给下游

产出：

- 首批 `Freshness Proof` 规则清单

完成判定：

- 不再出现“抓到数据就等于可消费”的隐式假设。

### 3.4 任务组 F：Attention 与阻塞暴露

#### Task F1：定义首批 `Attention Item` 模板

目标：

- 把 `M1` 的问题正式结构化，而不是留在日志或口头说明中。

首批模板至少覆盖：

- 数据缺失
- 数据过期
- 覆盖不足
- 无法回放
- 派生能力缺口

产出：

- `Attention Item` 最小字段模板
- 严重度与影响层说明

完成判定：

- 首批 `M1` 问题可正式暴露，并能被未来 `M5/M6` 消费。

#### Task F2：定义下游阻断规则

目标：

- 明确哪些 `M1` 问题会阻断 `M2/M3` 正式消费。

产出：

- 消费阻断矩阵：
  - 哪类问题只告警
  - 哪类问题必须阻断

完成判定：

- `M2/M3` 不会在 `M1` 质量失败时静默继续工作。

### 3.5 任务组 G：文件级实施顺序

#### Task G1：第一批文件修改顺序冻结

建议顺序：

1. `config/data_control/` 或 contracts/schema 位置  
2. `neotrade3/data_control/` 相关对象与质量输出  
3. `apps/api/main.py` 的最小对象投影  
4. 对应 `tests/unit` 或新建契约测试文件

完成判定：

- 文件改动顺序和依赖顺序一致，不会先改 API 再倒逼对象设计。

### 3.6 任务组 H：测试清单

#### Task H1：M1 首批契约测试清单冻结

首批至少包括：

- `D1` 字段契约测试
- `D7` 字段契约测试
- `D8` 首批派生字段契约测试
- `Freshness Proof` 判定测试
- 质量失败时的消费阻断测试

完成判定：

- `Phase 1` 的完成不是口头确认，而是测试可证明。

## 4. Phase 0-1 阻塞项

当前需要提前盯住的阻塞项：

- `B1`：`D7` 现实供给链是否足够稳定，如果不稳定，`M3` 后续执行约束会失真
- `B2`：`D8` 首批字段是否能收得足够小，否则 `Phase 1` 会失控膨胀
- `B3`：现有 API/表结构是否已有隐式旧口径，可能与新正式对象冲突
- `B4`：是否需要先建立统一 schema/contracts 放置位置，否则后续对象定义会分散

## 5. Phase 0-1 完成标志

满足以下条件时，视为 `Phase 0-1` 可以结束并进入 `Phase 2`：

- 已完成 `D1/D7/D8` 现实供给与字段契约冻结
- 已完成 `M1` 首批质量状态与 `Freshness Proof` 规则冻结
- 已完成首批 `Attention Item` 与消费阻断规则冻结
- 已完成第一批文件改动顺序与测试清单冻结
- `M2` 可以开始基于正式 `M1` 对象设计 `small_cycle` 首批工程落点

## 6. 下一步建议

这份清单确认后，下一步最合理的是继续拆：

- `Phase 2-3` 执行任务清单

也就是：

- `M2` small_cycle 首批工程任务
- `M3` 行为状态机与低频纪律首批工程任务

这样可以在不直接编码的情况下，把第一批真正的实施顺序完整排出来。
