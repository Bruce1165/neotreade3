# NeoTrade3 全站状态语义统一与 Hover 释义设计（设计定稿）

日期：2026-06-18  
范围：当前主前端 `neotrade3-dashboard` 中与交易状态、风险状态、执行队列状态相关的页面展示

## 1. 背景与目标

- 当前前端对同一底层状态存在文案冲突。
- 例如同一个 `risk_level === 'exit'`，在 `Overview` 中被显示为“卖出信号”，而在 `Lowfreq` 中又被显示为“回避”。
- 当前页面也已经存在多类细状态并行出现的情况，例如：
  - `可出手`
  - `跟随观察`
  - `大盘不配合`
  - `题材不配合`
  - `动能不足`
  - `回避`
  - `已排队`
  - `已放弃`
- 如果直接只统一颜色而不统一语义，会继续放大“同色不同义”或“同义不同色”的问题。

目标：
- 在不扩大到后端契约改造的前提下，统一前端全站的状态语义分层。
- 保留当前细状态文案，不粗暴压缩为 4 个固定文案。
- 为状态标签、风险标签、执行队列状态提供统一 hover 释义，帮助用户理解每个术语的业务含义。
- 让颜色只表达一级状态，让 tooltip 补充细状态解释与触发条件。

## 2. 设计范围

本次范围：
- `neotrade3-dashboard/src/pages/Lowfreq.jsx`
- `neotrade3-dashboard/src/pages/Overview.jsx`
- `neotrade3-dashboard/src/pages/StockCheck.jsx`
- `neotrade3-dashboard/src` 下新增的公共状态语义字典与 tooltip/标签组件

本次覆盖的展示对象：
- 状态标签
- 风险标签
- 执行队列状态

非目标：
- 不修改后端接口字段名。
- 不一次性重构所有页面的文案系统。
- 不扩展到普通表头、说明正文、数值字段。
- 不在本次范围内引入新的全站设计体系或复杂弹层系统。

## 3. 当前证据与语义冲突

### 3.1 当前前端已有状态来源

- `Lowfreq` 页面已有状态判断逻辑：
  - `buy_signal === true` -> `可出手`
  - `manual.buy_intent_pending === true` -> `已排队`
  - `manual.abandoned === true` -> `已放弃`
  - `roleLabel === '跟随'` -> `跟随观察`
  - `upwave_gate.market_ok === false` -> `大盘不配合`
  - `upwave_gate.concept_ok === false` -> `题材不配合`
  - `upwave_gate.stock_ok === false` -> `动能不足`
  - `risk_level === 'exit'` -> `回避`

- `Overview` 页面已有风险文案映射：
  - `risk_level === 'exit'` -> `卖出信号`
  - `risk_level === 'warn'` -> `危险状态`
  - `risk_level === 'ok'` -> `稳定状态`

### 3.2 当前后端已有动作/风险来源

- 候选池中的 `sell_signal/sell_reason` 只在该股票已经处于 `positions` 中时才生成，因此它本质上是“已有持仓的退出动作信号”，而不是泛化的禁入建议。
- 低频引擎 `check_sell_signal_v2()` 当前明确返回的卖出原因包括：
  - `sector_cooldown`
  - `stop_loss`
- 这说明“离场”应理解为对已有仓位的退出动作，而不是“不要买”的泛化标签。

### 3.3 当前问题

- “回避”和“离场”被混用。
- “卖出建议”与“离场信号”没有稳定边界，容易重复。
- “跟踪观察”和“不满足条件”目前都大量显示为灰色，视觉上不可区分。
- 当前只有零散 `title` 提示，没有统一解释口径，不足以支撑全站语义统一。

## 4. 方案比较

### 4.1 方案 A：只统一颜色，不统一语义

- 页面继续各自保留状态判断，只要求颜色尽量一致。
- 优点：改动最小。
- 缺点：无法解决“同义不同词”和“同词不同义”的根本问题。

### 4.2 方案 B：统一一级状态，保留细状态文案

- 全站先统一到 4 个一级状态：
  - `建仓`
  - `离场`
  - `跟踪观察`
  - `不满足条件`
- 页面继续显示各自细状态，但需要先映射到一级状态。
- 颜色由一级状态决定，tooltip 负责解释细状态含义。
- 优点：不丢信息，且能真正实现跨页面统一。

### 4.3 方案 C：全站只保留 4 个固定文案

- 所有细状态都不再展示，只显示 4 个一级状态。
- 优点：最整齐。
- 缺点：会丢失“为什么不满足条件”“为什么离场”的关键信息。

结论：
- 采用方案 B。

## 5. 语义模型

### 5.1 一级状态

- `建仓`
  - 含义：当前对象满足建仓或建仓流程中的正向动作语义。
- `离场`
  - 含义：当前对象为已有持仓，系统给出明确退出动作或退出级风险提示。
- `跟踪观察`
  - 含义：当前对象值得持续关注，但尚不是当前优先执行对象。
- `不满足条件`
  - 含义：当前对象不适合立即参与，或未通过建仓门槛。

### 5.2 细状态与一级状态映射

- `可出手` -> `建仓`
- `已排队` -> `建仓`
- `离场信号` -> `离场`
- `风险退出` -> `离场`
- `止损退出` -> `离场`
- `板块冷却退出` -> `离场`
- `观察` -> `跟踪观察`
- `跟随观察` -> `跟踪观察`
- `回避` -> `不满足条件`
- `大盘不配合` -> `不满足条件`
- `题材不配合` -> `不满足条件`
- `动能不足` -> `不满足条件`
- `已放弃` -> `不满足条件`

### 5.3 关键语义边界

- `离场信号`
  - 指已有持仓出现明确退出动作信号。
  - 不用于表达“不要新开仓”。

- `卖出建议`
  - 本次不作为独立一级或细状态保留。
  - 如页面已有该文案，应统一收敛到 `离场信号`。

- `风险退出`
  - 不是一级状态。
  - 它是 `离场` 下的细分原因类别，用于描述因风险控制而退出。

- `回避`
  - 不归入 `离场`。
  - 语义定义为“当前不建议新开仓参与”。

- `跟踪观察`
  - 与“不满足条件”不同。
  - 它表示当前对象仍在策略观察名单中，只是还不执行。

## 6. 颜色规则

- `建仓`：绿色
- `离场`：红色
- `跟踪观察`：蓝色
- `不满足条件`：灰色

约束：
- 颜色只由一级状态决定。
- 同一个一级状态在不同页面必须使用一致颜色。
- 细状态文案允许不同，但不得改变一级状态颜色。

说明：
- 本次不把 `warn` 单独提炼为第五类一级状态。
- 现有 `risk_level === 'warn'` 仍可保留橙色作为风险层提示，但在统一状态系统里应映射到明确的语义 key，由 tooltip 解释其含义。
- 若后续需要把“预警”升级为一级状态，应另起设计，不在本次范围内。

## 7. Hover 释义设计

### 7.1 覆盖范围

- 状态标签
- 风险标签
- 执行队列状态

不覆盖：
- 普通正文
- 普通表头
- 纯数值字段
- 与状态无关的标题或导航

### 7.2 tooltip 交互规则

- 鼠标移到标签本体时显示 tooltip。
- 标签本体仍保留短文案和颜色。
- tooltip 只负责释义，不替代主标签显示。
- 同一语义 key 在全站只维护一份解释文案。

### 7.3 tooltip 内容结构

- 第一行：当前标签名
- 第二行：所属一级状态
- 第三行：业务解释
- 第四行：触发条件或适用范围
- 第五行：与相近语义的区别，仅在必要时显示

### 7.4 tooltip 示例

- `离场信号`
  - 一级状态：`离场`
  - 含义：已有持仓触发明确退出动作
  - 触发：命中止损、板块冷却等卖出条件
  - 区别：这不是“不要买”，而是“已有仓位应退出”

- `回避`
  - 一级状态：`不满足条件`
  - 含义：当前不建议新开仓参与
  - 触发：风险过高或对象当前不适合参与
  - 区别：这不是卖出动作

- `跟随观察`
  - 一级状态：`跟踪观察`
  - 含义：进入观察名单，但不作为当前优先建仓对象
  - 触发：角色为跟随，暂不直接出手

- `动能不足`
  - 一级状态：`不满足条件`
  - 含义：当前信号强度不足，未达到建仓门槛

## 8. 前端实现结构

### 8.1 公共状态语义字典

新增一份全站唯一字典，建议放在 `neotrade3-dashboard/src` 的公共模块中。

字典结构至少包含：
- `semanticKey`
- `label`
- `group`
- `description`
- `trigger`
- `difference`

建议语义 key 示例：
- `entry_ready`
- `entry_queued`
- `exit_signal`
- `exit_risk_stop`
- `exit_sector_cooldown`
- `watch_general`
- `watch_follower`
- `not_qualified_avoid`
- `not_qualified_market`
- `not_qualified_concept`
- `not_qualified_momentum`
- `abandoned`
- `risk_ok`
- `risk_warn`

### 8.2 公共状态标签组件

新增一个公共状态标签组件。

职责：
- 根据 `semanticKey` 读取统一颜色。
- 根据 `semanticKey` 读取 tooltip 说明。
- 支持页面传入原始文案，但颜色和解释口径仍由字典控制。

### 8.3 页面映射职责

各页面不直接手写 tooltip 文案。

各页面只负责：
- 把原始字段映射为 `semanticKey`
- 决定是否显示该标签

公共组件负责：
- 显示文案
- 应用颜色
- 显示 tooltip

## 9. 页面接入落点

### 9.1 `Lowfreq`

- 候选池状态标签
- 板块/个股风险标签
- 持仓中与离场相关的风险或原因标签
- 人工干预记录状态字段

### 9.2 `Overview`

- 主线概念风险灯
- 热门板块中个股风险标签
- 执行队列中的状态与风险说明

### 9.3 `StockCheck`

- 单股核验最终结论标签
- 风险标签
- 若当前页面已显示建仓/观察/离场结论，则纳入统一状态字典

## 10. 实施顺序

严格顺序如下：

1. 建立公共语义字典
2. 建立公共状态标签 + tooltip 组件
3. 接入 `Lowfreq`
4. 接入 `Overview`
5. 接入 `StockCheck`
6. 调整相关页面测试
7. 做一次定点构建与页面验证

## 11. 验证

- 组件级验证：
  - 同一 `semanticKey` 在不同页面颜色一致
  - hover 后能看到统一释义

- 页面级验证：
  - `Lowfreq` 中 `可出手 / 跟随观察 / 动能不足 / 回避` 的颜色和释义正确
  - `Overview` 中 `risk_level === exit` 不再出现与 `Lowfreq` 冲突的文案语义
  - 执行队列中的卖出类状态与“不要买”的状态不再混淆

- 回归验证：
  - `npm test -- --run`
  - `npm run build`

## 12. 完成标准

- 全站状态展示收敛到统一一级状态语义。
- 细状态文案被保留，但颜色由一级状态统一控制。
- 状态标签、风险标签、执行队列状态均支持 hover 释义。
- “离场”和“回避”语义边界在前端展示层明确分开。
- 页面测试与构建通过。
