# 2026-07-23 混沌模型专项团队输出总控图

## 0. 目的

本文件只做一件事：

- 把混沌模型专项从“零散切片推进”切换到“围绕最终团队输出的项目控制模式”

本文件不定义：

- 具体代码实现步骤
- 参数最终值
- 单次 run 的详细分析报告
- 页面实现

本文件的职责是让后续每一轮推进都能回答 3 个问题：

- 现在服务的是哪一个最终团队输出
- 距离该输出还差哪些能力块
- 这轮留下的尾巴是什么，是否显式挂账

## 1. 使用规则

### 1.1 三个最终团队输出

后续混沌专项只围绕 3 个最终团队输出组织：

1. 当前应该专注哪些股票
2. 明天应该买入哪些股票
3. 明天应该卖出哪些股票

### 1.2 不允许隐藏尾巴

从本文件生效开始，任何未完成事项都必须归入以下 3 类之一：

- `Blocking`：不解决就不能进入下一阶段
- `Deferred`：明确后移，但不阻塞当前阶段结论
- `Rejected`：明确放弃，不再反复讨论

如果某项工作既不在当前结果里，也没有进入上述 3 类之一，则视为**隐藏尾巴**，不允许继续推进。

### 1.3 一轮只推进一个能力块

后续每一轮推进只允许：

- 服务一个最终团队输出
- 推进一个能力块
- 形成一个阶段结论

禁止再用“看起来都对的小切片”并行扩散，避免 owner 在推进过程中失去全局位置感。

## 4. 输出 A：当前应该专注哪些股票

### 4.1 当前定义

本对象不是买入指令，而是团队注意力分配对象。

它回答的问题是：

- 现在最值得研究、复盘、盯盘和进入会议讨论的是哪些股票

### 4.2 当前证据

- `stock_only` 在正式全 A 中期验证中具备正向分层能力
- 覆盖率约 `98%`
- `skipped_*` 统计可解释
- 中期方向与短窗方向没有系统性冲突
- 输出 A 的日报链路已经跑通，正式产物已生成：
  - [formal_all_a_share_20260720_v2.json](file:///Users/mac/NeoTrade3/.runtime_outputs/artifacts/chaos_focus_board/2026-07-20/chaos_focus_board_formal_all_a_share_20260720_v2.json)
- `2026-07-20` 全 A 正式日报 `v2` 的当前计数为：
  - `focus_list = 427`
  - `watch_list = 145`
  - `short_pulse_warning = 1123`
  - `missing_fundamentals = 107`
  - `state_not_positive = 3213`

### 4.3 当前缺口

- `Durability Gate` 当前仍偏宽，`durable_pass` 还可能放入高估值、增长证据偏弱的标的
- 基本面解释文案仍会出现 `净利下滑0.0%` 这类误导性表述
- 运行状态仍未统一：结果文件已落盘，但进程仍以非零退出结束

### 4.4 Blocking

- `A1`：收紧 `Durability Gate` 的 `durable_pass`，避免高估值、增长证据弱的标的进入 `focus_list`
- `A2`：修正基本面解释文案，避免出现 `净利下滑0.0%` 这类误导性描述
- `A3`：统一运行成功状态，避免“结果已产出但编排侧仍判失败”

### 4.5 通过标准

当前阶段，输出 A 视为通过，至少同时满足：

1. `L1 stock_only` 保持正式中期正向分层
2. 关注名单引入耐持有过滤后，不再主要由高脉冲、低持续性的标的主导
3. 每只股票都能给出最小证据解释：为什么进入专注名单、为什么不是短平快警示对象
4. 数据缺失对象不会混入正式三桶，而是进入数据质量排除列表

### 4.6 显式尾巴

- `Deferred`：`L2/L3` 增益未恢复前，不并入关注名单正式主口径
- `Deferred`：页面展示可后置
- `Rejected`：不把“看起来平滑、像价格走势”当成专注名单通过标准

## 5. 输出 B：明天应该买入哪些股票

### 5.1 当前定义

本对象不是神谕式确定答案，而是明日开仓候选对象。

它回答的问题是：

- 如果团队明天要开新仓，哪些股票值得进入买入检查队列

### 5.2 当前证据

- 当前只有状态识别侧的中期分层证据
- 还没有形成正式的 `M2 -> M3` 入场动作映射

### 5.3 当前缺口

- 缺买入触发条件
- 缺风险过滤条件
- 缺候选排序逻辑
- 缺 T+1 执行纪律
- 缺动作级验证口径

### 5.4 Blocking

- `B1`：定义 `buy_candidate` 最小对象
- `B2`：把 `stock_state_context` 翻译成可验证的买入动作条件
- `B3`：定义“可以关注但不能买”的阻断条件
- `B4`：建立 T+1 动作级回测与验证口径

### 5.5 通过标准

当前阶段，输出 B 视为通过，至少同时满足：

1. 明日买入候选有明确进入规则
2. 候选对象与专注名单有清晰边界
3. 动作级验证结果不与当前中期状态识别证据矛盾

### 5.6 显式尾巴

- `Deferred`：仓位管理与分批执行可后置
- `Deferred`：L2/L3 恢复增益前，不纳入买入主公式
- `Rejected`：不允许把“状态强”直接等同于“明天必买”

## 6. 输出 C：明天应该卖出哪些股票

### 6.1 当前定义

本对象不是简单的“看空名单”，而是持仓退出建议对象。

它回答的问题是：

- 对已经持有的股票，哪些已经出现减仓、止盈、止损、退出的证据

### 6.2 当前证据

- 当前只有状态衰竭与驱动切换的方向性语言
- 尚未形成正式的持仓生命周期状态机

### 6.3 当前缺口

- 缺持仓上下文对象
- 缺退出纪律
- 缺持有期风险与衰竭条件定义
- 缺退出动作的验证口径

### 6.4 Blocking

- `C1`：定义 `holding_state` / `sell_candidate` 最小对象
- `C2`：建立“衰竭 / 顶部 / 风险失控 / 正常震荡”的区分规则
- `C3`：建立退出动作的回测与审计链路

### 6.5 通过标准

当前阶段，输出 C 视为通过，至少同时满足：

1. 卖出候选严格依赖持仓上下文
2. 退出理由可解释、可审计
3. 退出建议能在样本上减少明显回撤或提升兑现质量

### 6.6 显式尾巴

- `Deferred`：卖出细分类型（止盈/止损/衰竭/再平衡）可后置细化
- `Rejected`：不允许只用无持仓上下文的状态信号直接产出卖出指令

## 7. 能力块顺序

为防止再次陷入“每个小切片都对，但整体方向感消失”，后续推进顺序冻结为：

1. `CB1`：总控图建立
2. `CB2`：`Durability Gate` 最小定义
3. `CB3`：`focus_list` / `watch_list` / `short_pulse_warning` 最小对象
4. `CB4`：`buy_candidate` 最小对象与动作映射
5. `CB5`：持仓生命周期与 `sell_candidate`
6. `CB6`：`L2/L3` 增益问题单独收口

其中：

- `CB2` 与 `CB3` 服务输出 A
- `CB4` 服务输出 B
- `CB5` 服务输出 C
- `CB6` 不得抢在 `CB2/CB3/CB4/CB5` 前重新扩散为主线
