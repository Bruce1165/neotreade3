# App Tushare 黄旗详情折叠设计

日期：2026-07-09

## 1. 目标

本切片只解决一个问题：

当 `App` 壳层检测到 `Tushare` 积分不足时，黄旗横幅应保留稳定主文案，同时把辅助诊断细节收进折叠区，避免主行信息噪音过大。

本切片的目标不是改变黄旗语义或判定逻辑，而是收口展示层结构。

## 2. 范围

包含：

- `neotrade3-dashboard/src/App.jsx` 中 `GlobalBanner` 的最小展示层改动
- 一个新的聚焦测试载体，例如 `neotrade3-dashboard/src/App.tushareBanner.test.jsx`

不包含：

- `GlobalBanner` 的触发条件变化
- Header 环境文案改写
- 全局 API 错误横幅
- 侧边栏导航或 `OpsCenter`
- 新的黄旗字段、新的告警级别或新的业务语义

## 3. 现有证据

当前工作树中，[App.jsx](file:///Users/mac/NeoTrade3/neotrade3-dashboard/src/App.jsx) 的 `GlobalBanner` 已存在一条未提交的展示改动线：

- 现状基线是将细节字段平铺在主行
- 工作树中已存在把细节收进 `details` 折叠区的雏形

这说明本切片不是凭空发明的新需求，而是对现有可验证展示线进行边界收口。

## 4. 职责归属

`GlobalBanner` 继续只承担一件事：

- 当 `Tushare` 主源受影响时，在 `App` 壳层展示统一黄旗提醒

本切片不改变这个职责，只改变“细节如何展示”。

因此：

- 主文案继续承担提醒职责
- 折叠区承担辅助诊断信息职责
- 横幅不承担修复、重试或更复杂的状态管理职责

## 5. 展示设计

### 5.1 主文案

默认主行必须继续显示：

- `Tushare 黄旗：`
- `检测到 Tushare 积分不足，日线主源可能受影响。`

本切片不修改这两句文案的含义。

### 5.2 折叠区

辅助诊断信息收进折叠区，入口文案为：

- `查看详细信息`

折叠区承接的细节字段范围固定为：

- 最近一次受影响时间
- 受影响接口
- 最近一次恢复时间
- 最近一次恢复接口

### 5.3 空值处理

若恢复类字段不存在：

- 隐藏对应字段项
- 不输出空占位
- 不改变主文案

若黄旗不触发，则整个横幅不显示。

## 6. 交互边界

本切片只使用原生 `details / summary` 交互形态。

本切片不处理：

- 自定义展开动画
- 展开状态持久化
- 横幅关闭按钮
- 多层级信息结构

交互目标只有一个：

- 默认简洁
- 需要时可展开查看细节

## 7. 测试设计

测试重点不是验证浏览器原生 `details` 的全部行为，而是验证展示边界已经正确收口。

推荐最小测试集合：

1. `credit_insufficient=false` 时不显示黄旗
2. `credit_insufficient=true` 且有细节字段时：
   - 显示主文案
   - 显示 `查看详细信息` 入口
   - 不再把旧平铺细节直接断言为主行结构

如现有 [App.test.jsx](file:///Users/mac/NeoTrade3/neotrade3-dashboard/src/App.test.jsx) 过于混杂，更推荐新增聚焦测试文件：

- `neotrade3-dashboard/src/App.tushareBanner.test.jsx`

## 8. 提交边界

目标实现提交只允许包含：

- `App.jsx` 中 `GlobalBanner` 的最小 hunk
- 对应聚焦测试文件，或 `App.test.jsx` 的最小相关 hunk

必须排除：

- Header 文案修改
- 全局 API 错误横幅
- `OpsCenter` 路由与导航
- 侧边栏标签或 active 规则变化

## 9. 验证要求

预期最小验证闭环：

- 运行 `Tushare` 黄旗聚焦测试
- 若触及 `App.test.jsx`，仅补跑相关 `App` 测试
- 使用精确暂存或索引快照，确保提交中不带其他 `App` 漂移

## 10. 结论

本切片的本质不是功能扩展，而是：

- 保留黄旗主提醒
- 压缩默认噪音
- 把辅助诊断信息移入折叠区

只有先把这条展示线从 `App` 混杂漂移中拆出来，后续 `Header` 文案、导航 IA 等其他 `App` 主题线，才能继续保持原子化推进。
