# 2026-07-23 混沌模型专项基线与三层状态上下文设计

## 0. 目的

本设计文档只做三件事：

- 固化当前混沌模型专项已确认的阶段性基线
- 冻结 M2 三层状态上下文体系的最小契约方向
- 明确未来一小步的最小推进顺序，避免单线推进过深、留下过多 TODO

本文件不是实现方案，不定义代码修改步骤，不定义参数最终值，不替代 `PROJECT_STATUS.md` 的当前状态真相源。

## 1. 当前阶段共识

### 1.1 专项总原则

当前阶段的推进原则是：

- 把明确的部分落实
- 查缺补漏
- 夯实基础

直接含义：

- 不围绕混沌线单线深挖到明显超出 `M1-M6` 承接能力的深度
- 不继续堆远端 TODO
- 每前进一步，都必须检查上游事实、下游消费、验证链、治理链、展示链是否接得住

### 1.2 当前阶段定位

当前阶段，混沌模型不直接承担完整交易动作主责任；应先作为“状态上下文主变量体系”推进。

这一定义包含三层：

- `L1` 个股自身状态
- `L2` 板块/概念上下文状态
- `L3` 全市场背景状态

优先级固定为：

- `L1 > L2 > L3`

含义：

- 个股判断首先读取个股自身阴阳关系与历史沿革
- 再读取其所处板块/概念的阴阳关系与历史沿革
- 最后读取全市场背景约束

因此，不再将“全市场主变量”定义为单层对象，而是改为“三层状态上下文体系”。

### 1.3 当前阶段边界

当前阶段已确认：

- `M2` 为三层状态上下文体系的正式主归属
- `M4` 只读消费，用于验证与分层解释
- `M6` 只读消费，用于展示与观测
- 当前阶段的主要验证目标是“先证中期有效”，而不是继续围绕 `5D/10D` 短窗结果推进动作主线

当前阶段明确不做：

- 不直接切换到混沌动作主线
- 不围绕更复杂的 L2 传播图展开
- 不继续深挖动作层规则
- 不继续做大规模参数搜索

## 2. 为什么现在先做状态上下文，而不是动作主线

### 2.1 证据边界

当前最强证据集中在：

- 全 A 股、全历史日度混沌快照已完成落盘
- 全市场按日阴阳比例可直接读取
- `regime_combo` 在短窗评估中优于纯 `point(net_energy)`

但当前尚未被正式证明的部分包括：

- 对 `20D/40D/60D` 中期窗口的稳定有效性
- 对中期持有与离场链条的成熟支持

因此，当前阶段更合理的推进方式是：

- 先冻结状态对象
- 先补中期验证
- 暂不把混沌模型推进为完整交易动作主引擎

### 2.2 系统平衡约束

NeoTrade3 是 `M1-M6` 闭环系统，不允许某一条线远超其他环节成熟度。

如果当前继续单线深挖混沌动作层，将出现以下风险：

- `M2` 语义未定，`M3` 已开始硬接
- `M4` 仍只验证短窗，无法证明主目标
- `M5` 难以治理未冻结的对象与版本
- `M6` 无法承接仍在漂移的状态语义

因此，本阶段应以“闭环可核实”为第一优先。

## 3. M2 三层状态上下文体系

## 3.1 总体定义

`M2` 的职责是定义状态语义，而不是直接定义交易动作。

`M2` v1 正式冻结为：

- 以 `stock_state_context` 为主语义
- 以 `theme_state_context` 为关键上下文
- 以 `market_state_context` 为背景约束
- 通过 `state_context_link` 显式表达跨层关系

对象版本表达规则同时冻结为：

- 对象名固定不带版本后缀
- 对象版本通过对象内部 `object_version = "v1"` 表达

所有对象均只承载状态语义，不承载动作语义与未来收益预测。

### 3.2 L1：个股自身状态

对象名：

- `stock_state_context`

定位：

- 个股主语义对象

回答的问题：

- 这只股票今天自身的阴阳关系如何
- 它当前所处驱动段是否成立
- 相对自身历史，这一驱动段是在增强、衰减还是高噪声切换

最小字段类别：

- `identity`
- `core_energy`
- `regime`
- `dynamics`
- `coverage_quality`
- `audit_ref`
- `version_ref`

边界：

- L1 不负责解释板块和全市场
- L1 不被 L2/L3 覆盖

### 3.3 L2：板块/概念上下文状态

对象名：

- `theme_state_context`

定位：

- 板块/概念统一上下文对象
- v1 不拆“行业对象”和“概念对象”

回答的问题：

- 该股所处的局部生态，是在支持它还是削弱它
- 板块/概念自身是启动、扩散、分歧，还是退潮
- 个股相对其主题上下文，是同步、超强还是掉队

最小字段类别：

- `identity`
- `core_energy`
- `regime`
- `dynamics`
- `coverage_quality`
- `audit_ref`
- `version_ref`

边界：

- L2 是关键上下文，不是主语义
- L2 不能替代 L1 对个股自身状态的定义
- L2 当前阶段只定义对象，不展开相关主题传播图和多主题融合算法

### 3.4 L3：全市场背景状态

对象名：

- `market_state_context`

定位：

- 全市场背景约束对象

回答的问题：

- 当前市场整体是阳主导、阴主导，还是噪声切换
- 市场整体是在提供顺风、逆风，还是高噪声背景

最小字段类别：

- `identity`
- `core_energy`
- `regime`
- `dynamics`
- `coverage_quality`
- `audit_ref`
- `version_ref`

边界：

- L3 只做背景约束
- L3 不得直接成为个股动作的第一决定项
- L3 不得覆盖 L1/L2 语义

### 3.5 跨层关联对象

对象名：

- `state_context_link`

定位：

- 显式表达 L1/L2/L3 的引用关系与优先级

最小字段类别：

- `identity`
- `l1_ref`
- `l2_refs`
- `l3_ref`
- `context_priority`
- `context_validity`
- `version_ref`

边界：

- 只表达关系，不表达动作
- 不在 link 对象里重复上层语义字段

### 3.6 三层关系的正式解释顺序

当层间出现差异或冲突时，解释顺序必须固定为：

1. 先解释 L1：该股自身当前是什么状态
2. 再解释 L2：板块/概念是在支持还是削弱该股
3. 最后解释 L3：当前全市场背景是在顺风、逆风还是高噪声

禁止：

- 用 L3 直接覆盖 L1
- 用 L2 直接改写 L1
- 在 M6 里重新发明第四种解释顺序

## 4. 消费关系

### 4.1 M2

职责：

- 定义并输出 L1/L2/L3 与 link 对象
- 维护对象语义一致性

### 4.2 M3

职责：

- 只读消费 M2 状态对象，作为未来动作链上下文

边界：

- 当前阶段不直接以混沌状态上下文替代完整动作主线
- 不反向定义 M2 语义

### 4.3 M4

职责：

- 只读消费三层状态对象
- 在不同 L2/L3 背景下验证 L1 的中期有效性

边界：

- 不允许把评估标签回灌为 M2 在线语义
- 不允许把 M4 指标上提为 M2 状态真值

### 4.4 M5

职责：

- 治理版本推进
- 审批对象、权重、阈值等版本化调整

边界：

- 不静默在线改写 M2 当日状态语义

### 4.5 M6

职责：

- 展示三层状态对象
- 形成共享观察入口

边界：

- 只读消费
- 不发明新语义
- 不将状态字段渲染成交易动作建议

## 5. 当前阶段验证目标

当前阶段不以短窗领先作为最终目标，而以“中期有效性”作为下一阶段门槛。

当前阶段需补齐的正式验证目标是：

- `20D/40D/60D` 是否存在稳定分层能力
- 加入 L2/L3 上下文后，L1 的中期有效性是否改善
- 中期验证结果是否与短窗结果方向一致

在上述问题未被正式回答前，不宣称：

- 混沌模型已适合承担中期动作主线
- 当前参数已成熟
- 当前对象已经可以完整替代现有交易链

## 6. 最小推进清单

### 6.1 任务 1：固化专项基线

目标：

- 把当前已确认基线写入正式真相源

完成标准：

- 后续讨论默认继承该基线，不再反复争论前提

### 6.2 任务 2：冻结 M2 最小对象契约

目标：

- 正式冻结 `L1/L2/L3 + link` 的最小对象边界

完成标准：

- 能明确回答对象是什么、不是什么，谁能消费、谁不能改写

### 6.3 任务 3：建立中期验证口径

目标：

- 把验证焦点从 `5D/10D` 拉到 `20D/40D/60D`

完成标准：

- 形成一套统一的中期验证口径与结果表

### 6.4 任务 4：核实 L2 最小承接能力

目标：

- 确认 `theme_state_context` 是否已有最小事实支撑

完成标准：

- 给出清晰结论：可立最小对象 / 当前不可立及其缺口

### 6.5 任务 5：预留 M6 最小承接口径

目标：

- 让三层状态对象未来可观察、可共享、可核实

完成标准：

- M6 的最小承接范围明确

## 7. 当前阶段明确不做

当前阶段不纳入本轮的事项：

- L2 传播图
- 行业/概念拆对象
- 多主题融合算法
- 更细动作规则
- 大规模参数搜索
- 新增远端 TODO

## 8. 通过条件

本设计在当前阶段的通过条件是：

- 已确认的基线进入正式文档体系
- 三层状态上下文的对象与边界不再漂移
- 中期验证口径被正式建立
- L2 承接能力有明确结论
- M6 明确其最小承接口径

在以上条件未满足前，不进入更深实现讨论，不扩展新的大范围设计。
