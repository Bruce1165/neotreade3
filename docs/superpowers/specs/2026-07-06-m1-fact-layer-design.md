# M1 事实层详细设计

日期：2026-07-06

## 1. 文档目标

本文档用于冻结 NeoTrade3 量化体系中 `M1 事实层` 的详细设计，明确其职责边界、数据域、源级策略、任务契约、质量证明、Attention 机制和第一阶段实施范围。

本文档解决的问题：

- `M1` 到底是什么，不是什么。
- `M1` 应向上游交付哪些正式对象。
- 当前事实数据源按什么等级使用。
- 数据何时算“完整 / 真实 / 有效”。
- `M1` 的问题如何进入治理闭环。
- 第一阶段应该先建设哪些能力。

本文档不展开：

- `M2` 的周期识别算法。
- `M3` 的交易规则细节。
- 各数据源的逐接口字段清单。
- 编排器/调度器本身的实现细节。

## 2. 设计背景

当前项目中的事实数据能力已经部分存在，但分散在多个模块内：

- `daily_prices / stocks` 已形成当前最成熟的事实主链。
- Tushare 市场情报、指数权重、公告/研报/基金等能力在代码中已存在较多入口，但并未全部进入统一生产主链。
- 派生事实大量分散在 `engine / factor_matrix / analysis / screeners` 内部，尚未统一收口为正式事实层产物。

用户已确认的关键约束：

- `M1` 是所有分析判断的事实地基，必须完整、真实、有效。
- `Tushare` 是当前唯一可信赖的首要权威数据源。
- 其他来源在没有完成单独论证前，不得与 `Tushare` 并列为正式主源。
- `M1` 需要先于 `M2/M3/M4/M5` 成为正式可依赖对象。
- `M1` 的数据抓取契约必须清楚，但 `M1` 不直接拥有调度执行权。

## 3. M1 的正式定义

`M1` 不是单纯的数据抓取模块。

`M1` 的正式定义为：

- 外部数据接入
- 事实标准化
- 派生事实生成
- 数据质量证明
- 任务契约定义
- Attention 与缺口状态输出

统一构成的事实层。

`M1` 的核心承诺只有四个：

- 事实真实
- 时点一致
- 口径稳定
- 缺失可见

## 4. M1 的职责边界

### 4.1 M1 负责

- 接入外部事实源。
- 对事实做标准化和统一落库。
- 生成统一派生事实。
- 对事实做质量审计、完整性检查、新鲜度证明和回放一致性检查。
- 定义每类事实的刷新契约、依赖关系、完成标准和发布门禁。
- 对上游输出正式事实对象和质量状态。

### 4.2 M1 不负责

- 不负责周期识别。
- 不负责交易决策。
- 不负责 benchmark 评分。
- 不负责前端解释。
- 不直接拥有调度器的执行控制权。

### 4.3 与调度器的关系

`M1` 不直接组织或拥有定时任务调度器，但 `M1` 拥有任务契约定义权。

也就是说：

- `M1` 负责定义：
  - 何时刷新
  - 依赖谁先完成
  - 什么叫完成
  - 什么情况下允许发布
  - 什么情况下必须阻断
- 调度系统负责：
  - 什么时候跑
  - 按什么顺序跑
  - 失败后如何重试
  - 如何执行这些契约

## 5. M1 的正式输出对象

`M1` 向上游交付的对象分为四类：

### 5.1 正式事实表

至少包括：

- `F1` 日线行情事实表
- `F2` 股票基础事实表
- `F3` 财务事实表
- `F4` 指数与市场事实表
- `F5` 市场情报事实表
- `F6` 主题/概念映射事实表
- `F7` 交易日历事实表

### 5.2 正式派生事实表

由 `M1` 统一派生、统一口径、统一版本化，至少包括：

- `PF1` 个股派生事实
- `PF2` 市场派生事实
- `PF3` 板块派生事实
- `PF4` 资金参与代理事实
- `PF5` 情报聚合事实

### 5.3 数据质量状态

至少包括：

- `Q1` Source Status
- `Q2` Freshness Status
- `Q3` Coverage Status
- `Q4` Consistency Status
- `Q5` Replay Status

### 5.4 任务契约状态

至少包括：

- `T1` Source Refresh Contract
- `T2` Dependency Contract
- `T3` Completion Contract
- `T4` Publish Gate
- `T5` Degradation Policy

## 6. M1 的一级数据域

`M1` 的一级数据域定义为以下八类：

- `D1` 个股行情事实
- `D2` 指数与市场结构事实
- `D3` 板块/行业事实
- `D4` 资金与参与结构事实
- `D5` 基金/机构行为代理事实
- `D6` 事件与主题驱动事实
- `D7` 元数据与映射事实
- `D8` 派生事实与标准化视图

## 7. M1 的三层状态模型

为了防止“理想能力”和“现有能力”混写，`M1` 的每个数据域必须明确处于三层状态之一：

- `L1 生产态`
- `L2 已具备未编排`
- `L3 尚未闭环`

### 7.1 状态判定规则

`L1 生产态`

- 已有真实数据源
- 已有正式落库
- 已进入稳定主链或生产任务
- 已被上游核心模块持续消费

`L2 已具备未编排`

- 代码能力、接口、建表或同步入口已存在
- 但尚未形成稳定生产调度与发布闭环
- 不能默认作为正式能力承诺

`L3 尚未闭环`

- 只有适配器、局部抓取、研究性入口或弱接线
- 尚未完成“抓取 -> 落库 -> 标准化 -> 审计 -> 上游消费”全链路

### 7.2 当前映射

#### D1 个股行情事实

- 当前状态：`L1`
- 说明：`daily_prices` 是当前最成熟的事实主链

#### D2 指数与市场结构事实

- 当前状态：`L1 + L2`
- 说明：基础指数/市场事实已被部分分析模块消费，但还没有以统一正式对象发布

#### D3 板块/行业事实

- 当前状态：`L1 + L2`
- 说明：基础映射已存在，但更强的板块结构事实尚未统一收口

#### D4 资金与参与结构事实

- 当前状态：`L2`
- 说明：已有可派生代理，但尚未正式定义为统一事实域

#### D5 基金/机构行为代理事实

- 当前状态：`L2`
- 说明：代码能力与同步入口已存在，但未全面进入生产日更主链

#### D6 事件与主题驱动事实

- 当前状态：`L2 + L3`
- 说明：部分 Tushare 情报能力已具备，但新闻/事件类源多数未闭环

#### D7 元数据与映射事实

- 当前状态：`L1 + L2`
- 说明：`stocks` 和 `trading_calendar_cache` 已存在，但历史映射一致性尚未正式收口

#### D8 派生事实与标准化视图

- 当前状态：`L2`
- 说明：派生事实大量存在，但分散在多个模块内部，尚未统一收口为正式 M1 产物

## 8. 数据源优先级与可信度等级

`M1` 不采用“谁能抓到就先用谁”的策略，必须采用数据源等级制度。

### 8.1 四级数据源等级

#### `S1 正式权威源`

- 可作为正式事实真值来源
- 允许直接进入 `M1` 正式事实表
- 可作为 `M2/M3/M4/M5` 的基础输入

当前唯一明确属于 `S1` 的源：

- `Tushare`

#### `S2 补充源`

- `Tushare` 无法直接提供，或提供不完整
- 经论证后可作为补充字段来源
- 不得反向覆盖 `S1`

当前默认没有自动进入 `S2` 的来源，需逐项论证。

#### `S3 兜底源`

- 仅在 `S1` 明确缺失或短时失败时启用
- 用于维持有限连续性
- 不得静默变成主口径

当前项目中明确属于 `S3` 语义的对象：

- `Tencent safety-net`

#### `S4 研究源`

- 可抓取、可比对、可研究
- 但不进入当前正式事实主链

当前更接近 `S4` 的对象包括：

- 财联社
- 巨潮
- mootdx 相关能力
- 其他尚未标准化、未编排、未审计的来源

### 8.2 源级硬规则

- `R1`：默认优先使用 `S1`
- `R2`：`S2/S3/S4` 不得静默覆盖 `S1`
- `R3`：跨源冲突必须生成 `Consistency Status`
- `R4`：启用 `S3` 必须显式标记并留下回补要求
- `R5`：回测/benchmark 必须优先消费 `S1` 或明确可接受的 `S2`

## 9. M1 的任务契约

`M1` 拥有任务契约定义权，但不拥有调度执行权。

### 9.1 单个任务契约的最低字段

- `task_id`
- `data_domain`
- `source_tier`
- `primary_source`
- `fallback_source`
- `trigger_type`
- `expected_frequency`
- `dependency_tasks`
- `required_freshness`
- `completion_criteria`
- `publish_gate`
- `fallback_policy`
- `recovery_policy`
- `blocking_scope`

### 9.2 任务类型

- `T-A` 抓取任务
- `T-B` 标准化任务
- `T-C` 组合/派生任务
- `T-D` 发布任务

### 9.3 刷新频率原则

按事实类型分四类：

- `F1` 交易日高频刚性事实
- `F2` 交易日低频增强事实
- `F3` 慢变量事实
- `F4` 研究级事实

### 9.4 依赖顺序

`M1` 的依赖顺序固定为：

1. 原始抓取
2. 基础标准化
3. 元数据/映射校准
4. 派生事实生成
5. 质量审计
6. 发布判定

## 10. 发布门禁与阻断规则

### 10.1 Publish Gate

至少包括：

- `G1` Primary Source Gate
- `G2` Freshness Gate
- `G3` Coverage Gate
- `G4` Consistency Gate
- `G5` Replay Gate

### 10.2 阻断规则

分为：

- `B1` 强阻断
- `B2` 弱阻断
- `B3` 可降级

### 10.3 fallback 与恢复

- 只有 `S1` 明确缺失或短时失败，才允许进入 `S3`
- fallback 必须留下来源标记、覆盖范围、起止时间、待回补状态
- `S1` 恢复后必须优先回补并更新质量状态

## 11. 数据质量、完整性与新鲜度证明

### 11.1 三个核心概念

`完整`

- 关键字段齐全
- 覆盖范围满足最小宇宙要求
- 缺失状态可见且可量化

`真实`

- 来源可追溯
- 口径明确
- 同字段不会被多个来源静默混写
- 历史时点可回放

`有效`

- 对特定上游任务来说可安全消费
- 不会因缺失、过期、错位、冲突而误导上游判断

### 11.2 四类质量证明

- `P1` 完整性证明
- `P2` 真实性证明
- `P3` 新鲜度证明
- `P4` 适用性证明

### 11.3 Freshness Proof

`Freshness Proof` 是一组可审计证据，不是“任务跑过了”的口头判断。

最低字段包括：

- `target_date`
- `effective_trade_date`
- `primary_source`
- `actual_source`
- `source_finished_at`
- `publish_batch_id`
- `coverage_ratio`
- `freshness_passed`
- `fallback_used`
- `blocking_reason`

### 11.4 覆盖率与冲突检测

`M1` 必须正式检查：

- `active_stock_coverage_ratio`
- `backtest_universe_coverage_ratio`
- `financial_universe_coverage_ratio`
- `index_constituent_coverage_ratio`
- `sector_mapping_coverage_ratio`

并且必须检测：

- 同字段跨源差异
- 主源与 fallback 差异
- 历史映射前后不一致
- 派生视图与底层事实不一致

### 11.5 Replay Integrity

`Replay Integrity` 是正式质量门禁，至少检查：

- 历史板块/行业映射时点一致性
- 财报/公告的发布时间与可见时间
- 指数权重历史可回放性
- 概念/主题成员历史是否可还原

## 12. Attention Item、缺口分级与治理接口

`M1` 的问题不能只留在日志或状态码中，必须结构化为正式 Attention Item。

### 12.1 M1 产生的主要 Attention 类型

- `A1` Primary Source Missing
- `A2` Fallback Activated
- `A3` Coverage Below Threshold
- `A4` Cross-Source Conflict
- `A5` Replay Integrity Broken
- `A6` Data Capability Gap

### 12.2 缺口分级

严重度：

- `P0` 阻断级
- `P1` 高风险级
- `P2` 中风险级
- `P3` 研究级

类型：

- 运行失败型
- 质量不足型
- 结构缺失型
- 冲突漂移型
- 治理升级型

### 12.3 上游影响传播

每个 Attention Item 必须至少声明：

- `impacts_M2`
- `impacts_M3`
- `impacts_M4`
- `impacts_M5`
- `impact_mode`：`block / degrade / warn`

### 12.4 与 M5 的关系

`M1` 负责：

- 发现问题
- 结构化定义问题
- 输出质量状态与 Attention

`M5` 负责：

- 消费这些问题
- 生成 Change Request
- 排优先级
- 决定实验/升级/人工关注路径

## 13. 第一阶段实施范围

### 13.1 第一阶段必须完成

#### `P1-A` 日线行情主链正式化

- 正式定义 `D1` 字段契约
- 固化 `Tushare` 主源、`Tencent safety-net` 兜底源
- 建立行情 freshness proof、coverage gate、fallback 规则

#### `P1-B` 元数据与交易日历正式化

- 正式定义 `D7`
- 固化股票状态、基础映射、交易日历的真值边界
- 建立 replay 最小约束

#### `P1-C` 派生事实统一化

优先收口：

- N 日收益
- 波动率/回撤
- 峰值收益与距峰值回撤
- 距前高/前低
- 市场广度基础视图
- 板块强度基础视图
- 活跃度/参与度基础代理

#### `P1-D` 最小质量治理闭环

至少实现：

- `Source Status`
- `Freshness Proof`
- `Coverage Status`
- 第一批正式 Attention Item

### 13.2 第二阶段优先建设

- `P2-A` 板块/行业结构事实正式化
- `P2-B` 资金与参与结构事实正式化
- `P2-C` 基金/机构行为代理正式化

### 13.3 第三阶段或研究态

- `P3-A` 事件与主题驱动事实正式化
- `P3-B` 研究源升级评估

### 13.4 第一阶段明确不做

- 不追求把所有数据源都拉通
- 不把研究源接进正式主链
- 不在 `M1` 中直接构建周期识别逻辑
- 不在 `M1` 中直接写 benchmark 评分
- 不把调度器本身并入 `M1`

## 14. 当前结论

本文档冻结以下结论：

- `M1` 是量化体系的事实地基，不只是抓取模块。
- `Tushare` 是当前唯一 `S1` 正式权威源。
- `M1` 必须同时输出正式事实、正式派生事实、质量状态和任务契约状态。
- `M1` 不拥有调度器，但拥有任务契约定义权。
- `完整 / 真实 / 有效` 必须转化为 Freshness、Coverage、Consistency、Replay 等硬性检查。
- `M1` 的问题必须通过 Attention Item 进入治理体系。
- 第一阶段应优先完成：
  - 日线行情主链正式化
  - 元数据与交易日历正式化
  - 核心派生事实统一化
  - 最小质量治理闭环

## 15. 不在本文档中解决的问题

以下问题明确不在本文档中展开：

- `M2` 如何消费 `M1` 形成多尺度周期状态。
- `M3` 如何把 `M1 + M2` 转成具体交易决策。
- 调度器本身如何重构。
- 各外部数据源的逐字段实施对接方案。
- Attention Queue 的具体前端页面设计。

这些内容将进入后续子模块设计文档。
