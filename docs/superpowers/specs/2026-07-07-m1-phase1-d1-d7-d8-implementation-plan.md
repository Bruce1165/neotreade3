# M1 Phase 1 实施计划（D1 / D7 / D8）

## 1. 目标

本计划只覆盖 `M1 Phase 1` 的首批正式供给落地，不扩展到 `M2/M3` 正式对象实现。

本轮目标只有三个：

1. 将 `D1 / D7 / D8` 从“代码中存在的现实资产”收口为“可正式消费的最小对象”。
2. 为这三个对象建立最小质量门禁，使下游能够区分“数据存在”和“正式可消费”。
3. 在不重构全仓库的前提下，为后续 `M2/M3` 提供单一、可回链、可版本化的首批 `M1` 消费面。

本轮必须产出的核心结果：

- `D1` 有最小正式对象定义与生成路径
- `D7` 有最小正式对象定义与生成路径
- `D8` 有最小正式对象定义与生成路径
- 首批质量状态与 `Freshness Proof` 有明确暴露方式
- API 至少能投影首批正式对象，供后续 `M2/M3` 消费

## 2. 不在本轮完成

- `M2 small_cycle` 正式实现
- `M3 identify_state / tracking_state / entry_state / hold_state / exit_state` 正式实现
- `D2-D6` 正式供给工程
- 前端页面重做
- `M4/M5/M6` 正式消费面工程
- 全量历史回放治理重构

## 3. 当前实施起点

### 3.1 已有现实基础

- `D1` 已有清晰生产主链：`launchd -> update_daily_prices_authoritative -> daily_pipeline_run_view() -> update_daily_prices_authoritative_view() -> daily_prices`
- `D7` 已有稳定运行消费面：`trading_day_view()` 及其底层 `trading_calendar_cache / trading_calendar / trading_calendar_meta`
- `D7` 的最小证券主数据当前主要来自 `stocks`
- `D8` 已有可复用 primitive 路径：`signals["trading_profile"]`

### 3.2 当前结构性缺口

- 正式对象还停留在文档层，尚未独立落到稳定代码位置
- `D8` 当前仍嵌在 API 逻辑中，尚不是正式 `M1` 对象生成链
- `D7` 虽可用，但当前“独立权威程度不足”的边界尚未转成正式质量语义
- `M2/M3` 后续还无法只依赖正式 `M1` 对象消费

## 4. 实施原则

- 先对象，再生成，再门禁，再投影，再验证。
- 先保证边界不漂移，再追求字段覆盖更多。
- 不允许把 `apps/api/main.py` 中的临时计算继续当作长期正式对象边界。
- 不允许把高阶分析对象伪装成 `D8` 扩展字段。
- 不允许在质量不足时静默降级为“照样给下游消费”。
- 不允许为了首批落地而扩散式修改前端、引擎和无关模块。

## 5. 建议落点

首批建议改动边界严格限制在以下位置：

- `neotrade3/data_control/`
- `apps/api/main.py`
- `config/data_control/`
- 必要时新增 `contracts` / `schemas` / `projection` 级代码位置

首批不建议大改：

- `lowfreq_engine_v16_advanced.py`
- 前端页面
- `neotrade3/analysis/` 的高阶分析模块
- 与本轮无关的调度、实验室、学习、治理代码

## 6. 总体分段

本计划建议分为五段执行：

- `P1-A`：正式对象与代码落点建立
- `P1-B`：D1 / D7 / D8 对象生成链收口
- `P1-C`：质量状态与 Freshness Proof 落地
- `P1-D`：最小 API 投影与消费边界切换
- `P1-E`：验证、回归与收口

## 7. 分段实施计划

### P1-A：正式对象与代码落点建立

目标：

- 给 `D1 / D7 / D8` 建立明确的代码级正式对象落点，避免继续散落在 API 临时逻辑中。

任务：

- 选定正式对象承载位置
- 定义首批对象的数据结构
- 定义对象版本字段或对象版本常量
- 定义对象构造入口和输入依赖

建议交付：

- `d1_daily_price_fact`
- `d7_security_master_minimal`
- `d7_trading_day_status`
- `pf1_trading_profile`

完成判定：

- 代码中已经存在首批正式对象定义位置
- 后续生成链不再需要“边写 SQL 边拼 dict 边决定字段名”

### P1-B：D1 / D7 / D8 对象生成链收口

目标：

- 把当前分散现实能力收口成正式生成链。

#### D1

任务：

- 从 `daily_prices` 收口 `D1` 正式对象映射
- 固定正式字段与底层列映射关系
- 明确 `turnover_rate` 当前是可空正式字段而非硬门禁字段
- 固定 `volume_shares` 的股数语义

完成判定：

- 任意目标交易日可从 `daily_prices` 稳定投影出 `d1_daily_price_fact`

#### D7

任务：

- 从 `stocks` 收口 `d7_security_master_minimal`
- 从 `trading_day_view()` 收口 `d7_trading_day_status`
- 把“覆盖不足时可未知”的日历语义正式化
- 显式暴露 `calendar_source`

完成判定：

- `D7` 不再混写为“基础主数据 + 日历 + 停牌推断 + 任意未来映射”的杂合结构

#### D8

任务：

- 将 `signals["trading_profile"]` 从 API 临时计算提升为正式对象生成路径
- 收紧 5 日 / 20 日窗口语义
- 禁止 partial window 继续伪装成正式 `5d/20d` 指标
- 固定 `as_of_trade_date` 语义

完成判定：

- `pf1_trading_profile` 可以独立生成
- 不再需要通过 market intelligence 相关逻辑间接拿 primitive fields

### P1-C：质量状态与 Freshness Proof 落地

目标：

- 让下游能正式判断“是否可消费”，而不是靠隐含约定。

首批至少落地以下状态：

- `source_status`
- `freshness_status`
- `coverage_status`
- `replay_status`

任务：

- 为 `D1` 定义首批 freshness 判断口径
- 为 `D7` 定义覆盖充分与未知状态口径
- 为 `D8` 定义窗口充分性与派生可用性口径
- 为质量失败建立统一暴露方式
- 建立第一批 `attention_item` 语义

关键收口：

- `D1` 数据不存在，不等于“给空对象继续跑”
- `D7` 覆盖不足，不等于“默认非交易日”
- `D8` 窗口不足，不等于“照样输出部分结果”

完成判定：

- 任意对象都能回答“可否正式消费”
- 失败状态可以结构化暴露给后续 `M2/M3`

### P1-D：最小 API 投影与消费边界切换

目标：

- 提供首批正式对象的最小投影面，并明确后续消费边界。

任务：

- 在 API 中增加或收口正式对象投影入口
- 保持现有运行能力不被破坏
- 明确哪些旧路径仍属于兼容读法，哪些路径已经切到正式对象
- 为后续 `M2/M3` 提供单一读取入口

建议策略：

- 首批优先增加“正式对象投影路径”
- 不强制第一步删除全部旧逻辑
- 但新消费面必须明确指向正式对象，而不是继续复用临时拼装 payload

完成判定：

- 后续 `M2/M3` 可以规划为只消费正式对象投影
- API 层不再继续放大临时结构的语义地位

### P1-E：验证、回归与收口

目标：

- 验证首批正式供给成立且未破坏当前主链。

验证维度：

- 对象结构验证
- 字段语义验证
- 质量门禁验证
- 回归兼容验证

建议验证动作：

- `D1`：验证目标交易日行情对象映射是否完整
- `D7`：验证交易日、最近交易日、覆盖区间、来源元信息是否稳定输出
- `D8`：验证 5 日 / 20 日窗口不足时是否返回 `null`
- API：验证正式对象投影路径可稳定返回
- 回归：验证现有主链未因本轮收口而被破坏

建议测试重点：

- 单元测试优先，不做低价值大面积 UI 回归
- 对 `null / unknown / coverage insufficient` 这类边界状态做针对性测试

完成判定：

- `D1 / D7 / D8` 已具备首批正式消费面
- `M2/M3` 的后续实现不再需要先回头重定义 `M1`

## 8. 文件级实施顺序

建议按以下顺序推进：

1. 新增或确定正式对象定义位置
2. 收口 `D1` 对象生成
3. 收口 `D7` 对象生成
4. 收口 `D8` 对象生成
5. 落质量状态与 `Freshness Proof`
6. 增最小 API 投影
7. 补单元测试与回归验证

原因：

- `D8` 依赖 `D1`
- `D7` 的交易日状态会影响 `D8` 与下游消费语义
- 如果先做 API 而不先做对象，仍会延续旧临时结构

## 9. 风险点

### 风险 1：D8 边界再次漂移

表现：

- 把 `theme_momentum`、candidate tags、`market_phase` 等对象重新混入 `D8`

控制：

- 所有 `D8` 字段必须逐项对应契约文档
- 非 primitive 派生一律留在 `M1` 之外

### 风险 2：D7 被写大

表现：

- 把停牌、复牌、完整映射体系、扩展主数据一并塞入首批正式对象

控制：

- `D7` 首批只允许最小证券主数据与交易日状态

### 风险 3：质量门禁流于展示

表现：

- 有状态字段，但下游仍忽略并继续消费

控制：

- 在 API 和对象层同时明确失败语义
- 后续 `M2/M3` 必须按质量状态决定是否消费

### 风险 4：实现扩散

表现：

- 为了首批 `M1` 收口而修改大量无关模块

控制：

- 首批只改计划中明确列出的后端落点

## 10. 验收标准

本计划完成后，应满足以下标准：

1. `D1 / D7 / D8` 已有代码级正式对象，不再只是文档定义。
2. 这三个对象已有稳定生成链，不再依赖 API 内部临时拼接逻辑。
3. 首批质量状态与 `Freshness Proof` 已能明确区分“存在”与“正式可消费”。
4. API 已能最小投影正式对象，供后续 `M2/M3` 接入。
5. 高阶分析对象仍明确留在 `M1` 正式契约之外。

## 11. 下一步

本计划确认后，下一步不应直接大面积编码，而应先拆出一份只覆盖本计划的实现任务清单，压缩到：

- 文件级落点
- 对象级交付物
- 测试级动作
- 阻塞项与决策点

这样进入实现时，边界最稳定，也最不容易发生范围蔓延。
