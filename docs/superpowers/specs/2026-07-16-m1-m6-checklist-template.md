# M1~M6 分层验收 Checklist 模板

日期：2026-07-16

## 1. 文档目标

为 NeoTrade3 的 M1~M6 六层提供可核验、可落地、可计算完成度的统一 Checklist 模板，用于：

- 每层“完成/未完成”的客观判定
- 每层“完成度”的可证伪统计（基于勾选项与证据）
- 每次切片交付后的验收记录落盘

## 2. 使用约定

### 2.1 Checklist 语义

- 每个 Checklist 项必须能被一个或多个“证据”直接支撑（文件/接口/测试/配置/运行产物）。
- 若某项需要“失败即阻断”语义，则在该项的“失败策略”写明 fail-closed；否则写明 degraded（仅展示降级，不阻断主链）。
- 任何不可核验的描述（例如“感觉稳定”“大概可用”）不应出现在 Checklist 项中。

### 2.2 完成度计算口径（可选）

若需要输出数值完成度，使用以下口径（避免主观百分比）：

- 完成度 = 已完成项数 / 总项数
- “已完成项”判定：对应证据可被定位（路径存在、端点可调用、测试可运行或已有可复现脚本）
- 若某项是 fail-closed 门禁项，则必须提供“失败路径的可复现证据”（例如单测覆盖 400/404/500）才可计为完成

## 3. M1 事实层（Fact Layer）

### 3.1 目标

提供可信、统一口径、可追溯的事实数据与质量状态，为上游 M2~M6 提供可依赖的事实地基。

### 3.2 Checklist

- [ ] 事实对象清单已冻结（至少包含核心事实表/派生事实表的范围与命名）
- [ ] 主事实链路的刷新契约已定义（输入/输出/依赖/频率/完成条件）
- [ ] 数据质量证明可输出（完整性、新鲜度、时点一致性、口径稳定性）
- [ ] 缺失与异常可见（Attention 或等价机制，能指向缺口原因与定位线索）
- [ ] 关键事实读取具备只读 API 或内部稳定访问入口（含错误语义）

### 3.3 证据字段

- 文件：事实层设计文档、数据域定义、刷新契约定义
- 数据：事实表/派生事实表 schema 或迁移脚本
- 接口：只读查询端点或内部 service 调用入口
- 测试：关键事实可用性、边界条件、失败语义（fail-closed / degraded）
- 运行产物：质量状态快照、缺口状态快照（如有）

### 3.4 退出条件

- 事实主链具备可复现的“完成判定”，上游不再依赖脚本猜测事实是否可用

## 4. M2 周期识别层（Cycle Layer）

### 4.1 目标

在 M1 事实地基上识别市场/个股的周期结构与阶段状态，并对外输出可消费的周期对象与质量状态。

### 4.2 Checklist

- [ ] 周期对象契约已冻结（字段、语义、版本、可追溯来源）
- [ ] 周期识别入口已存在（可调用/可复现），且依赖清单明确（M1 事实输入范围）
- [ ] 周期结果可读回（只读 API 或内部入口），支持按 key 查询与列表检索
- [ ] 周期质量状态可输出（例如数据不足、模型未收敛、输入缺失等原因枚举）
- [ ] 失败策略明确：关键契约/解析失败 fail-closed；展示可降级 degraded

### 4.3 证据字段

- 文件：M2 设计文档、对象契约定义、版本策略
- 代码：M2 入口函数/服务、对象序列化/反序列化
- 接口：read/list/download（如对外提供）
- 测试：正常路径 + 输入缺失/非法输入/边界数据集
- 运行产物：周期对象样本、运行 ledger（如有）

### 4.4 退出条件

- M2 输出对象可被 M3/M4 稳定消费，且缺失原因可定位

## 5. M3 决策引擎层（Decision Layer）

### 5.1 目标

在 M1/M2 的正式对象基础上输出可审计的交易决策（信号/约束/理由），并保证决策链路可追溯与可回放。

### 5.2 Checklist

- [ ] 决策对象契约已冻结（信号、理由、约束、版本、run_id/source_run_id）
- [ ] 决策生成入口可复现（运行参数、依赖输入、输出落点明确）
- [ ] 决策结果具备 readback/list/download 能力（对外 API 或内部入口）
- [ ] 决策失败语义明确（输入缺失/契约不满足 fail-closed；展示可降级 degraded）
- [ ] 决策可审计（至少包含输入引用与关键派生/中间状态的定位线索）

### 5.3 证据字段

- 文件：M3 设计文档、对象契约定义、错误码/错误语义
- 代码：决策引擎入口、序列化、落盘与读回
- 接口：read/list/download 端点（如对外提供）
- 测试：正常路径 + fail-closed 边界（400/404/500）+ degraded 展示路径
- 运行产物：决策 ledger、决策 artifact（如有）

### 5.4 退出条件

- 决策链路可回放，且外部可通过 readback/download 获取全量证据

## 6. M4 基准评估层（Benchmark Layer）

### 6.1 目标

对 M3 的决策/交易表现进行基准化评估与对比，输出可解释的 benchmark 结果与证据引用。

### 6.2 Checklist

- [ ] Benchmark 对象契约已冻结（评分、对比基准、时间窗口、版本、证据引用）
- [ ] Benchmark 生成入口可复现（输入范围、时间窗口、输出落点）
- [ ] Benchmark 结果可读回（read/list/download 或内部入口）
- [ ] 解释与证据引用可追溯（能定位到 M3/M1 输入或运行产物）
- [ ] 失败策略明确：契约与解析 fail-closed；展示可降级 degraded

### 6.3 证据字段

- 文件：M4 设计文档、对象契约定义
- 代码：benchmark runner、结果聚合、证据引用解析
- 接口：read/list/download（如对外提供）
- 测试：正常路径 + 输入缺失/非法 + 证据引用失效 fail-closed
- 运行产物：benchmark ledger/artifact（如有）

### 6.4 退出条件

- benchmark 输出能稳定驱动 M5 的治理判断与 M6 的交付展示

## 7. M5 治理层（Evolution Controller / Governance Layer）

### 7.1 目标

对候选、验证、拒绝、状态迁移等治理对象提供可审计的闭环机制，并确保“只读聚合可对外，写入路径有门禁”。

### 7.2 Checklist

- [ ] 治理对象契约已冻结（candidate、final_validation、rejection、status_transition、handoff 等）
- [ ] 治理只读 API 三件套齐备（read/list/download，必要时 +download-ledger）
- [ ] 聚合索引能力存在（按 source_run_id 等关键键一站式查看链路可用性）
- [ ] fail-closed 语义一致（空参 400、缺文件 404、非法/内部错误不泄露细节；按系统约定返回 500/4xx）
- [ ] 下载路径具备 root 限制，防 path traversal

### 7.3 证据字段

- 文件：治理对象定义、错误语义与安全约束说明
- 代码：read/list/download handler、root 限制实现（示例：token 归一化 + resolve+relative_to(root)）
- 测试：API 200/400/404/500 覆盖、path traversal 覆盖（示例：`../` 触发 400）
- 运行产物：ledger/artifact 目录结构样本

### 7.4 退出条件

- 治理闭环的每个对象均可被外部审计读取，且链路聚合可定位缺口

## 8. M6 交付与可观测层（Delivery & Observability Layer）

### 8.1 目标

将系统运行状态、门禁、下一步操作建议对外交付为稳定可消费的界面/接口，并保证关键状态可审计、可复现。

### 8.2 Checklist

- [ ] 关键状态汇总入口存在（例如 ops-center/summary 或等价端点）
- [ ] 状态透传齐备（strategy_id/version/status、data_freshness、运行门禁等）
- [ ] next_action 为门禁驱动（仅在门禁满足时输出可执行指令/示例）
- [ ] degraded 路径不阻塞主功能，但能明确提示风险与定位线索
- [ ] 对外输出包含必要的证据引用（可跳转的 URL、可下载的配置或产物）

### 8.3 证据字段

- 文件：交付页面/接口契约、门禁定义
- 接口：汇总端点 payload 示例
- 测试：门禁分支、degraded 分支、指令/示例输出稳定性
- 运行证据：一次可复现的端到端运行记录（含 date 策略与边界错误）

### 8.4 退出条件

- 外部操作者无需阅读代码即可判断“现在能不能跑、下一步做什么、风险在哪里”
