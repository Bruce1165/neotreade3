# NeoTrade3 前端文案审计第 2 批修正（设计定稿）

日期：2026-06-17  
范围：`neotrade3-dashboard` 全局公共文案 2 处

## 1. 背景

- 第一轮前端文案审计已完成，并已修正 `Lowfreq` 与 `Screeners` 中 5 处高风险文案。
- 第二轮继续审计后，确认当前正式前端里仍有 2 处“字面上在陈述事实，但代码并未真实支撑”或“与 authoritative 正式口径不一致”的公共文案。
- 这 2 处都位于全局公共区域，影响所有页面，不宜继续保留原文。

## 2. 本轮问题

### 2.1 Header 中的 `API: Connected`

现状：
- `Header()` 里直接写死：
  - `API: Connected`

问题：
- 该文案不是动态状态，而是静态文本。
- 当前代码中没有任何 API 健康检查、连接状态订阅或条件判断支撑“Connected”这个事实性表达。
- 因此前端即使在 API 不可用时，也仍可能显示 `Connected`，属于事实性不准确文案。

### 2.2 `Tushare 黄旗` 文案中的“备用数据源可能不可用”

现状：
- 当前黄旗文案为：
  - `检测到积分不足，备用数据源可能不可用。`

问题：
- 当前 authoritative 正式口径已经明确：
  - `Tushare` 是主源
  - `Tencent` 仅作 `daily_prices` safety-net
- 因此 `Tushare` 积分不足的直接影响是“主源可能受影响”，而不是“备用数据源可能不可用”。
- 原文会把影响方向表达反，属于与正式口径不一致。

## 3. 方案比较

### 3.1 方案 A：改成静态中性文案 + 修正黄旗影响描述

- `API: Connected` 改成中性、不宣称连接状态的文案
- `Tushare 黄旗` 改成直接描述主源影响的文案

优点：
- 改动最小
- 风险最低
- 与当前真实代码能力保持一致

缺点：
- Header 不再提供“看起来像实时状态”的表达

### 3.2 方案 B：保留 `Connected`，补做真实连接检测

- 通过健康检查接口或数据状态接口驱动 Header 状态

优点：
- 如果实现完整，表达更强

缺点：
- 已超出本轮“纯文案修正”边界
- 会引入新的状态管理与错误处理

### 3.3 结论

- 采用方案 A。

## 4. 设计

### 4.1 Header 文案

当前：
- `API: Connected`

改为：
- `API: 本地模式`

理由：
- 当前前端默认本机访问环境，不需要在 Header 虚构一个实时连接状态。
- `本地模式` 是中性事实表达，不会误导用户以为页面正在做实时连接监测。

### 4.2 GlobalBanner 黄旗文案

当前：
- `检测到积分不足，备用数据源可能不可用。`

改为：
- `检测到 Tushare 积分不足，日线主源可能受影响。`

理由：
- 与当前正式 authoritative 口径一致。
- 直接指向真实受影响对象：`Tushare` 主源。
- 不再把影响误表述为“备用数据源”。

### 4.3 本轮边界

- 仅修改上述 2 处公共文案。
- 不新增 API 健康检查逻辑。
- 不改动黄旗显示触发条件。
- 不扩展到其他 Header / Sidebar / 页面级说明文案。

## 5. 验证

- 更新相关前端测试，覆盖：
  - Header 中不再出现 `Connected`
  - Header 出现新的静态文案
  - 黄旗触发时显示新的主源影响文案
- 运行：
  - `npm run test -- --run`
  - `npm run build`

## 6. 完成标准

- 页面 Header 不再用静态文字宣称 API 已连接。
- `Tushare 黄旗` 文案与 authoritative 主源口径一致。
- 前端测试与构建通过。
