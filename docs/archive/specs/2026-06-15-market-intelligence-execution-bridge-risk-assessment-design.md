# 主线审阅建议层桥接低频执行层风险评估（以交易风险为主）

日期：2026-06-15

## 1. 评估目标

- 本文只回答一个问题：当前 `market_intelligence` 建议层，是否适合正式桥接到低频执行层。
- 本文不包含以下内容：
  - 不设计新的执行接口
  - 不设计新的状态机
  - 不设计新的前端交互
  - 不直接推动代码实现
- 本文的判断角度以交易风险为主，而不是以“功能是否已经存在”为主。

## 2. 当前事实基础

### 2.1 当前执行层的性质

- 当前低频执行层仍属于模拟执行通道，核心行为是：
  - 生成 `buy_intent / sell_intent`
  - 在 `execute_date` 执行
  - 失败时顺延
  - 写入 ledger 和告警
- 当前执行链路已经具备“意图 -> 执行 -> 顺延 -> 归档”的闭环能力。
- 因此，一旦把上层建议层正式接入执行层，影响的不是页面展示，而是实际的排队、成交、顺延与记账结果。

### 2.2 当前主线审阅系统的性质

- 当前 `market_intelligence` 已具备：
  - 主题榜
  - 统一候选清单
  - 建议层
  - 主线审阅面板
  - 决策摘要
- 当前前端已在 `neotrade3-dashboard/` 中完成独立主线审阅页承接，可用于人工审阅。

### 2.3 当前真实库的数据约束

- 当前真实库已确认存在：
  - `daily_prices`
  - `ths_concept_daily`
  - `stocks`
- 当前真实库未见：
  - `fund_portfolios`
  - `research_reports`
  - `report_consensus`
  - `institutional_surveys`
- 这意味着当前真实运行状态下：
  - `trading_leader` 路径可产出候选
  - `config_leader / institutional_attention` 路径缺少真实数据源支撑

### 2.4 当前建议层的真实运行结果

- 当前统一候选在真实库中基本呈现为：
  - 候选几乎全部来自 `trading_leader`
  - `candidate_type_count=1` 为绝对主流
  - 多角色共振当前基本无法形成
- 为控制误抬升风险，系统已经做了最小收紧：
  - `单角色 + trading_leader + AI 宽口径` 从 `推荐` 下调为 `观察`
- 在这一规则下，当前真实结果已出现：
  - `推荐 = 0`
  - `观察 > 0`
- 这说明当前建议层已更适合作为审阅信号，而不是执行信号。

## 3. 交易风险分层

### 3.1 高风险：单一路径证据被直接放大为执行动作

- 当前真实候选几乎全部来自 `trading_leader`，意味着如果桥接执行层，执行意图将被单一路径证据主导。
- `trading_leader` 的本质是交易活跃度与近端强度代理，不等于配置型共识，也不等于机构确认。
- 在执行层中，这种单一路径证据会被放大为：
  - 交易排队
  - 执行价格尝试
  - 失败顺延
  - 资金占用与持仓路径变化
- 这类放大风险不是页面误导，而是会直接改变模拟账户行为。

### 3.2 高风险：宽口径标签被误用为执行准入

- 当前 AI 标签采用宽口径纳入，目标是“保留可能相关”，而不是“证明足够可交易”。
- 即使当前已经对建议层做了收紧，仍不能说明：
  - 这些候选已具备执行层所需的充分证据
  - 这些候选可以稳定跨越交易噪声
- 如果在当前阶段把建议层直接接入执行层，等于把“宽召回的审阅标签”升级成“执行准入条件”，存在明显口径错位。

### 3.3 中风险：建议层当前处于阶段性失真状态

- 当前真实结果已经出现“全观察、无推荐”。
- 这不是逻辑错误，而是当前数据就绪性与风险控制下的阶段性结果。
- 但这也说明：
  - 当前建议层还没有形成稳定、可直接下游消费的分层输出
  - 若现在强行桥接执行层，只能得到两个不理想结果之一：
    - 要么几乎不触发执行
    - 要么为了触发执行而回退风险控制

### 3.4 中风险：解释锚点虽已改善，但仍不等于执行确定性

- 当前 `best_concept_name` 已通过多轮最小修正，解释质量有明显改善。
- 但“解释更合理”并不等于“执行更安全”。
- 锚点解释更适合支撑人工审阅，不足以单独作为正式执行准入证据。

### 3.5 低风险：桥接评估本身可先行，不影响现有系统

- 当前若只停留在文档评估层，不修改执行接口、不新增自动触发，就不会引入新的实质交易风险。
- 因此，当前阶段最安全的推进方式是：
  - 先做评估
  - 暂不桥接
  - 保持主线审阅系统作为人工上层决策参考

## 4. 核心判断

### 4.1 当前不建议正式桥接执行层

- 结论：**当前不建议把主线审阅建议层正式桥接到低频执行层。**

### 4.2 原因

- 原因 1：证据结构不完整
  - 当前几乎只有 `trading_leader` 真正产出，缺少 `config / institutional` 两条路径支撑。
- 原因 2：执行风险会被直接放大
  - 审阅层信号一旦接入执行层，将立刻影响排队、顺延、成交和记账。
- 原因 3：当前建议层定位仍偏人工审阅
  - 当前输出更适合作为“先看主线、再看候选”的人工判断辅助，而不是自动执行入口。
- 原因 4：当前风险控制与执行目标尚未兼容
  - 现有风险控制把大量单交易型候选降为 `观察`，这有助于降低误买，但也说明当前系统还没进入“适合自动执行”的证据成熟阶段。

## 5. 当前阻断项

- 阻断项 1：真实库缺少 `config_leader` 数据源
- 阻断项 2：真实库缺少 `institutional_attention` 数据源
- 阻断项 3：统一候选当前缺乏多角色共振分布
- 阻断项 4：建议层当前缺乏稳定可消费的“推荐层”输出
- 阻断项 5：当前系统虽已达到“审阅可用”，但尚未达到“执行可用”

## 6. 建议结论

- 当前阶段建议：
  - 保持 `market_intelligence` 作为上层人工审阅系统
  - 不将其正式桥接为低频执行入口
  - 在 `config / institutional` 数据源未补齐之前，不推动执行层接入
- 当前最合理的定位是：
  - `trading-only + 审阅可用`
  - 而不是 `证据完整 + 可执行`

## 7. 后续动作建议

- 短期：
  - 继续保持主线审阅页用于人工复核
  - 不新增执行入口
- 中期：
  - 补齐 `config / institutional` 两条候选路径的数据源
  - 观察统一候选是否开始出现稳定的多角色共振
- 长期：
  - 待证据结构完整后，再单独设计“桥接条件与回滚条件”

## 8. 结论摘要

- 当前不建议正式桥接执行层。
- 不是因为页面、接口或排序未完成，而是因为当前真实运行环境下，执行层将直接放大单一路径交易证据的风险。
- 当前系统已达到“可审阅”，但尚未达到“可正式下接执行”。
