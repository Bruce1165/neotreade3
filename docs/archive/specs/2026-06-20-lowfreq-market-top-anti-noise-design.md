# 低频离场抗噪进化：`market_top` 延迟确认设计

日期：2026-06-20  
范围：`lowfreq_engine_v16_advanced.py` 中 `market_top` 的判定与执行语义、对应审计输出、A/B 回测验证口径

## 1. 背景

在最近一轮 `2025 Top200` 容量实验中，模型已经明显改善了“看见牛股但买不进去”的问题：

- 放宽市值带后，`picked_count` 从 `50` 升至 `135`
- `5000万 + 8仓 + 仅一字板阻断` 实验下，`bought_count` 从 `11` 升至 `25`
- `held_to_top_count` 从 `1` 升至 `3`

但提前离场问题仍然显著：

- `market_top` 仍是主要提前离场原因
- 大量仓位在主升段中的正常回调或震仓阶段被卖出

当前卖出主链为：

1. `market_top`
2. `market_drawdown`
3. `stock_drawdown`

其中 `market_top` 当前仍采用“单日结构成立即卖出”的语义。这会把低频交易中常见的短时回调、均线扰动和广度收缩误判成真正见顶。

用户已明确要求：

- 下一阶段优先解决“如何摆脱干扰（回调 / 震仓），更持久拿住”
- 低频交易应当能够明确区别噪音与真实见顶
- 离场信号逻辑需要持续调整，并形成可循环进化的闭环

## 2. 目标

本轮设计只解决一个核心问题：

- 让 `market_top` 从“单日触发即卖”演化为“延迟确认后再卖”，优先过滤主升段中的正常回调 / 震仓噪音

本轮不解决：

- 本金按需投入统计口径
- 分层减仓
- 强趋势豁免
- `market_drawdown` / `stock_drawdown` 的阈值重构
- 更换 `market_top` 的底层结构因子

## 3. 方案比较

### 3.1 方案 A：延迟确认

- `market_top` 第一次成立时不直接卖出，而是进入观察状态
- 只有在短窗口内再次成立，才确认卖出

优点：

- 直接对应用户提出的“抗噪”目标
- 改动范围小，便于做 A/B 对照
- 不引入新的复杂卖出分支

缺点：

- 真正见顶时，离场会比当前模型更晚

### 3.2 方案 B：分层离场

- 第一次结构转弱先减仓，后续再确认是否清仓

优点：

- 能同时兼顾抗噪与风险控制

缺点：

- 显著扩大问题范围
- 引入新的仓位管理语义，不适合本轮单变量验证

### 3.3 方案 C：强趋势豁免

- 对个股趋势足够强的持仓，即使市场转弱也暂缓卖出

优点：

- 更贴近“拿住牛股”

缺点：

- 将市场信号与个股趋势状态混合，边界更复杂
- 容易与后续趋势持有策略耦合

结论：

- 本轮采用 `方案 A：延迟确认`

## 4. 设计原则

- 本轮只修改 `market_top` 的执行语义，不修改其底层因子定义
- 只改一个变量，保证实验结果可归因
- `market_drawdown` 与 `stock_drawdown` 继续保留即时生效
- 必须增加可审计的中间状态，支撑后续“循环进化”
- 不允许把“抗噪”直接做成“整体迟钝”，因此必须保留 A/B 验证和失败样本复盘

## 5. 现状

当前 `market_top` 条件为：

- 跌破 `20 日均线`
- `20 日均线` 走弱（当前口径：`ma20_today <= ma20_3d_ago`）
- 对应市场样本中，站上 `20 日均线` 的股票占比 `< 40%`

当前执行语义：

- 若 `breadth_weak` 成立，且 `break_ma20` 或 `ma20_weak` 任一成立
- 当天直接触发 `SellSignal("market_top", ...)`

这是当前抗噪不足的根源。

## 6. 新设计

### 6.1 核心思路

保留现有 `market_top` 结构条件不变，但将其执行语义改为两阶段：

1. `watch`
   - 第一次满足 `market_top` 结构条件时，只记录观察态，不卖出
2. `confirm`
   - 在短窗口内再次满足，则确认卖出

### 6.2 推荐参数

- `MARKET_TOP_WATCH_WINDOW = 3`
- `MARKET_TOP_CONFIRM_HITS = 2`

含义：

- 第一次触发后，进入 `3` 个交易日观察窗口
- 在该窗口内累计满足 `2` 次，确认 `market_top`
- 如果窗口到期仍未累计到 `2` 次，则视为噪音，观察态失效

### 6.3 状态机

每笔持仓新增 `market_top_watch_state`，状态只有：

- `idle`
- `watching`

其中：

- `idle -> watching`
  - 第一次满足 `market_top` 条件
  - 记录：
    - `watch_start_date`
    - `watch_expire_date`
    - `watch_hits`
    - `watch_last_reason_snapshot`
- `watching -> sell`
  - 在窗口内再次满足 `market_top` 条件，且累计命中数达到 `2`
  - 当天正式触发 `SellSignal("market_top", ...)`
- `watching -> idle`
  - 观察窗口过期，累计命中数不足 `2`
  - 视为一次噪音扰动，状态清空

### 6.4 抖动处理

在 `watching` 期间：

- 当天不满足条件，不立即重置状态
- 只要仍处于窗口内，后续再次满足仍可累计命中
- 这样可以避免“满足一天 -> 回落一天 -> 再满足一天”被拆成两次独立噪音

### 6.5 与其他离场层共存

本轮推荐将卖出顺序调整为：

1. `market_drawdown`
2. `stock_drawdown`
3. `market_top_confirmed`

语义如下：

- 若 `market_drawdown <= -10%`，当天直接卖
- 若 `stock_drawdown <= -20%`，当天直接卖
- 只有“结构转弱但尚未形成更硬的环境/个股恶化”时，才进入 `market_top watch`

这样能避免 `market_top` 再次作为单日噪音型条件压在最前面一刀切。

## 7. 审计输出

本轮必须为 `market_top` 增加可追踪事件，以支持后续循环进化。

最少记录三类事件：

- `market_top_watch_started`
- `market_top_watch_expired`
- `market_top_confirmed`

每条事件至少记录：

- 日期
- 股票代码
- 市场代理
- `break_ma20`
- `ma20_weak`
- `breadth_ratio`
- 当前观察窗口第几天
- 当前累计命中数

## 8. A/B 验证口径

必须做单变量对照：

- `A组`：当前基线离场逻辑
- `B组`：只加入 `market_top` 延迟确认

两组必须保持以下条件一致：

- 市值带
- 初始资金 / 资金实验口径
- 持仓数
- 成交约束
- 买入逻辑

## 9. 成功标准

本轮成功不只看收益，至少同时满足以下方向性改进：

- `market_top` 卖出次数下降
- `held_to_top_count` 上升
- “卖出后继续大涨”的误杀样本占比下降
- 总回撤不上升过多
- 总收益不出现显著恶化

## 10. 循环进化闭环

后续每轮离场迭代都按同一闭环推进：

1. 统计 `market_top` 卖出样本
2. 找出误杀样本（卖后继续涨）
3. 找出迟钝样本（延迟后回撤扩大或收益恶化）
4. 只针对占比最高的失效模式改一条规则
5. 重跑 A/B 对照

这样保证“抗噪”与“迟钝”两个方向都被量化，而不是凭主观感觉反复调规则。

## 11. 风险

- 延迟确认会让真正见顶时的离场更晚
- 将 `market_top` 放到 `market_drawdown` 与 `stock_drawdown` 之后，可能会改变部分样本的卖出归因
- 如果审计输出不完整，后续将难以判断改动到底减少了噪音，还是只是拖慢了反应速度
