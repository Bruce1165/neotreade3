# NeoTrade3 外网 API 超时韧性优化设计（设计定稿）

日期：2026-06-18  
范围：当前主前端 `neotrade3-dashboard` 在外网访问场景下的页面首屏与刷新链路

## 1. 背景与目标

- 当前外网访问时，经常出现页面首屏或刷新阶段 API 请求超时。
- 现有前端页面普遍使用 `Promise.all()` 并发请求多个接口。
- 一旦其中任意一个接口超时或失败，整页会直接进入错误状态。
- 前端默认请求超时仅为 `15000ms`。
- `frontend_gateway` 代理层当前超时为 `300000ms`，明显高于前端超时，因此问题更可能出在前端加载编排而不是网关硬超时。

目标：
- 降低外网场景下“单个慢接口拖垮整页”的概率。
- 页面应优先显示核心块，重块允许延迟加载。
- 某个区块请求失败时，只影响该区块，不影响整页主要信息可用性。
- 在不修改后端接口契约的前提下，先通过前端加载结构优化提高韧性。

## 2. 现状证据

### 2.1 前端默认超时偏短

- `fetchApi()` 默认超时为 `15000ms`。
- 这对外网环境下的冷启动、网络抖动、重接口计算并不稳妥。

### 2.2 `Overview` 存在典型“一个失败全页失败”

- `Overview` 首屏通过 `Promise.all()` 一次并发 5 个接口：
  - `/api/data/status`
  - `/api/sectors/hot`
  - `/api/concepts/mainline`
  - `/api/lowfreq/execution/queue`
  - `/api/lowfreq/backtest/window-summary`
- 其中回测窗口接口超时设置为 `30000ms`，本身就是重接口。
- 当前实现里任一接口失败，都会触发页面级 `catch`，导致整页报错。

### 2.3 `Screeners` 同样存在整页级耦合

- `Screeners` 页面使用 `Promise.all()` 并发拉：
  - `/api/screeners`
  - `/api/screeners/runs`
  - `/api/screeners/bulk-runs`
- 任一接口失败也会导致整页进入错误态。

### 2.4 网关层当前不是主要瓶颈

- `frontend_gateway` 代理请求超时配置为 `300000ms`。
- 这说明目前更可能是前端主动超时或整页级失败传播，而不是网关太早断开。

## 3. 问题归因

本次问题的核心不在于“所有接口绝对性能都不足”，而在于：

- 页面把多个不同重量级接口绑成一个成功/失败整体。
- 页面没有区块级 loading/error 隔离。
- 页面把重接口和首屏核心信息放在同一请求阶段。
- 默认前端超时对外网场景偏保守。

因此，当前更应优先解决“首屏编排脆弱性”，而不是先做大规模后端性能工程。

## 4. 方案比较

### 4.1 方案 A：只调大前端超时

- 仅把默认 `15000ms` 提升到更长。
- 页面仍保留 `Promise.all()`。

优点：
- 改动最小。

缺点：
- 只能延后失败时间。
- 不能解决“单点失败拖垮整页”。

### 4.2 方案 B：页面分块加载，局部失败不拖全页

- 按页面区块拆分数据请求。
- 核心区块优先加载。
- 次核心和重区块分阶段加载。
- 每个区块独立处理 loading/error/retry。

优点：
- 与当前问题最匹配。
- 风险低，不依赖后端改造。
- 能显著改善外网体感和实际可用性。

缺点：
- 需要调整页面状态组织方式。

### 4.3 方案 C：前端分块 + 后端聚合/缓存

- 在方案 B 基础上，再增加轻量聚合接口或缓存层。

优点：
- 效果最好。

缺点：
- 范围扩大到后端接口与缓存策略，不适合作为第一步。

结论：
- 本次采用方案 B。

## 5. 设计原则

- 页面优先展示核心信息，不等待所有区块都返回。
- 单个区块失败，不阻塞其它区块渲染。
- 重接口应后置，不与首屏核心块绑定为同一成功条件。
- 页面整体错误只在核心块整体失败时显示。
- 不先修改后端字段与接口语义。

## 6. 页面拆分方案

### 6.1 `Overview`

核心块：
- `/api/data/status`
- `/api/concepts/mainline`

次核心块：
- `/api/sectors/hot`
- `/api/lowfreq/execution/queue`

重块：
- `/api/lowfreq/backtest/window-summary`

规则：
- 页面先请求核心块。
- 核心块成功后即可显示总览主内容。
- 次核心块随后加载，不阻塞首屏。
- 回测块独立 loading/error/retry，不再拖垮整页。

### 6.2 `Lowfreq`

核心块：
- `/api/market-phase`

次核心块：
- `/api/sectors/hot?mode=ths_concept&include_portfolio=0&include_sell_signal=0`

重块：
- 历史回测结果
- 报告列表
- 下载区

规则：
- 页面先显示市场阶段。
- 候选池和热门板块作为第二阶段加载。
- 回测和报告区延后，或在用户进入相关 tab/区块时再加载。

### 6.3 `Screeners`

核心块：
- `/api/screeners`

次核心块：
- `/api/screeners/runs`

重块：
- `/api/screeners/bulk-runs`
- 单个筛选器配置详情

规则：
- 先显示筛选器列表。
- 最近运行记录单独加载。
- 批量运行记录与配置详情延后，不阻塞首屏。

## 7. 状态管理设计

### 7.1 页面不再只维护单一 `loading/error`

每个页面改为维护多个 block 状态。

每个 block 至少包含：
- `data`
- `loading`
- `error`
- `loaded`

### 7.2 页面级错误边界

- 页面总错误仅用于“核心块全失败”的情况。
- 非核心块失败时，只在对应区块显示局部错误。

### 7.3 区块级重试

- 每个区块提供独立 retry。
- retry 只重新请求该 block，不重刷整个页面。

## 8. 请求编排设计

### 8.1 不再使用整页 `Promise.all()`

整页请求编排不再要求“所有请求都一起成功才 setData”。

可采用两类实现方式之一：
- 分阶段串并结合：
  - 先发核心块
  - 再发次核心块
  - 最后发重块
- 或 `Promise.allSettled()`：
  - 但结果必须按 block 分开写回状态
  - 不能再统一 `catch` 成页面级错误

### 8.2 分阶段加载顺序

标准顺序：
1. 核心块
2. 次核心块
3. 重块

如果页面存在 tab 式内容：
- 非首屏 tab 对应的重块延迟到 tab 激活时再请求。

## 9. 超时策略

不采用“所有接口统一加大到同一超时”的做法。

采用分层超时：
- 核心轻接口：`10000ms - 15000ms`
- 次核心接口：`20000ms`
- 重接口：`30000ms - 45000ms`

原则：
- 核心块要快速失败并可提示重试。
- 重块允许更长等待，但不得阻塞首屏。

## 10. 公共抽象

### 10.1 新增轻量 block 请求 helper 或 hook

建议新增一个轻量公共抽象，例如：
- `useAsyncBlock`
- 或 `loadBlock()`

职责仅限：
- 发请求
- 设置 `loading/error/data/loaded`
- 支持 retry

非目标：
- 不在本次引入全局缓存层
- 不引入复杂状态管理库
- 不把所有页面逻辑都抽成统一框架

### 10.2 `fetchApi()` 保持基础能力

- `fetchApi()` 继续作为底层请求工具。
- 页面通过显式 `timeoutMs` 指定不同块的超时。
- 本次不把 `fetchApi()` 升级成复杂调度器。

## 11. UI 行为

每个区块必须独立处理以下三态：
- `loading`
- `error`
- `loaded`

区块错误展示要求：
- 显示简短错误信息
- 提供“重试”按钮
- 不遮挡其它区块

区块 loading 展示要求：
- 可使用简单 loading text 或 skeleton
- 不要求本次统一成一套复杂骨架屏体系

## 12. 实施顺序

严格顺序如下：

1. 改 `Overview`
2. 改 `Screeners`
3. 改 `Lowfreq`
4. 抽取必要的 block 请求 helper
5. 补页面级回归测试
6. 执行前端构建验证

说明：
- `Overview` 最典型，收益最大，应优先落地。
- `Screeners` 结构较清晰，适合作为第二个页面。
- `Lowfreq` 页面更大，放在最后更稳妥。

## 13. 测试与验证

### 13.1 页面定点测试

至少覆盖：
- 某个 block 失败时，其它 block 仍能渲染
- retry 只重试对应 block
- 重块失败时页面核心块仍可用

### 13.2 构建验证

- `npm test -- --run ...`
- `npm run build`

### 13.3 手工验证重点

- 外网或模拟慢网环境下，`Overview` 首屏不再因回测块慢而整页报错
- `Screeners` 列表可先显示，运行记录慢时只影响该区块
- `Lowfreq` 核心信息可先显示，重块延后加载

## 14. 完成标准

- 页面首屏请求不再采用“单点失败拖垮整页”的加载结构。
- 每个关键页面均具备区块级 loading/error/retry。
- 重接口从首屏强依赖中拆出。
- 页面在外网慢网络场景下仍能优先展示核心信息。
- 定点测试与前端构建通过。
