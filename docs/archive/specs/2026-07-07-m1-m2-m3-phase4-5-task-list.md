# M1 第一阶段 + M2/M3 最小闭环 Phase 4-5 执行任务清单

日期：2026-07-07  
对应计划：`docs/superpowers/specs/2026-07-07-m1-m2-m3-minimal-closed-loop-implementation-plan.md`

## 1. 任务清单目标

本清单只覆盖实施蓝图中的：

- `Phase 4`：M1-M2-M3 集成回放与最小接口投影
- `Phase 5`：测试、回归与验收收口

本清单的目标不是扩大功能范围，而是把第一批最小正式主链真正闭合到：

- API 可消费
- 回放可重建
- 测试可验证
- 回归可比较
- 验收可判定

本清单不覆盖：

- `M4/M5/M6` 的完整工程实现
- 前端大规模页面重构
- 完整的告警平台或治理平台
- 中周期/大周期接入
- 复杂报表、分析平台与多角色交付面

## 2. Phase 4：M1-M2-M3 集成回放与最小接口投影

### 2.1 目标

- 让 `M1 -> M2 -> M3` 成为一条可跑、可回放、可最小消费的正式闭环。
- 为后续 `M4/M5/M6` 预留最小消费面，但不提前实现它们的完整工程。

### 2.2 任务组 A：最小 API 投影面

#### Task A1：冻结首批 API/Schema 输出对象

目标：

- 将 `M1/M2/M3` 首批正式对象以最小权威输出形式接入 API。

至少投影：

- `M1`
  - `D1` 最小正式事实对象
  - `D7` 最小正式事实对象
  - `D8` 最小正式派生对象
  - 质量状态 / `Freshness Proof`
- `M2`
  - `cycle_state`
  - `state_stability_level`
  - `state_transition_log`
  - `confidence`
  - `invalidation`
- `M3`
  - `identify_state`
  - `tracking_state`
  - `entry_state`
  - `hold_state`
  - `exit_state`
  - `decision_lifecycle_log`

建议落点：

- `apps/api/main.py`
- 对应 serializer/schema/view model 位置

完成判定：

- 外部消费者不再需要从混杂旧字段中猜正式对象。

#### Task A2：冻结最小接口字段纪律

目标：

- 防止 API 在首批接线时重新压平正式语义。

必须满足：

- 保留来源层边界
- 不压平退出分类
- 不隐藏局部/全局退出语义字段
- 不省略版本字段与 trace 引用

至少保留：

- `rule_version`
- `input_data_version`
- 必要的 `trace_id`
- `local_exit_semantics`
- `global_thesis_end_semantics`

完成判定：

- API 可被后续 `M4/M5/M6` 稳定消费，不需要二次猜测。

### 2.3 任务组 B：M1-M2-M3 单股最小回放链

#### Task B1：冻结单股回放输入集合

目标：

- 明确回放一个单股时间窗口所需的最小对象集合。

至少包括：

- 对应日期窗口内的 `M1 D1/D7/D8`
- 同窗口内的 `M2` 周期状态对象
- 同窗口内的 `M3` 行为状态对象
- 相关版本字段

完成判定：

- 回放不再依赖“重新执行所有旧逻辑才能勉强还原”。

#### Task B2：冻结单股回放输出结构

目标：

- 为单股回放定义统一输出结构。

至少包括：

- 某日 `M1` 事实状态
- 某日 `M2` 周期状态
- 某日 `M3` 行为状态
- 关键状态迁移时间点
- 退出语义分类
- 生命周期 trace 引用

完成判定：

- 同一只股票的最小正式链路已可稳定重建。

#### Task B3：冻结回放护栏

目标：

- 确保首批回放不发生未来信息倒灌或后验美化。

必须满足：

- 回放只使用对应时点正式对象
- 若对象后续重算，需显式展示版本差异
- 禁止以当前最终结果覆盖当时对象状态

完成判定：

- 回放输出与 `M2` 的 replay consistency 约束一致。

### 2.4 任务组 C：M4/M5/M6 最小消费面预留

#### Task C1：为 `M4` 预留最小消费面

目标：

- 使 `M4` 后续能正式评估 `M3`，但不在本阶段实现 `M4`。

至少预留：

- 行为状态序列
- 状态迁移理由对象
- 持有质量语义字段
- 退出分类字段
- 协同风险相关标记

完成判定：

- `M4` 后续接入时不需要重新改写 `M3` 输出结构。

#### Task C2：为 `M5` 预留最小消费面

目标：

- 使 `M5` 后续能构造 `diagnostic_chain`，但不在本阶段实现 `M5`。

至少预留：

- 生命周期链
- 低频纪律命中/违反记录
- guardrail 触发记录
- trace/ref/version 对象

完成判定：

- `M5` 后续接入时不需要再次向 `M3` 要“补充解释字段”。

#### Task C3：为 `M6` 预留最小消费面

目标：

- 使 `M6` 后续能做高保真投影，但不在本阶段做页面重构。

至少预留：

- 来源层标记
- 正式分类字段
- 回链所需 trace
- 局部/全局退出语义字段

完成判定：

- `M6` 后续接入时不会因字段缺失而自行补造语义。

### 2.5 任务组 D：兼容与过渡策略

#### Task D1：冻结旧字段兼容映射表

目标：

- 明确旧 API/旧页面/旧引擎输出字段与新正式对象的映射关系。

至少梳理：

- 旧 buy/sell 相关字段
- 旧 tracking/audit 字段
- 旧 wave/phase 相关字段

完成判定：

- 首批接线不会因“旧字段还在被消费”而直接破坏现有链路。

#### Task D2：冻结过渡期输出策略

目标：

- 明确首批是：
  - 并行输出旧字段 + 新正式字段
  - 还是直接切换到新字段

建议：

- 首批优先并行输出，待回归通过后再考虑收口旧字段。

完成判定：

- 接线策略明确，不在实施时临时摇摆。

### 2.6 Phase 4 完成标志

- API 已能最小投影 `M1/M2/M3` 正式对象
- 单股最小回放链已可稳定重建
- `M4/M5/M6` 所需最小消费面已预留
- 旧字段兼容策略已明确

## 3. Phase 5：测试、回归与验收收口

### 3.1 目标

- 验证最小正式主链不只“能输出”，而且“能验证、能回归、能验收”。

### 3.2 任务组 E：测试收口

#### Task E1：冻结首批测试矩阵

目标：

- 把 `M1/M2/M3` 的首批测试组织成统一矩阵，而不是散落补丁。

建议矩阵：

- `M1` 契约与门禁测试
- `M2` 状态机与回放边界测试
- `M3` 行为状态机与低频纪律测试
- API 投影结构测试
- 单股回放结构测试

完成判定：

- 第一批测试清单完整且可执行。

#### Task E2：冻结重点回归清单

目标：

- 把现有低频引擎的重要回归面保留下来，避免对象化重构破坏已有增量成果。

至少回归：

- `tests/unit/test_lowfreq_engine_v16_signal_convergence.py`
- `tests/unit/test_lowfreq_engine_v16_sell_logic.py`
- `tests/unit/test_lowfreq_intent_conflicts.py`

同时补充：

- tracking 门槛是否仍生效
- 局部/全局退出分离是否成立
- 退出分类是否稳定

完成判定：

- 新正式对象链没有破坏现有低频关键保护。

### 3.3 任务组 F：双轨对比与验收

#### Task F1：冻结双轨对比范围

目标：

- 定义“旧行为口径 vs 新正式对象口径”的首轮对比方式。

至少比较：

- 行为状态序列
- tracking 是否被绕过
- 退出分类是否更清晰
- 局部/全局退出是否分离
- 单股回放是否更完整

完成判定：

- 首批实施不是“直接替换”，而是有一轮明确的结构对比。

#### Task F2：冻结首批验收标准

目标：

- 将“最小闭环成立”压成正式验收条件。

首批验收至少包括：

- `M1` 正式供给对象已可用
- `M2` `small_cycle` 正式对象可被回放
- `M3` 行为状态链已正式对象化
- 局部/全局退出语义已严格分离
- API 可稳定输出最小正式对象
- 单股最小回放链成立
- 关键回归测试通过

完成判定：

- 验收不再停留在“看起来差不多”。

### 3.4 任务组 G：阻塞与失败处理

#### Task G1：冻结验收失败分流规则

目标：

- 若首批验收不通过，明确问题是回退到哪一阶段处理。

建议分流：

- 数据契约问题 -> 回到 `Phase 1`
- `small_cycle` 状态/回放问题 -> 回到 `Phase 2`
- 行为语义/低频纪律问题 -> 回到 `Phase 3`
- API/回放投影问题 -> 回到 `Phase 4`

完成判定：

- 验收失败后不会陷入“所有问题一起重开”的混乱状态。

#### Task G2：冻结首批阻塞项

当前重点阻塞项：

- 若旧 API/页面强依赖旧字段，双轨兼容成本可能上升
- 若 `M2` 回放版本字段仍不稳定，回放验收可能被卡住
- 若 `M3` 行为状态与现有引擎执行流程耦合过深，生命周期审计输出可能需要过渡层

完成判定：

- 首批验收的主要阻塞点已显式暴露。

### 3.5 Phase 5 完成标志

- 首批测试矩阵已冻结并执行
- 重点回归清单已通过或完成归因
- 双轨对比已完成
- 首批验收标准已完成判定
- 失败分流规则已明确

## 4. Phase 4-5 完成标志

满足以下条件时，视为第一批 `M1 第一阶段 + M2/M3 最小闭环` 任务链真正闭合：

- `M1` 正式对象已能被 `M2/M3` 消费
- `M2` `small_cycle` 已能正式回放
- `M3` 行为状态链与退出语义已正式输出
- API 已能稳定投影最小正式对象
- 单股最小回放链已成立
- 关键测试与回归通过
- 验收结果已明确可判定
