# 低频卖出逻辑重构设计（设计定稿）

日期：2026-06-19  
范围：`lowfreq_engine_v16_advanced.py` 卖出主链、市场代理映射、持仓期市场回撤计算、`market_top` 结构型判定、对应单元测试

## 1. 背景与目标

- 当前低频卖出主链仍保留旧语义：
  - `-5%` 硬止损
  - 盈利后追踪止损
  - `partial_profit`
  - `target_reached`
  - `max_hold`
  - `sector_cooldown`
- 用户已明确要求本轮卖出逻辑只围绕三层条件重构：
  - 个股自买入后高点回调 `20%`
  - 个股对应市场代理自买入后运行高点回撤 `10%`
  - 模型判断大盘见顶后，直接清仓
- 用户已进一步确认：
  - `-5%` 硬止损本轮先移除
  - “回调”和“止损”不是同一概念
  - “个股回调 20% / 大盘回调 10%”按“各自高点回撤”计算
  - “大盘”不做全市场单一指数，而是参考个股所在市场：
    - `科创板`
    - `创业板`
    - `上证`
    - `深圳`
  - `市场代理 10% 回撤` 采用口径 A：
    - 仅从该笔持仓的买入日开始计算对应市场代理的运行峰值

目标：
- 将卖出主链收敛为用户指定的三层结构。
- 把“大盘见顶”从容易误杀牛股的幅度型触发，改为结构型确认。
- 明确指出会导致“低级错误”的条件，不做隐性保留。

## 2. 当前问题

### 2.1 旧卖出链与当前业务口径冲突

- 当前活跃卖出入口是 `check_sell_signal_v2(...)`。
- 其卖出顺序仍是：
  - 硬止损
  - 盈利后追踪止损
  - 分批止盈
  - 目标收益
  - 到期卖出
  - 板块人气消散
- 这与当前业务要求明显不一致。

### 2.2 “大盘见顶”不能用小幅回落代替

- 用户已明确指出：行情启动过程中的正常回调非常常见。
- 因此，如果用“近期高点回落超过 `5%`”作为见顶依据，会系统性过早清仓强趋势个股。
- 该条件会造成明显误判，本设计明确废弃。

### 2.3 仓库中没有现成的稳定市场归属字段

- 当前 `stocks` 表只有：
  - `code`
  - `name`
  - `sector_lv1`
  - 其他财务/行业字段
- 没有现成的 `exchange` / `market` 字段可直接用于市场归属。
- 因此，市场代理映射必须基于股票代码规则完成，而不能假设数据库已有显式市场标签。

## 3. 方案比较

### 3.1 方案 A：统一大盘代理

- 所有股票统一参考同一个大盘代理。

优点：
- 实现最简单。

缺点：
- 与用户明确要求不一致。
- 无法区分创业板股和上证主板股处于不同市场节奏。

### 3.2 方案 B：按持仓所属市场映射代理

- 每只股票按代码规则映射到对应市场代理：
  - `688*` -> `科创板`
  - `300*` -> `创业板`
  - `60*` -> `上证`
  - `00* / 002*` -> `深圳`
- 卖出时只看该笔持仓自己的市场代理。

优点：
- 与用户口径一致。
- 保留了市场环境差异。

缺点：
- 需要新增市场映射与市场代理序列计算。

### 3.3 方案 C：按板块主题映射代理

- 例如 AI、半导体、新能源分别使用不同主题代理。

优点：
- 更细。

缺点：
- 当前需求没有要求。
- 显著扩大问题范围，且主题代理定义更不稳定。

结论：
- 本次采用方案 B。

## 4. 设计原则

- 不保留与当前业务口径冲突的旧卖出条件。
- “大盘见顶”采用结构型确认，不采用小幅回撤类单点条件。
- “市场代理 10% 回撤”与“个股 20% 回调”都采用买入日起的峰值口径。
- 市场代理映射必须显式、可解释、可测试。
- 对无法可靠映射或无法可靠计算的情况，宁可返回“无市场卖出信号”，也不做隐性错误映射。

## 5. 新卖出主链设计

### 5.1 卖出优先级

新的卖出顺序固定为：

1. `market_top`
   - 大盘见顶，直接清仓
2. `market_drawdown`
   - 该笔持仓对应市场代理自买入后峰值回撤 `10%`
3. `stock_drawdown`
   - 个股自买入后峰值回调 `20%`

本轮明确停用：
- `stop_loss`
- `trailing stop`
- `partial_profit`
- `target_reached`
- `max_hold`
- `sector_cooldown`

### 5.2 个股回调 20%

定义：
- 从该笔持仓买入日起，记录个股持仓期峰值。
- 当前价格相对持仓峰值回调达到 `20%` 时触发卖出。

公式：

```text
stock_drawdown_pct = (current_price - stock_peak_price) / stock_peak_price * 100
```

触发条件：

```text
stock_drawdown_pct <= -20
```

说明：
- 这里的峰值不是买入价，而是买入后运行过程中形成的最高价。
- 该逻辑只反映个股本身是否已明显走弱。

### 5.3 市场代理 10% 回撤

定义：
- 对每笔持仓，按股票代码映射到一个市场代理。
- 从该笔持仓买入日起，记录该市场代理的持仓期峰值。
- 当前市场代理相对该峰值回撤达到 `10%` 时触发卖出。

公式：

```text
market_drawdown_pct = (current_market_value - market_peak_value) / market_peak_value * 100
```

触发条件：

```text
market_drawdown_pct <= -10
```

说明：
- 这是环境型离场条件，不要求个股自己已经回撤 `20%`。
- 它的含义是：该笔持仓所处市场环境已经明显转弱。
- 峰值从买入日起计算，不拿持仓前的旧高点参与判断。

### 5.4 大盘见顶

定义：
- 大盘见顶不是“小幅回落”，而是对应市场结构性转弱。
- 见顶一旦成立，不等待 `10%` 市场回撤和 `20%` 个股回调，直接清仓。

本次设计采用 `2/3` 组合确认：

1. 对应市场代理跌破 `20 日均线`
2. 对应市场代理 `20 日均线` 走平或拐头向下
3. 对应市场样本中，站上 `20 日均线` 的股票占比 `< 40%`

触发条件：
- 同一交易日满足以上 `2` 条及以上，判定 `market_top`

说明：
- 不使用“近期高点回落超过 `5%`”作为见顶条件。
- 不使用“单日跌破 `20 日均线`”单独触发清仓。
- `market_top` 是比 `market_drawdown` 更高优先级的结构型清仓条件。
- 本设计将“`20 日均线` 走平或拐头向下”写死为：

```text
ma20_today <= ma20_3d_ago
```

- 这样避免把单日均线抖动误判成趋势反转。

## 6. 市场代理映射设计

### 6.1 映射规则

- `688*` -> `kcb`（科创板）
- `300*` -> `cyb`（创业板）
- `60*` -> `sse`（上证）
- `00*` -> `szse`（深圳）
- `002*` -> `szse`（深圳）

### 6.2 未命中映射规则的处理

以下情况不允许静默错映射：
- 北交所代码
- ETF
- 指数代码
- 其他非 A 股股票代码

处理原则：
- 返回“无市场代理”
- 不触发 `market_top`
- 不触发 `market_drawdown`
- 仅允许 `stock_drawdown` 参与判断
- 同时在 `details` 或内部日志中留下可审计信息

原因：
- 错映射会直接制造“低级错误”，比如把非目标市场资产硬套进上证或深圳代理。

## 7. 市场代理序列设计

### 7.1 代理构造方向

- 当前设计不依赖数据库中的官方指数行情代码。
- 第一版使用对应市场股票横截面构造市场代理序列。
- 例如：
  - `科创板`：按 `688*` 股票集合构造代理
  - `创业板`：按 `300*` 股票集合构造代理
  - `上证`：按 `60*` 股票集合构造代理
  - `深圳`：按 `00* / 002*` 股票集合构造代理

### 7.2 代理计算原则

- 代理值必须是单日可重复计算的时序值。
- 第一版优先使用简单、可解释的等权市场代理。
- 不在本轮引入复杂权重模型，避免扩大问题范围。

唯一口径：
- 使用对应市场样本的日收益等权累计成代理序列。

原因：
- `daily_prices` 已有按日行情，可直接支持该计算。
- 该口径实现简单、可解释、可复现。
- 相比“标准化价格中位数”，它在实现上更直接，不需要额外引入价格尺度归一化规则。

本次设计要求：
- 不允许和“标准化价格中位数”并存。
- 测试必须锁定该等权累计口径的结果语义。

## 8. 低级错误风险点

以下条件若直接实施，会产生明显“低级错误”，本设计明确标出：

### 8.1 用“近期高点回落超过 5%”判定见顶

问题：
- 强趋势行情中常见正常回调。
- 会过早清仓强股，留下弱股。

处理：
- 明确废弃，不进入实现。

### 8.2 用“单日跌破 20 日均线”单独判定见顶

问题：
- 单日波动噪声过大。
- 很容易把震荡洗盘误判成顶部。

处理：
- 只能作为 `market_top` 组合条件的一部分。

### 8.3 对未知代码前缀做静默市场映射

问题：
- 会把不属于目标市场的资产错误归到上证/深圳/创业板/科创板。
- 这是直接的错误归类。

处理：
- 未命中映射规则时，禁止市场卖出条件生效。

### 8.4 同时保留旧卖出链条件

问题：
- 会让新卖出逻辑被旧条件提前触发覆盖。
- 回测结果将无法归因。

处理：
- 本轮必须整体停用旧卖出链条件。

## 9. 代码改造范围

### 9.1 核心改造

文件：
- `lowfreq_engine_v16_advanced.py`

改造点：
- 重写 `check_sell_signal_v2(...)`
- 新增市场归属解析函数
- 新增市场代理读取/计算函数
- 新增持仓期市场峰值计算函数
- 新增 `market_top` 结构判断函数

### 9.2 测试改造

文件：
- `tests/unit/test_lowfreq_engine_v16_sell_logic.py`

改造方向：
- 删除或改写旧测试语义
- 补充新卖出链测试

最少测试集合：
- `688*` 映射到科创板
- `300*` 映射到创业板
- `60*` 映射到上证
- `00* / 002*` 映射到深圳
- `market_top` 优先于另外两个条件
- `market_drawdown` 在 `market_top` 不成立时单独触发
- `stock_drawdown` 在前两者不成立时触发
- 三条件都不满足时返回 `None`
- 未知前缀代码不会误触发市场卖出
- 旧条件不会误触发

## 10. 验收标准

- 卖出主链只保留三层新语义。
- 回测产物中的卖出原因可明确区分：
  - `market_top`
  - `market_drawdown`
  - `stock_drawdown`
- 不再出现旧卖出理由混入新结果的情况。
- 未知代码前缀不会被静默错映射。
- 单元测试能够覆盖市场映射、优先级和边界处理。

## 11. 结论

- 本次卖出逻辑重构采用“按持仓所属市场映射代理”的方案。
- `市场代理 10% 回撤` 与 `个股 20% 回调` 都从买入日起计算峰值。
- `大盘见顶` 明确采用结构型确认，不采用 `5%` 小回落判顶。
- 如后续实现中发现新的“低级错误”风险，应先中止实现并向用户确认，而不是擅自保留或替换条件。
