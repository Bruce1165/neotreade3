# NeoTrade3 前端文案审计第 3 批修正（设计定稿）

日期：2026-06-17  
范围：`neotrade3-dashboard` 中 2 处“建议层/执行层”弱风险文案

## 1. 背景

- 前端文案审计前两批已完成，并已修正：
  - 收益/回测/市场环境口径混淆
  - 筛选器页统计标题不准确
  - 公共 Header / 黄旗中两处事实性不准确文案
- 第 3 轮审计继续聚焦“弱风险文案”，重点检查：
  - 是否会把建议层误读成执行层
  - 是否会把说明性对象误读成已确认结论
- 当前确认有 2 处最值得优先收紧的文案：
  - 全局页头 `团队控制台：执行 / 监控 / 复盘`
  - `主线审阅` 页中的 `焦点候选`

## 2. 本轮问题

### 2.1 全局页头的“执行 / 监控 / 复盘”

现状：
- Header 固定文案为：
  - `团队控制台：执行 / 监控 / 复盘`

问题：
- 这是全局文案，会覆盖所有页面语义。
- 但当前系统里，至少 `主线审阅` 明确仍然属于建议层 / 审阅层，而不是正式执行层。
- 因此“执行”这个词放在全局总描述里，存在把建议层页面误读为执行层页面的弱风险。

### 2.2 `主线审阅` 页中的“焦点候选”

现状：
- `主线审阅` 页高优先级卡片标题为：
  - `焦点候选`

问题：
- 该卡片处于页面核心位置，且展示“推荐 / 观察 / 回避”等强结论标签。
- 但当前页面的正式定位仍是建议层 / 审阅层，不是执行层。
- “焦点候选”这个说法本身不算错，但它缺少“建议层”限定，容易被误读为已选定执行对象。

## 3. 方案比较

### 3.1 方案 A：只收紧文案，不改布局

- Header 改成中性、非执行导向的总描述
- `焦点候选` 改成更明确的建议层文案

优点：
- 改动最小
- 风险最低
- 不影响页面结构与业务字段

缺点：
- 只能降低误读，不能彻底替代更强的结构性提示

### 3.2 方案 B：增加解释型说明块

- 保留现有标题，同时增加一段说明：
  - 当前仅为建议层，不构成执行指令

优点：
- 解释更充分

缺点：
- 页面更重
- 本轮改动超出“最小收紧文案”边界

### 3.3 结论

- 采用方案 A。

## 4. 设计

### 4.1 Header 文案

当前：
- `团队控制台：执行 / 监控 / 复盘`

改为：
- `团队控制台：审阅 / 监控 / 复盘`

理由：
- `审阅` 更符合当前系统对建议层页面的覆盖性描述。
- 保留 `监控 / 复盘` 不变，避免扩大语义调整范围。
- 这样既不否认系统中存在执行相关数据，也不会把整个前端默认表述成执行层入口。

### 4.2 `主线审阅` 页卡片标题

当前：
- `焦点候选`

改为：
- `焦点候选（建议层）`

理由：
- 保留“焦点候选”这个已被用户和页面结构熟悉的核心词。
- 只追加最短限定词 `建议层`，不改动其它卡片内容。
- 这样可以明确告诉用户：这里是审阅聚焦对象，不是执行确认对象。

## 5. 本轮边界

- 仅修改上述 2 处文案。
- 不新增解释面板。
- 不重做 `主线审阅` 页布局。
- 不调整 `推荐 / 观察 / 回避` 本身的状态标签。
- 不改动任何后端字段、统计口径或推荐逻辑。

## 6. 验证

- 更新相关前端测试，覆盖：
  - Header 显示新的 `审阅 / 监控 / 复盘`
  - `主线审阅` 页显示 `焦点候选（建议层）`
- 运行：
  - `npm run test -- --run`
  - `npm run build`

## 7. 完成标准

- 全局页头不再把整个 Dashboard 默认描述为“执行层控制台”。
- `主线审阅` 的高优先级候选卡不再缺少建议层限定。
- 前端测试与构建通过。
