# NeoTrade3 文档生命周期治理规范

## 1. 目的

本规范用于避免 NeoTrade3 在后续升级、迁移和重构过程中持续积累以下噪音：

- 过期目录
- 过期文档
- 无主文档
- 被替代后仍保留在活跃路径中的旧材料

本规范只约束第一方项目文档与文档目录，不约束第三方依赖、自带 license、或外部导入资料。

## 2. 真相源分层

NeoTrade3 的文档只允许按以下 4 层理解：

1. `PROJECT_STATUS.md`
   - 当前状态真相源
   - 当前边界、当前约束、当前下一步，只以本文件为准
2. `docs/handoffs/`
   - 会话续接层
   - 只记录事实、证据路径、续接建议，不替代当前状态真相源
3. `docs/superpowers/specs/`
   - 设计/计划层
   - 记录 design、plan、task-list、contract-definition 等实施前后约束
4. `docs/archive/`
   - 历史留存层
   - 只承担历史保留，不承担当前正式口径

任何新文档都必须先明确属于哪一层；未能归入上述 4 层的文档，不应创建。

## 3. 文档状态

第一方文档统一使用以下状态：

- `active`
  - 当前仍在使用，且内容允许被当作现行口径引用
- `reference`
  - 仍可作为背景材料参考，但不能单独作为当前正式口径
- `archived`
  - 已退出活跃路径，只保留历史留痕

除 `PROJECT_STATUS.md` 外，任何文档如果没有显式声明状态，默认不得被视为当前正式口径。

## 4. 文档头要求

新增或实质更新的第一方 Markdown 文档，顶部应补齐以下字段：

```md
Status: active|reference|archived
Owner: <模块/责任人>
Scope: <文档覆盖边界>
Canonical: <当前真相源路径；若本文自身即真相源，则写 self>
Supersedes: <被本文件替代的旧文件路径；无则写 none>
Superseded_by: <替代本文的新文件路径；未被替代则写 none>
Last_reviewed: YYYY-MM-DD
```

说明：

- `Canonical` 用于说明“当前以谁为准”
- `Supersedes` 与 `Superseded_by` 用于显式建立替代关系
- 历史旧文档不要求一次性补齐，但从现在开始新增或实质更新的文档必须遵守

## 5. 新目录准入规则

禁止无约束新增文档目录。

如果确需新增目录，必须同时满足：

1. 目录下存在 `README.md`
2. `README.md` 中写明：
   - 目录用途
   - 目录 owner
   - 允许存放的文档类型
   - 退出条件或归档条件
3. 该目录无法被现有层级承接

如果上述 3 条不能同时满足，则不得新增目录，应复用现有目录。

## 6. 替代与归档规则

任何新文档替代旧文档时，必须在同一刀内完成以下动作：

1. 新文档落地
2. 旧文档写明 `Superseded_by`
3. 如旧文档不再属于活跃路径，则移动到 `docs/archive/` 对应子目录
4. 更新相关目录 `README.md` 或 `PROJECT_STATUS.md` 引导入口

禁止以下做法：

- 新文档已生效，但旧文档继续留在活跃路径且没有替代说明
- 仅通过口头说明“旧文档已废弃”，不做文件级标记
- 为临时阶段额外开目录，工作结束后无人收口

## 7. 目录用途边界

### 7.1 `docs/handoffs/`

- 只放当前会话续接或阶段 handoff
- 当内容只剩历史价值、不再承担续接职责时，应转入 `docs/archive/handover/`

### 7.2 `docs/superpowers/specs/`

- 只放 design / plan / task-list / contract-definition / repo-audit
- 如果某文档仅剩历史价值，且未来不会再作为活跃 spec 被引用，应迁入 archive 而不是继续保留在活跃 specs 根下

### 7.3 `docs/archive/`

- 只放已退出活跃路径的历史材料
- archive 中材料必须明确自己不是当前正式口径

## 8. 会话收尾要求

每次较大工作结束前，必须额外检查一次：

1. 是否产生了新的临时目录
2. 是否产生了新的无主文档
3. 是否产生了被新文档替代但仍处于活跃路径的旧文档
4. 是否产生了与当前主题无关的残留文件

如果存在上述任一情况，应在收尾阶段完成清理、归档或标记，不允许直接带入下一轮工作。

## 9. 最小执行策略

本规范采用保守执行方式：

- 先约束新增
- 再逐步治理存量

因此，现阶段的强制要求只有两条：

1. 从现在开始，新增/实质更新的第一方文档必须具备生命周期头
2. 从现在开始，新增目录必须同时带 `README.md` 与退出条件

存量旧文档的系统性清理，按主题逐步推进，不要求一次性完成。
