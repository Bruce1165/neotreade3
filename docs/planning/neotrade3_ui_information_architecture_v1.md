# NeoTrade3 UI 信息架构 v1（建议稿）

**Last Updated**: 2026-05-22

---

## 0. UI 目标与约束

本文件把 UI 从“模块/实现细节的汇总页”收敛为“面向团队用户的结果层”，并明确运维层的边界。

### 0.1 关键约束（已确认）

- 默认只展示结果层，隐藏:
  - raw payload
  - 运行流水与台账细节（run ledger / batch ledger 等）
  - 配置与路径（env var / DB path / 内部文件结构）
- 所有展示与交互文案必须使用中文
- 页面内容/功能组织围绕用户的使用要求定义
- 特殊功能必须自解释（名称、触发条件、输入、输出、失败提示）
- 每个页面条目与展示方式逐项与你确认后再固化

### 0.2 两层界面规则

参考已有原则文档 [neotrade3_ui_design_principles.md](file:///Users/mac/NeoTrade3/docs/migration/neotrade3_ui_design_principles.md#L88-L97):

- 结果层（默认）: 面向团队用户的结论、建议动作、候选与解释
- 运维层（受控）: 面向维护/排障的健康度、执行链路、契约校验、可回放入口

## 1. 页面与信息架构（v1 建议）

v1 建议收敛为 6 个一级入口（结果层 4 个 + 运维层 2 个）。

### 1.1 结果层（默认）

#### A. 今日总览（首页）

- 用户问题:
  - 今天系统能不能跑
  - 当前候选集是什么、变化是什么
  - 确定性/评分有什么变化
  - 现在最应该做的下一步是什么
- 主展示区（默认可见，仅摘要卡片）:
  - 系统状态摘要（就绪/阻塞/异常）
  - 候选集摘要（数量、变化、top 变化）
  - 确定性/评分摘要（分布、上升/下降的标的）
  - 今日问题摘要（只展示“结论+下一步”，不展示内部堆栈/路径）
- 主动作（默认可见）:
  - 刷新
  - 下载: 候选集 CSV / 今日矩阵摘要 CSV
  - 查看: 单标的解释 / 候选变更

#### B. 候选集（候选与各池）

- 用户问题:
  - 候选集从哪里来（高层解释）
  - 为什么某标的进/出
  - 下一步该关注谁
- 主展示区:
  - 股票信息列表（至少包含: 代码、名称、评分、确定性）
  - 进出变化（diff 视图）
  - 每个标的的 CHECK 结果一句话结论（自然语言 + 1～2 个关键数值证据）
- 主动作:
  - 下载 CSV
  - 进入单标的解释页
- 必须隐藏:
  - 单个筛选器的原始 JSON 输出（除非进入运维层或开发者模式）

#### C. 单标的解释（个股解释）

- 用户问题:
  - 为什么是它（因子贡献与证据）
  - 现在属于哪种交易段/风险状态（v1 可做最小表达）
- 主展示区:
  - 结论卡片: 当前评分/确定性、变化方向、关键结论
  - 因子贡献摘要: 因子列表 + 当前值 + 方向 + 简短解释
  - 历史趋势（v1 可做最小折线/表格）
- 主动作:
  - 下载: 单标的因子明细 CSV
  - 返回候选集

#### D. 因子矩阵（矩阵浏览）

- 用户问题:
  - 矩阵现在长什么样
  - 哪些因子在变得更有效/更无效
  - 分布/分位情况如何
- 主展示区:
  - 矩阵摘要（按因子/按标的维度的聚合）
  - 因子健康度（最小版本: 缺失率、稳定性、覆盖率）
  - 矩阵历史（最小版本: 行数、覆盖率、分布）
- 主动作:
  - 下载: 矩阵摘要 CSV / 因子健康 CSV

### 1.2 运维层（受控入口）

#### E. 运行健康（运维：运行健康）

- 目的:
  - 让维护者快速定位“卡在哪里、该怎么修”，但不把内部细节推给普通用户
- 主展示区（默认展示为摘要）:
  - 今日 run 状态（成功/失败/阻塞）
  - 阶段级摘要（data_control / screening / matrix / learning）
  - 失败与阻塞的“短原因 + 建议动作”
- 下钻（需要显式展开）:
  - task 级状态与耗时
  - 错误类别（不展示内部路径与堆栈，除非开发者模式）

#### F. 契约与配置健康（运维：契约与配置）

- 目的:
  - 让维护者知道“配置是不是有效、契约有没有破”，并提供最短修复路径
- 主展示区:
  - 契约校验摘要（pass/fail）
  - 关键配置项是否缺失（仅显示“缺失什么”，不显示实际路径/密钥）
- 下钻:
  - 配置契约失败详情（受控展示）

## 2. UI 模板方案（至少 3 套，供你选 1 套作为 v1 方向）

### 方案 1：决策驾驶舱（推荐默认）

- 结构:
  - 首页强聚合（卡片 + 今日结论 + 下一步）
  - 候选/单标的解释作为下钻
  - 运维层入口弱化、默认隐藏
- 适用:
  - 团队用户只关心“结论与行动”
- 风险:
  - 需要我们先把“结论字段”定义清楚，否则会变成空壳

### 方案 2：研究工作台（面向解释与复盘）

- 结构:
  - 左侧对象（候选集/标的/因子）
  - 右侧解释面板（贡献、证据、历史）
  - 首页弱化，强调“解释”能力
- 适用:
  - 团队需要频繁复盘“为什么选/为什么不选”
- 风险:
  - 对数据口径与解释字段要求更高

### 方案 3：流程监控（面向“能不能跑”）

- 结构:
  - 首页以运行状态为主（就绪/阻塞/失败）
  - 结果只给摘要与下载入口
  - 适合作为运维层主入口，而非团队默认入口
- 适用:
  - 当前阶段系统不稳定，需要优先提升可用性
- 风险:
  - 容易滑回“用户被迫关注内部细节”的旧模式

### 方案 4：迁移对照视图（可选增强）

- 结构:
  - 将 v2 的关键输出条目作为“回归对照基线”展示（不依赖 v2 运行时）
  - 用于提升信心与验收可追溯性
- 适用:
  - 迁移阶段需要逐项确认“v3 没丢能力”
- 风险:
  - 需要先定义“对照口径”与取数来源

## 3. 展示方式候选（每项都需你确认后落地）

### 3.1 候选变化（diff）

- 方式 A: 增/删两列列表 + 变更原因摘要
- 方式 B: 单表格标注进/出 + 变化标签

### 3.2 因子贡献（解释）

- 方式 A: 表格（因子、值、方向、贡献、解释）
- 方式 B: Top N 因子卡片 + 可展开表格

### 3.4 CHECK 原因（一句话结论）

- 默认格式: 结论 + 关键原因 + 1～2 个关键数值证据
- 示例:
  - 未通过：近 20 日趋势不连续（有效段 12/20），成交量不满足阈值（放量日 1/5）

### 3.3 确定性/评分趋势

- 方式 A: 折线（近 N 日）+ 当前值摘要
- 方式 B: 分位分布条 + 上升/下降榜单

## 4. 待你逐项确认清单（确认后再实施）

### 4.1 首页（今日总览）

- [ ] 首页必须展示哪些卡片（建议 5～8 个）
- [ ] “下一步建议动作”的文案模板（例如: 数据未就绪、先更新交易日历等）
- [ ] 是否需要“今日候选集变化”作为首页强曝光模块

### 4.2 候选集

- [ ] 候选列表字段（symbol、名称、评分、确定性、关键因子标签等）
- [ ] “进/出原因”的最小解释字段
- [ ] CSV 下载格式（列顺序、字段名、编码）

### 4.3 单标的解释

- [ ] 因子解释需要展示哪些证据字段（最小可用范围）
- [ ] 历史窗口（20/60/120 交易日等）与默认值

### 4.4 因子矩阵

- [ ] v1 需要展示哪些矩阵健康指标（缺失率/覆盖率/稳定性等的最小集合）
- [ ] 是否需要因子“有效/无效”的初版判定（如果需要，判定口径）

### 4.5 运维层入口

- [ ] 运维层开启方式（dev 参数 / API key / 本地开关）
- [ ] 运维层允许展示到什么深度（到 task 级？是否允许错误堆栈？）
